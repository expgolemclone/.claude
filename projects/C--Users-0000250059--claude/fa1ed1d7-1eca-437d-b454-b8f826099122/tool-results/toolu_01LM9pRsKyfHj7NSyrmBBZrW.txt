     5Ôćĺ# Hooks reference
     6Ôćĺ
     7Ôćĺ> Reference for Claude Code hook events, configuration schema, JSON input/output formats, exit codes, async hooks, prompt hooks, and MCP tool hooks.
     8Ôćĺ
     9Ôćĺ<Tip>
    10Ôćĺ  For a quickstart guide with examples, see [Automate workflows with hooks](/en/hooks-guide).
    11Ôćĺ</Tip>
    12Ôćĺ
    13ÔćĺHooks are user-defined shell commands or LLM prompts that execute automatically at specific points in Claude Code's lifecycle. Use this reference to look up event schemas, configuration options, JSON input/output formats, and advanced features like async hooks and MCP tool hooks. If you're setting up hooks for the first time, start with the [guide](/en/hooks-guide) instead.
    14Ôćĺ
    15Ôćĺ## Hook lifecycle
    16Ôćĺ
    17ÔćĺHooks fire at specific points during a Claude Code session. When an event fires and a matcher matches, Claude Code passes JSON context about the event to your hook handler. For command hooks, this arrives on stdin. Your handler can then inspect the input, take action, and optionally return a decision. Some events fire once per session, while others fire repeatedly inside the agentic loop:

    25ÔćĺThe table below summarizes when each event fires. The [Hook events](#hook-events) section documents the full input schema and decision control options for each one.
    26Ôćĺ
    27Ôćĺ| Event                | When it fires                                                                                               |
    28Ôćĺ| :------------------- | :---------------------------------------------------------------------------------------------------------- |
    29Ôćĺ| `SessionStart`       | When a session begins or resumes                                                                            |
    30Ôćĺ| `UserPromptSubmit`   | When you submit a prompt, before Claude processes it                                                        |
    31Ôćĺ| `PreToolUse`         | Before a tool call executes. Can block it                                                                   |
    32Ôćĺ| `PermissionRequest`  | When a permission dialog appears                                                                            |
    33Ôćĺ| `PostToolUse`        | After a tool call succeeds                                                                                  |
    34Ôćĺ| `PostToolUseFailure` | After a tool call fails                                                                                     |
    35Ôćĺ| `Notification`       | When Claude Code sends a notification                                                                       |
    36Ôćĺ| `SubagentStart`      | When a subagent is spawned                                                                                  |
    37Ôćĺ| `SubagentStop`       | When a subagent finishes                                                                                    |
    38Ôćĺ| `Stop`               | When Claude finishes responding                                                                             |
    39Ôćĺ| `TeammateIdle`       | When an [agent team](/en/agent-teams) teammate is about to go idle                                          |
    40Ôćĺ| `TaskCompleted`      | When a task is being marked as completed                                                                    |
    41Ôćĺ| `ConfigChange`       | When a configuration file changes during a session                                                          |
    42Ôćĺ| `WorktreeCreate`     | When a worktree is being created via `--worktree` or `isolation: "worktree"`. Replaces default git behavior |
    43Ôćĺ| `WorktreeRemove`     | When a worktree is being removed, either at session exit or when a subagent finishes                        |
    44Ôćĺ| `PreCompact`         | Before context compaction                                                                                   |
    45Ôćĺ| `SessionEnd`         | When a session terminates                                                                                   |

    47Ôćĺ### How a hook resolves
    48Ôćĺ
    49ÔćĺTo see how these pieces fit together, consider this `PreToolUse` hook that blocks destructive shell commands. The hook runs `block-rm.sh` before every Bash tool call:
    50Ôćĺ
    51Ôćĺ```json  theme={null}
    52Ôćĺ{
    53Ôćĺ  "hooks": {
    54Ôćĺ    "PreToolUse": [
    55Ôćĺ      {
    56Ôćĺ        "matcher": "Bash",
    57Ôćĺ        "hooks": [
    58Ôćĺ          {
    59Ôćĺ            "type": "command",
    60Ôćĺ            "command": ".claude/hooks/block-rm.sh"
    61Ôćĺ          }
    62Ôćĺ        ]
    63Ôćĺ      }
    64Ôćĺ    ]
    65Ôćĺ  }
    66Ôćĺ}
    67Ôćĺ```
    68Ôćĺ
    69ÔćĺThe script reads the JSON input from stdin, extracts the command, and returns a `permissionDecision` of `"deny"` if it contains `rm -rf`:
    70Ôćĺ
    71Ôćĺ```bash  theme={null}
    72Ôćĺ#!/bin/bash
    73Ôćĺ# .claude/hooks/block-rm.sh
    74ÔćĺCOMMAND=$(jq -r '.tool_input.command')
    75Ôćĺ
    76Ôćĺif echo "$COMMAND" | grep -q 'rm -rf'; then
    77Ôćĺ  jq -n '{
    78Ôćĺ    hookSpecificOutput: {
    79Ôćĺ      hookEventName: "PreToolUse",
    80Ôćĺ      permissionDecision: "deny",
    81Ôćĺ      permissionDecisionReason: "Destructive command blocked by hook"
    82Ôćĺ    }
    83Ôćĺ  }'
    84Ôćĺelse
    85Ôćĺ  exit 0  # allow the command
    86Ôćĺfi
    87Ôćĺ```

    95Ôćĺ<Steps>
    96Ôćĺ  <Step title="Event fires">
    97Ôćĺ    The `PreToolUse` event fires. Claude Code sends the tool input as JSON on stdin to the hook:
    98Ôćĺ
    99Ôćĺ    ```json  theme={null}
   100Ôćĺ    { "tool_name": "Bash", "tool_input": { "command": "rm -rf /tmp/build" }, ... }
   101Ôćĺ    ```
   102Ôćĺ  </Step>
   103Ôćĺ
   104Ôćĺ  <Step title="Matcher checks">
   105Ôćĺ    The matcher `"Bash"` matches the tool name, so `block-rm.sh` runs. If you omit the matcher or use `"*"`, the hook runs on every occurrence of the event. Hooks only skip when a matcher is defined and doesn't match.
   106Ôćĺ  </Step>
   107Ôćĺ
   108Ôćĺ  <Step title="Hook handler runs">
   109Ôćĺ    The script extracts `"rm -rf /tmp/build"` from the input and finds `rm -rf`, so it prints a decision to stdout:
   110Ôćĺ
   111Ôćĺ    ```json  theme={null}
   112Ôćĺ    {
   113Ôćĺ      "hookSpecificOutput": {
   114Ôćĺ        "hookEventName": "PreToolUse",
   115Ôćĺ        "permissionDecision": "deny",
   116Ôćĺ        "permissionDecisionReason": "Destructive command blocked by hook"
   117Ôćĺ      }
   118Ôćĺ    }
   119Ôćĺ    ```
   120Ôćĺ
   121Ôćĺ    If the command had been safe (like `npm test`), the script would hit `exit 0` instead, which tells Claude Code to allow the tool call with no further action.
   122Ôćĺ  </Step>
   123Ôćĺ
   124Ôćĺ  <Step title="Claude Code acts on the result">
   125Ôćĺ    Claude Code reads the JSON decision, blocks the tool call, and shows Claude the reason.
   126Ôćĺ  </Step>
   127Ôćĺ</Steps>

   130Ôćĺ
   131Ôćĺ## Configuration
   132Ôćĺ
   133ÔćĺHooks are defined in JSON settings files. The configuration has three levels of nesting:
   134Ôćĺ
   135Ôćĺ1. Choose a [hook event](#hook-events) to respond to, like `PreToolUse` or `Stop`
   136Ôćĺ2. Add a [matcher group](#matcher-patterns) to filter when it fires, like "only for the Bash tool"
   137Ôćĺ3. Define one or more [hook handlers](#hook-handler-fields) to run when matched
   138Ôćĺ
   139ÔćĺSee [How a hook resolves](#how-a-hook-resolves) above for a complete walkthrough with an annotated example.
   140Ôćĺ
   141Ôćĺ<Note>
   142Ôćĺ  This page uses specific terms for each level: **hook event** for the lifecycle point, **matcher group** for the filter, and **hook handler** for the shell command, prompt, or agent that runs. "Hook" on its own refers to the general feature.
   143Ôćĺ</Note>
   144Ôćĺ
   145Ôćĺ### Hook locations
   146Ôćĺ
   147ÔćĺWhere you define a hook determines its scope:
   148Ôćĺ
   149Ôćĺ| Location                                                   | Scope                         | Shareable                          |
   150Ôćĺ| :--------------------------------------------------------- | :---------------------------- | :--------------------------------- |
   151Ôćĺ| `~/.claude/settings.json`                                  | All your projects             | No, local to your machine          |
   152Ôćĺ| `.claude/settings.json`                                    | Single project                | Yes, can be committed to the repo  |
   153Ôćĺ| `.claude/settings.local.json`                              | Single project                | No, gitignored                     |
   154Ôćĺ| Managed policy settings                                    | Organization-wide             | Yes, admin-controlled              |
   155Ôćĺ| [Plugin](/en/plugins) `hooks/hooks.json`                   | When plugin is enabled        | Yes, bundled with the plugin       |
   156Ôćĺ| [Skill](/en/skills) or [agent](/en/sub-agents) frontmatter | While the component is active | Yes, defined in the component file |
   157Ôćĺ
   158ÔćĺFor details on settings file resolution, see [settings](/en/settings). Enterprise administrators can use `allowManagedHooksOnly` to block user, project, and plugin hooks. See [Hook configuration](/en/settings#hook-configuration).
   159Ôćĺ
   160Ôćĺ### Matcher patterns
   161Ôćĺ
   162ÔćĺThe `matcher` field is a regex string that filters when hooks fire. Use `"*"`, `""`, or omit `matcher` entirely to match all occurrences. Each event type matches on a different field:
   163Ôćĺ
   164Ôćĺ| Event                                                                                           | What the matcher filters  | Example matcher values                                                             |
   165Ôćĺ| :---------------------------------------------------------------------------------------------- | :------------------------ | :--------------------------------------------------------------------------------- |
   166Ôćĺ| `PreToolUse`, `PostToolUse`, `PostToolUseFailure`, `PermissionRequest`                          | tool name                 | `Bash`, `Edit\|Write`, `mcp__.*`                                                   |
   167Ôćĺ| `SessionStart`                                                                                  | how the session started   | `startup`, `resume`, `clear`, `compact`                                            |
   168Ôćĺ| `SessionEnd`                                                                                    | why the session ended     | `clear`, `logout`, `prompt_input_exit`, `bypass_permissions_disabled`, `other`     |
   169Ôćĺ| `Notification`                                                                                  | notification type         | `permission_prompt`, `idle_prompt`, `auth_success`, `elicitation_dialog`           |
   170Ôćĺ| `SubagentStart`                                                                                 | agent type                | `Bash`, `Explore`, `Plan`, or custom agent names                                   |
   171Ôćĺ| `PreCompact`                                                                                    | what triggered compaction | `manual`, `auto`                                                                   |
   172Ôćĺ| `SubagentStop`                                                                                  | agent type                | same values as `SubagentStart`                                                     |
   173Ôćĺ| `ConfigChange`                                                                                  | configuration source      | `user_settings`, `project_settings`, `local_settings`, `policy_settings`, `skills` |
   174Ôćĺ| `UserPromptSubmit`, `Stop`, `TeammateIdle`, `TaskCompleted`, `WorktreeCreate`, `WorktreeRemove` | no matcher support        | always fires on every occurrence                                                   |
   175Ôćĺ
   176ÔćĺThe matcher is a regex, so `Edit|Write` matches either tool and `Notebook.*` matches any tool starting with Notebook. The matcher runs against a field from the [JSON input](#hook-input-and-output) that Claude Code sends to your hook on stdin. For tool events, that field is `tool_name`. Each [hook event](#hook-events) section lists the full set of matcher values and the input schema for that event.
   177Ôćĺ
   178ÔćĺThis example runs a linting script only when Claude writes or edits a file:
   179Ôćĺ
   180Ôćĺ```json  theme={null}
   181Ôćĺ{
   182Ôćĺ  "hooks": {
   183Ôćĺ    "PostToolUse": [
   184Ôćĺ      {
   185Ôćĺ        "matcher": "Edit|Write",
   186Ôćĺ        "hooks": [
   187Ôćĺ          {
   188Ôćĺ            "type": "command",
   189Ôćĺ            "command": "/path/to/lint-check.sh"
   190Ôćĺ          }
   191Ôćĺ        ]
   192Ôćĺ      }
   193Ôćĺ    ]
   194Ôćĺ  }
   195Ôćĺ}
   196Ôćĺ```
   197Ôćĺ
   198Ôćĺ`UserPromptSubmit`, `Stop`, `TeammateIdle`, `TaskCompleted`, `WorktreeCreate`, and `WorktreeRemove` don't support matchers and always fire on every occurrence. If you add a `matcher` field to these events, it is silently ignored.
   199Ôćĺ
   200Ôćĺ#### Match MCP tools
   201Ôćĺ
   202Ôćĺ[MCP](/en/mcp) server tools appear as regular tools in tool events (`PreToolUse`, `PostToolUse`, `PostToolUseFailure`, `PermissionRequest`), so you can match them the same way you match any other tool name.
   203Ôćĺ
   204ÔćĺMCP tools follow the naming pattern `mcp__<server>__<tool>`, for example:
   205Ôćĺ
   206Ôćĺ* `mcp__memory__create_entities`: Memory server's create entities tool
   207Ôćĺ* `mcp__filesystem__read_file`: Filesystem server's read file tool
   208Ôćĺ* `mcp__github__search_repositories`: GitHub server's search tool
   209Ôćĺ
   210ÔćĺUse regex patterns to target specific MCP tools or groups of tools:
   211Ôćĺ
   212Ôćĺ* `mcp__memory__.*` matches all tools from the `memory` server
   213Ôćĺ* `mcp__.*__write.*` matches any tool containing "write" from any server
   214Ôćĺ
   215ÔćĺThis example logs all memory server operations and validates write operations from any MCP server:
   216Ôćĺ
   217Ôćĺ```json  theme={null}
   218Ôćĺ{
   219Ôćĺ  "hooks": {
   220Ôćĺ    "PreToolUse": [
   221Ôćĺ      {
   222Ôćĺ        "matcher": "mcp__memory__.*",
   223Ôćĺ        "hooks": [
   224Ôćĺ          {
   225Ôćĺ            "type": "command",
   226Ôćĺ            "command": "echo 'Memory operation initiated' >> ~/mcp-operations.log"
   227Ôćĺ          }
   228Ôćĺ        ]
   229Ôćĺ      },
   230Ôćĺ      {
   231Ôćĺ        "matcher": "mcp__.*__write.*",
   232Ôćĺ        "hooks": [
   233Ôćĺ          {
   234Ôćĺ            "type": "command",
   235Ôćĺ            "command": "/home/user/scripts/validate-mcp-write.py"
   236Ôćĺ          }
   237Ôćĺ        ]
   238Ôćĺ      }
   239Ôćĺ    ]
   240Ôćĺ  }
   241Ôćĺ}
   242Ôćĺ```
   243Ôćĺ
   244Ôćĺ### Hook handler fields
   245Ôćĺ
   246ÔćĺEach object in the inner `hooks` array is a hook handler: the shell command, LLM prompt, or agent that runs when the matcher matches. There are three types:
   247Ôćĺ
   248Ôćĺ* **[Command hooks](#command-hook-fields)** (`type: "command"`): run a shell command. Your script receives the event's [JSON input](#hook-input-and-output) on stdin and communicates results back through exit codes and stdout.
   249Ôćĺ* **[Prompt hooks](#prompt-and-agent-hook-fields)** (`type: "prompt"`): send a prompt to a Claude model for single-turn evaluation. The model returns a yes/no decision as JSON. See [Prompt-based hooks](#prompt-based-hooks).
   250Ôćĺ* **[Agent hooks](#prompt-and-agent-hook-fields)** (`type: "agent"`): spawn a subagent that can use tools like Read, Grep, and Glob to verify conditions before returning a decision. See [Agent-based hooks](#agent-based-hooks).
   251Ôćĺ
   252Ôćĺ#### Common fields
   253Ôćĺ
   254ÔćĺThese fields apply to all hook types:
   255Ôćĺ
   256Ôćĺ| Field           | Required | Description                                                                                                                                   |
   257Ôćĺ| :-------------- | :------- | :-------------------------------------------------------------------------------------------------------------------------------------------- |
   258Ôćĺ| `type`          | yes      | `"command"`, `"prompt"`, or `"agent"`                                                                                                         |
   259Ôćĺ| `timeout`       | no       | Seconds before canceling. Defaults: 600 for command, 30 for prompt, 60 for agent                                                              |
   260Ôćĺ| `statusMessage` | no       | Custom spinner message displayed while the hook runs                                                                                          |
   261Ôćĺ| `once`          | no       | If `true`, runs only once per session then is removed. Skills only, not agents. See [Hooks in skills and agents](#hooks-in-skills-and-agents) |
   262Ôćĺ
   263Ôćĺ#### Command hook fields
   264Ôćĺ
   265ÔćĺIn addition to the [common fields](#common-fields), command hooks accept these fields:
   266Ôćĺ
   267Ôćĺ| Field     | Required | Description                                                                                                         |
   268Ôćĺ| :-------- | :------- | :------------------------------------------------------------------------------------------------------------------ |
   269Ôćĺ| `command` | yes      | Shell command to execute                                                                                            |
   270Ôćĺ| `async`   | no       | If `true`, runs in the background without blocking. See [Run hooks in the background](#run-hooks-in-the-background) |
   271Ôćĺ
   272Ôćĺ#### Prompt and agent hook fields
   273Ôćĺ
   274ÔćĺIn addition to the [common fields](#common-fields), prompt and agent hooks accept these fields:
   275Ôćĺ
   276Ôćĺ| Field    | Required | Description                                                                                 |
   277Ôćĺ| :------- | :------- | :------------------------------------------------------------------------------------------ |
   278Ôćĺ| `prompt` | yes      | Prompt text to send to the model. Use `$ARGUMENTS` as a placeholder for the hook input JSON |
   279Ôćĺ| `model`  | no       | Model to use for evaluation. Defaults to a fast model                                       |
   280Ôćĺ
   281ÔćĺAll matching hooks run in parallel, and identical handlers are deduplicated automatically. Handlers run in the current directory with Claude Code's environment. The `$CLAUDE_CODE_REMOTE` environment variable is set to `"true"` in remote web environments and not set in the local CLI.
   282Ôćĺ
   283Ôćĺ### Reference scripts by path
   284Ôćĺ
   285ÔćĺUse environment variables to reference hook scripts relative to the project or plugin root, regardless of the working directory when the hook runs:
   286Ôćĺ
   287Ôćĺ* `$CLAUDE_PROJECT_DIR`: the project root. Wrap in quotes to handle paths with spaces.
   288Ôćĺ* `${CLAUDE_PLUGIN_ROOT}`: the plugin's root directory, for scripts bundled with a [plugin](/en/plugins).
   289Ôćĺ
   290Ôćĺ<Tabs>
   291Ôćĺ  <Tab title="Project scripts">
   292Ôćĺ    This example uses `$CLAUDE_PROJECT_DIR` to run a style checker from the project's `.claude/hooks/` directory after any `Write` or `Edit` tool call:
   293Ôćĺ
   294Ôćĺ    ```json  theme={null}
   295Ôćĺ    {
   296Ôćĺ      "hooks": {
   297Ôćĺ        "PostToolUse": [
   298Ôćĺ          {
   299Ôćĺ            "matcher": "Write|Edit",
   300Ôćĺ            "hooks": [
   301Ôćĺ              {
   302Ôćĺ                "type": "command",
   303Ôćĺ                "command": "\"$CLAUDE_PROJECT_DIR\"/.claude/hooks/check-style.sh"
   304Ôćĺ              }
   305Ôćĺ            ]
   306Ôćĺ          }
   307Ôćĺ        ]
   308Ôćĺ      }
   309Ôćĺ    }
   310Ôćĺ    ```
   311Ôćĺ  </Tab>
   312Ôćĺ
   313Ôćĺ  <Tab title="Plugin scripts">
   314Ôćĺ    Define plugin hooks in `hooks/hooks.json` with an optional top-level `description` field. When a plugin is enabled, its hooks merge with your user and project hooks.
   315Ôćĺ
   316Ôćĺ    This example runs a formatting script bundled with the plugin:
   317Ôćĺ
   318Ôćĺ    ```json  theme={null}
   319Ôćĺ    {
   320Ôćĺ      "description": "Automatic code formatting",
   321Ôćĺ      "hooks": {
   322Ôćĺ        "PostToolUse": [
   323Ôćĺ          {
   324Ôćĺ            "matcher": "Write|Edit",
   325Ôćĺ            "hooks": [
   326Ôćĺ              {
   327Ôćĺ                "type": "command",
   328Ôćĺ                "command": "${CLAUDE_PLUGIN_ROOT}/scripts/format.sh",
   329Ôćĺ                "timeout": 30
   330Ôćĺ              }
   331Ôćĺ            ]
   332Ôćĺ          }
   333Ôćĺ        ]
   334Ôćĺ      }
   335Ôćĺ    }
   336Ôćĺ    ```
   337Ôćĺ
   338Ôćĺ    See the [plugin components reference](/en/plugins-reference#hooks) for details on creating plugin hooks.
   339Ôćĺ  </Tab>
   340Ôćĺ</Tabs>
   341Ôćĺ
   342Ôćĺ### Hooks in skills and agents
   343Ôćĺ
   344ÔćĺIn addition to settings files and plugins, hooks can be defined directly in [skills](/en/skills) and [subagents](/en/sub-agents) using frontmatter. These hooks are scoped to the component's lifecycle and only run when that component is active.
   345Ôćĺ
   346ÔćĺAll hook events are supported. For subagents, `Stop` hooks are automatically converted to `SubagentStop` since that is the event that fires when a subagent completes.
   347Ôćĺ
   348ÔćĺHooks use the same configuration format as settings-based hooks but are scoped to the component's lifetime and cleaned up when it finishes.
   349Ôćĺ
   350ÔćĺThis skill defines a `PreToolUse` hook that runs a security validation script before each `Bash` command:
   351Ôćĺ
   352Ôćĺ```yaml  theme={null}
   353Ôćĺ---
   354Ôćĺname: secure-operations
   355Ôćĺdescription: Perform operations with security checks
   356Ôćĺhooks:
   357Ôćĺ  PreToolUse:
   358Ôćĺ    - matcher: "Bash"
   359Ôćĺ      hooks:
   360Ôćĺ        - type: command
   361Ôćĺ          command: "./scripts/security-check.sh"
   362Ôćĺ---
   363Ôćĺ```
   364Ôćĺ
   365ÔćĺAgents use the same format in their YAML frontmatter.
   366Ôćĺ
   367Ôćĺ### The `/hooks` menu
   368Ôćĺ
   369ÔćĺType `/hooks` in Claude Code to open the interactive hooks manager, where you can view, add, and delete hooks without editing settings files directly. For a step-by-step walkthrough, see [Set up your first hook](/en/hooks-guide#set-up-your-first-hook) in the guide.
   370Ôćĺ
   371ÔćĺEach hook in the menu is labeled with a bracket prefix indicating its source:
   372Ôćĺ
   373Ôćĺ* `[User]`: from `~/.claude/settings.json`
   374Ôćĺ* `[Project]`: from `.claude/settings.json`
   375Ôćĺ* `[Local]`: from `.claude/settings.local.json`
   376Ôćĺ* `[Plugin]`: from a plugin's `hooks/hooks.json`, read-only
   377Ôćĺ
   378Ôćĺ### Disable or remove hooks
   379Ôćĺ
   380ÔćĺTo remove a hook, delete its entry from the settings JSON file, or use the `/hooks` menu and select the hook to delete it.
   381Ôćĺ
   382ÔćĺTo temporarily disable all hooks without removing them, set `"disableAllHooks": true` in your settings file or use the toggle in the `/hooks` menu. There is no way to disable an individual hook while keeping it in the configuration.
   383Ôćĺ
   384ÔćĺThe `disableAllHooks` setting respects the managed settings hierarchy. If an administrator has configured hooks through managed policy settings, `disableAllHooks` set in user, project, or local settings cannot disable those managed hooks. Only `disableAllHooks` set at the managed settings level can disable managed hooks.
   385Ôćĺ
   386ÔćĺDirect edits to hooks in settings files don't take effect immediately. Claude Code captures a snapshot of hooks at startup and uses it throughout the session. This prevents malicious or accidental hook modifications from taking effect mid-session without your review. If hooks are modified externally, Claude Code warns you and requires review in the `/hooks` menu before changes apply.
   387Ôćĺ
   388Ôćĺ## Hook input and output
   389Ôćĺ
   390ÔćĺHooks receive JSON data via stdin and communicate results through exit codes, stdout, and stderr. This section covers fields and behavior common to all events. Each event's section under [Hook events](#hook-events) includes its specific input schema and decision control options.
   391Ôćĺ
   392Ôćĺ### Common input fields
   393Ôćĺ
   394ÔćĺAll hook events receive these fields via stdin as JSON, in addition to event-specific fields documented in each [hook event](#hook-events) section:
   395Ôćĺ
   396Ôćĺ| Field             | Description                                                                                                                                |
   397Ôćĺ| :---------------- | :----------------------------------------------------------------------------------------------------------------------------------------- |
   398Ôćĺ| `session_id`      | Current session identifier                                                                                                                 |
   399Ôćĺ| `transcript_path` | Path to conversation JSON                                                                                                                  |
   400Ôćĺ| `cwd`             | Current working directory when the hook is invoked                                                                                         |
   401Ôćĺ| `permission_mode` | Current [permission mode](/en/permissions#permission-modes): `"default"`, `"plan"`, `"acceptEdits"`, `"dontAsk"`, or `"bypassPermissions"` |
   402Ôćĺ| `hook_event_name` | Name of the event that fired                                                                                                               |
   403Ôćĺ
   404ÔćĺFor example, a `PreToolUse` hook for a Bash command receives this on stdin:
   405Ôćĺ
   406Ôćĺ```json  theme={null}
   407Ôćĺ{
   408Ôćĺ  "session_id": "abc123",
   409Ôćĺ  "transcript_path": "/home/user/.claude/projects/.../transcript.jsonl",
   410Ôćĺ  "cwd": "/home/user/my-project",
   411Ôćĺ  "permission_mode": "default",
   412Ôćĺ  "hook_event_name": "PreToolUse",
   413Ôćĺ  "tool_name": "Bash",
   414Ôćĺ  "tool_input": {
   415Ôćĺ    "command": "npm test"
   416Ôćĺ  }
   417Ôćĺ}
   418Ôćĺ```
   419Ôćĺ
   420ÔćĺThe `tool_name` and `tool_input` fields are event-specific. Each [hook event](#hook-events) section documents the additional fields for that event.
   421Ôćĺ
   422Ôćĺ### Exit code output
   423Ôćĺ
   424ÔćĺThe exit code from your hook command tells Claude Code whether the action should proceed, be blocked, or be ignored.
   425Ôćĺ
   426Ôćĺ**Exit 0** means success. Claude Code parses stdout for [JSON output fields](#json-output). JSON output is only processed on exit 0. For most events, stdout is only shown in verbose mode (`Ctrl+O`). The exceptions are `UserPromptSubmit` and `SessionStart`, where stdout is added as context that Claude can see and act on.
   427Ôćĺ
   428Ôćĺ**Exit 2** means a blocking error. Claude Code ignores stdout and any JSON in it. Instead, stderr text is fed back to Claude as an error message. The effect depends on the event: `PreToolUse` blocks the tool call, `UserPromptSubmit` rejects the prompt, and so on. See [exit code 2 behavior](#exit-code-2-behavior-per-event) for the full list.
   429Ôćĺ
   430Ôćĺ**Any other exit code** is a non-blocking error. stderr is shown in verbose mode (`Ctrl+O`) and execution continues.
   431Ôćĺ
   432ÔćĺFor example, a hook command script that blocks dangerous Bash commands:
   433Ôćĺ
   434Ôćĺ```bash  theme={null}
   435Ôćĺ#!/bin/bash
   436Ôćĺ# Reads JSON input from stdin, checks the command
   437Ôćĺcommand=$(jq -r '.tool_input.command' < /dev/stdin)
   438Ôćĺ
   439Ôćĺif [[ "$command" == rm* ]]; then
   440Ôćĺ  echo "Blocked: rm commands are not allowed" >&2
   441Ôćĺ  exit 2  # Blocking error: tool call is prevented
   442Ôćĺfi
   443Ôćĺ
   444Ôćĺexit 0  # Success: tool call proceeds
   445Ôćĺ```
   446Ôćĺ
   447Ôćĺ#### Exit code 2 behavior per event
   448Ôćĺ
   449ÔćĺExit code 2 is the way a hook signals "stop, don't do this." The effect depends on the event, because some events represent actions that can be blocked (like a tool call that hasn't happened yet) and others represent things that already happened or can't be prevented.
   450Ôćĺ
   451Ôćĺ| Hook event           | Can block? | What happens on exit 2                                                        |
   452Ôćĺ| :------------------- | :--------- | :---------------------------------------------------------------------------- |
   453Ôćĺ| `PreToolUse`         | Yes        | Blocks the tool call                                                          |
   454Ôćĺ| `PermissionRequest`  | Yes        | Denies the permission                                                         |
   455Ôćĺ| `UserPromptSubmit`   | Yes        | Blocks prompt processing and erases the prompt                                |
   456Ôćĺ| `Stop`               | Yes        | Prevents Claude from stopping, continues the conversation                     |
   457Ôćĺ| `SubagentStop`       | Yes        | Prevents the subagent from stopping                                           |
   458Ôćĺ| `TeammateIdle`       | Yes        | Prevents the teammate from going idle (teammate continues working)            |
   459Ôćĺ| `TaskCompleted`      | Yes        | Prevents the task from being marked as completed                              |
   460Ôćĺ| `ConfigChange`       | Yes        | Blocks the configuration change from taking effect (except `policy_settings`) |
   461Ôćĺ| `PostToolUse`        | No         | Shows stderr to Claude (tool already ran)                                     |
   462Ôćĺ| `PostToolUseFailure` | No         | Shows stderr to Claude (tool already failed)                                  |
   463Ôćĺ| `Notification`       | No         | Shows stderr to user only                                                     |
   464Ôćĺ| `SubagentStart`      | No         | Shows stderr to user only                                                     |
   465Ôćĺ| `SessionStart`       | No         | Shows stderr to user only                                                     |
   466Ôćĺ| `SessionEnd`         | No         | Shows stderr to user only                                                     |
   467Ôćĺ| `PreCompact`         | No         | Shows stderr to user only                                                     |
   468Ôćĺ| `WorktreeCreate`     | Yes        | Any non-zero exit code causes worktree creation to fail                       |
   469Ôćĺ| `WorktreeRemove`     | No         | Failures are logged in debug mode only                                        |
   470Ôćĺ
   471Ôćĺ### JSON output
   472Ôćĺ
   473ÔćĺExit codes let you allow or block, but JSON output gives you finer-grained control. Instead of exiting with code 2 to block, exit 0 and print a JSON object to stdout. Claude Code reads specific fields from that JSON to control behavior, including [decision control](#decision-control) for blocking, allowing, or escalating to the user.
   474Ôćĺ
   475Ôćĺ<Note>
   476Ôćĺ  You must choose one approach per hook, not both: either use exit codes alone for signaling, or exit 0 and print JSON for structured control. Claude Code only processes JSON on exit 0. If you exit 2, any JSON is ignored.
   477Ôćĺ</Note>
   478Ôćĺ
   479ÔćĺYour hook's stdout must contain only the JSON object. If your shell profile prints text on startup, it can interfere with JSON parsing. See [JSON validation failed](/en/hooks-guide#json-validation-failed) in the troubleshooting guide.
   480Ôćĺ
   481ÔćĺThe JSON object supports three kinds of fields:
   482Ôćĺ
   483Ôćĺ* **Universal fields** like `continue` work across all events. These are listed in the table below.
   484Ôćĺ* **Top-level `decision` and `reason`** are used by some events to block or provide feedback.
   485Ôćĺ* **`hookSpecificOutput`** is a nested object for events that need richer control. It requires a `hookEventName` field set to the event name.
   486Ôćĺ
   487Ôćĺ| Field            | Default | Description                                                                                                                |
   488Ôćĺ| :--------------- | :------ | :------------------------------------------------------------------------------------------------------------------------- |
   489Ôćĺ| `continue`       | `true`  | If `false`, Claude stops processing entirely after the hook runs. Takes precedence over any event-specific decision fields |
   490Ôćĺ| `stopReason`     | none    | Message shown to the user when `continue` is `false`. Not shown to Claude                                                  |
   491Ôćĺ| `suppressOutput` | `false` | If `true`, hides stdout from verbose mode output                                                                           |
   492Ôćĺ| `systemMessage`  | none    | Warning message shown to the user                                                                                          |
   493Ôćĺ
   494ÔćĺTo stop Claude entirely regardless of event type:
   495Ôćĺ
   496Ôćĺ```json  theme={null}
   497Ôćĺ{ "continue": false, "stopReason": "Build failed, fix errors before continuing" }
   498Ôćĺ```
   499Ôćĺ
   500Ôćĺ#### Decision control
   501Ôćĺ
   502ÔćĺNot every event supports blocking or controlling behavior through JSON. The events that do each use a different set of fields to express that decision. Use this table as a quick reference before writing a hook:
   503Ôćĺ
   504Ôćĺ| Events                                                                              | Decision pattern     | Key fields                                                                  |
   505Ôćĺ| :---------------------------------------------------------------------------------- | :------------------- | :-------------------------------------------------------------------------- |
   506Ôćĺ| UserPromptSubmit, PostToolUse, PostToolUseFailure, Stop, SubagentStop, ConfigChange | Top-level `decision` | `decision: "block"`, `reason`                                               |
   507Ôćĺ| TeammateIdle, TaskCompleted                                                         | Exit code only       | Exit code 2 blocks the action, stderr is fed back as feedback               |
   508Ôćĺ| PreToolUse                                                                          | `hookSpecificOutput` | `permissionDecision` (allow/deny/ask), `permissionDecisionReason`           |
   509Ôćĺ| PermissionRequest                                                                   | `hookSpecificOutput` | `decision.behavior` (allow/deny)                                            |
   510Ôćĺ| WorktreeCreate                                                                      | stdout path          | Hook prints absolute path to created worktree. Non-zero exit fails creation |
   511Ôćĺ| WorktreeRemove, Notification, SessionEnd, PreCompact                                | None                 | No decision control. Used for side effects like logging or cleanup          |
   512Ôćĺ
   513ÔćĺHere are examples of each pattern in action:
   514Ôćĺ
   515Ôćĺ<Tabs>
   516Ôćĺ  <Tab title="Top-level decision">
   517Ôćĺ    Used by `UserPromptSubmit`, `PostToolUse`, `PostToolUseFailure`, `Stop`, `SubagentStop`, and `ConfigChange`. The only value is `"block"`. To allow the action to proceed, omit `decision` from your JSON, or exit 0 without any JSON at all:
   518Ôćĺ
   519Ôćĺ    ```json  theme={null}
   520Ôćĺ    {
   521Ôćĺ      "decision": "block",
   522Ôćĺ      "reason": "Test suite must pass before proceeding"
   523Ôćĺ    }
   524Ôćĺ    ```
   525Ôćĺ  </Tab>
   526Ôćĺ
   527Ôćĺ  <Tab title="PreToolUse">
   528Ôćĺ    Uses `hookSpecificOutput` for richer control: allow, deny, or escalate to the user. You can also modify tool input before it runs or inject additional context for Claude. See [PreToolUse decision control](#pretooluse-decision-control) for the full set of options.
   529Ôćĺ
   530Ôćĺ    ```json  theme={null}
   531Ôćĺ    {
   532Ôćĺ      "hookSpecificOutput": {
   533Ôćĺ        "hookEventName": "PreToolUse",
   534Ôćĺ        "permissionDecision": "deny",
   535Ôćĺ        "permissionDecisionReason": "Database writes are not allowed"
   536Ôćĺ      }
   537Ôćĺ    }
   538Ôćĺ    ```
   539Ôćĺ  </Tab>
   540Ôćĺ
   541Ôćĺ  <Tab title="PermissionRequest">
   542Ôćĺ    Uses `hookSpecificOutput` to allow or deny a permission request on behalf of the user. When allowing, you can also modify the tool's input or apply permission rules so the user isn't prompted again. See [PermissionRequest decision control](#permissionrequest-decision-control) for the full set of options.
   543Ôćĺ
   544Ôćĺ    ```json  theme={null}
   545Ôćĺ    {
   546Ôćĺ      "hookSpecificOutput": {
   547Ôćĺ        "hookEventName": "PermissionRequest",
   548Ôćĺ        "decision": {
   549Ôćĺ          "behavior": "allow",
   550Ôćĺ          "updatedInput": {
   551Ôćĺ            "command": "npm run lint"
   552Ôćĺ          }
   553Ôćĺ        }
   554Ôćĺ      }
   555Ôćĺ    }
   556Ôćĺ    ```
   557Ôćĺ  </Tab>
   558Ôćĺ</Tabs>
   559Ôćĺ
   560ÔćĺFor extended examples including Bash command validation, prompt filtering, and auto-approval scripts, see [What you can automate](/en/hooks-guide#what-you-can-automate) in the guide and the [Bash command validator reference implementation](https://github.com/anthropics/claude-code/blob/main/examples/hooks/bash_command_validator_example.py).
   561Ôćĺ
   562Ôćĺ## Hook events
   563Ôćĺ
   564ÔćĺEach event corresponds to a point in Claude Code's lifecycle where hooks can run. The sections below are ordered to match the lifecycle: from session setup through the agentic loop to session end. Each section describes when the event fires, what matchers it supports, the JSON input it receives, and how to control behavior through output.
   565Ôćĺ
   566Ôćĺ### SessionStart
   567Ôćĺ
   568ÔćĺRuns when Claude Code starts a new session or resumes an existing session. Useful for loading development context like existing issues or recent changes to your codebase, or setting up environment variables. For static context that does not require a script, use [CLAUDE.md](/en/memory) instead.
   569Ôćĺ
   570ÔćĺSessionStart runs on every session, so keep these hooks fast.
   571Ôćĺ
   572ÔćĺThe matcher value corresponds to how the session was initiated:
   573Ôćĺ
   574Ôćĺ| Matcher   | When it fires                          |
   575Ôćĺ| :-------- | :------------------------------------- |
   576Ôćĺ| `startup` | New session                            |
   577Ôćĺ| `resume`  | `--resume`, `--continue`, or `/resume` |
   578Ôćĺ| `clear`   | `/clear`                               |
   579Ôćĺ| `compact` | Auto or manual compaction              |
   580Ôćĺ
   581Ôćĺ#### SessionStart input
   582Ôćĺ
   583ÔćĺIn addition to the [common input fields](#common-input-fields), SessionStart hooks receive `source`, `model`, and optionally `agent_type`. The `source` field indicates how the session started: `"startup"` for new sessions, `"resume"` for resumed sessions, `"clear"` after `/clear`, or `"compact"` after compaction. The `model` field contains the model identifier. If you start Claude Code with `claude --agent <name>`, an `agent_type` field contains the agent name.
   584Ôćĺ
   585Ôćĺ```json  theme={null}
   586Ôćĺ{
   587Ôćĺ  "session_id": "abc123",
   588Ôćĺ  "transcript_path": "/Users/.../.claude/projects/.../00893aaf-19fa-41d2-8238-13269b9b3ca0.jsonl",
   589Ôćĺ  "cwd": "/Users/...",
   590Ôćĺ  "permission_mode": "default",
   591Ôćĺ  "hook_event_name": "SessionStart",
   592Ôćĺ  "source": "startup",
   593Ôćĺ  "model": "claude-sonnet-4-6"
   594Ôćĺ}
   595Ôćĺ```
   596Ôćĺ
   597Ôćĺ#### SessionStart decision control
   598Ôćĺ
   599ÔćĺAny text your hook script prints to stdout is added as context for Claude. In addition to the [JSON output fields](#json-output) available to all hooks, you can return these event-specific fields:
   600Ôćĺ
   601Ôćĺ| Field               | Description                                                               |
   602Ôćĺ| :------------------ | :------------------------------------------------------------------------ |
   603Ôćĺ| `additionalContext` | String added to Claude's context. Multiple hooks' values are concatenated |
   604Ôćĺ
   605Ôćĺ```json  theme={null}
   606Ôćĺ{
   607Ôćĺ  "hookSpecificOutput": {
   608Ôćĺ    "hookEventName": "SessionStart",
   609Ôćĺ    "additionalContext": "My additional context here"
   610Ôćĺ  }
   611Ôćĺ}
   612Ôćĺ```
   613Ôćĺ
   614Ôćĺ#### Persist environment variables
   615Ôćĺ
   616ÔćĺSessionStart hooks have access to the `CLAUDE_ENV_FILE` environment variable, which provides a file path where you can persist environment variables for subsequent Bash commands.
   617Ôćĺ
   618ÔćĺTo set individual environment variables, write `export` statements to `CLAUDE_ENV_FILE`. Use append (`>>`) to preserve variables set by other hooks:
   619Ôćĺ
   620Ôćĺ```bash  theme={null}
   621Ôćĺ#!/bin/bash
   622Ôćĺ
   623Ôćĺif [ -n "$CLAUDE_ENV_FILE" ]; then
   624Ôćĺ  echo 'export NODE_ENV=production' >> "$CLAUDE_ENV_FILE"
   625Ôćĺ  echo 'export DEBUG_LOG=true' >> "$CLAUDE_ENV_FILE"
   626Ôćĺ  echo 'export PATH="$PATH:./node_modules/.bin"' >> "$CLAUDE_ENV_FILE"
   627Ôćĺfi
   628Ôćĺ
   629Ôćĺexit 0
   630Ôćĺ```
   631Ôćĺ
   632ÔćĺTo capture all environment changes from setup commands, compare the exported variables before and after:
   633Ôćĺ
   634Ôćĺ```bash  theme={null}
   635Ôćĺ#!/bin/bash
   636Ôćĺ
   637ÔćĺENV_BEFORE=$(export -p | sort)
   638Ôćĺ
   639Ôćĺ# Run your setup commands that modify the environment
   640Ôćĺsource ~/.nvm/nvm.sh
   641Ôćĺnvm use 20
   642Ôćĺ
   643Ôćĺif [ -n "$CLAUDE_ENV_FILE" ]; then
   644Ôćĺ  ENV_AFTER=$(export -p | sort)
   645Ôćĺ  comm -13 <(echo "$ENV_BEFORE") <(echo "$ENV_AFTER") >> "$CLAUDE_ENV_FILE"
   646Ôćĺfi
   647Ôćĺ
   648Ôćĺexit 0
   649Ôćĺ```
   650Ôćĺ
   651ÔćĺAny variables written to this file will be available in all subsequent Bash commands that Claude Code executes during the session.
   652Ôćĺ
   653Ôćĺ<Note>
   654Ôćĺ  `CLAUDE_ENV_FILE` is available for SessionStart hooks. Other hook types do not have access to this variable.
   655Ôćĺ</Note>
   656Ôćĺ
   657Ôćĺ### UserPromptSubmit
   658Ôćĺ
   659ÔćĺRuns when the user submits a prompt, before Claude processes it. This allows you
   660Ôćĺto add additional context based on the prompt/conversation, validate prompts, or
   661Ôćĺblock certain types of prompts.
   662Ôćĺ
   663Ôćĺ#### UserPromptSubmit input
   664Ôćĺ
   665ÔćĺIn addition to the [common input fields](#common-input-fields), UserPromptSubmit hooks receive the `prompt` field containing the text the user submitted.
   666Ôćĺ
   667Ôćĺ```json  theme={null}
   668Ôćĺ{
   669Ôćĺ  "session_id": "abc123",
   670Ôćĺ  "transcript_path": "/Users/.../.claude/projects/.../00893aaf-19fa-41d2-8238-13269b9b3ca0.jsonl",
   671Ôćĺ  "cwd": "/Users/...",
   672Ôćĺ  "permission_mode": "default",
   673Ôćĺ  "hook_event_name": "UserPromptSubmit",
   674Ôćĺ  "prompt": "Write a function to calculate the factorial of a number"
   675Ôćĺ}
   676Ôćĺ```
   677Ôćĺ
   678Ôćĺ#### UserPromptSubmit decision control
   679Ôćĺ
   680Ôćĺ`UserPromptSubmit` hooks can control whether a user prompt is processed and add context. All [JSON output fields](#json-output) are available.
   681Ôćĺ
   682ÔćĺThere are two ways to add context to the conversation on exit code 0:
   683Ôćĺ
   684Ôćĺ* **Plain text stdout**: any non-JSON text written to stdout is added as context
   685Ôćĺ* **JSON with `additionalContext`**: use the JSON format below for more control. The `additionalContext` field is added as context
   686Ôćĺ
   687ÔćĺPlain stdout is shown as hook output in the transcript. The `additionalContext` field is added more discretely.
   688Ôćĺ
   689ÔćĺTo block a prompt, return a JSON object with `decision` set to `"block"`:
   690Ôćĺ
   691Ôćĺ| Field               | Description                                                                                                        |
   692Ôćĺ| :------------------ | :----------------------------------------------------------------------------------------------------------------- |
   693Ôćĺ| `decision`          | `"block"` prevents the prompt from being processed and erases it from context. Omit to allow the prompt to proceed |
   694Ôćĺ| `reason`            | Shown to the user when `decision` is `"block"`. Not added to context                                               |
   695Ôćĺ| `additionalContext` | String added to Claude's context                                                                                   |
   696Ôćĺ
   697Ôćĺ```json  theme={null}
   698Ôćĺ{
   699Ôćĺ  "decision": "block",
   700Ôćĺ  "reason": "Explanation for decision",
   701Ôćĺ  "hookSpecificOutput": {
   702Ôćĺ    "hookEventName": "UserPromptSubmit",
   703Ôćĺ    "additionalContext": "My additional context here"
   704Ôćĺ  }
   705Ôćĺ}
   706Ôćĺ```
   707Ôćĺ
   708Ôćĺ<Note>
   709Ôćĺ  The JSON format isn't required for simple use cases. To add context, you can print plain text to stdout with exit code 0. Use JSON when you need to
   710Ôćĺ  block prompts or want more structured control.
   711Ôćĺ</Note>
   712Ôćĺ
   713Ôćĺ### PreToolUse
   714Ôćĺ
   715ÔćĺRuns after Claude creates tool parameters and before processing the tool call. Matches on tool name: `Bash`, `Edit`, `Write`, `Read`, `Glob`, `Grep`, `Task`, `WebFetch`, `WebSearch`, and any [MCP tool names](#match-mcp-tools).
   716Ôćĺ
   717ÔćĺUse [PreToolUse decision control](#pretooluse-decision-control) to allow, deny, or ask for permission to use the tool.
   718Ôćĺ
   719Ôćĺ#### PreToolUse input
   720Ôćĺ
   721ÔćĺIn addition to the [common input fields](#common-input-fields), PreToolUse hooks receive `tool_name`, `tool_input`, and `tool_use_id`. The `tool_input` fields depend on the tool:
   722Ôćĺ
   723Ôćĺ##### Bash
   724Ôćĺ
   725ÔćĺExecutes shell commands.
   726Ôćĺ
   727Ôćĺ| Field               | Type    | Example            | Description                                   |
   728Ôćĺ| :------------------ | :------ | :----------------- | :-------------------------------------------- |
   729Ôćĺ| `command`           | string  | `"npm test"`       | The shell command to execute                  |
   730Ôćĺ| `description`       | string  | `"Run test suite"` | Optional description of what the command does |
   731Ôćĺ| `timeout`           | number  | `120000`           | Optional timeout in milliseconds              |
   732Ôćĺ| `run_in_background` | boolean | `false`            | Whether to run the command in background      |
   733Ôćĺ
   734Ôćĺ##### Write
   735Ôćĺ
   736ÔćĺCreates or overwrites a file.
   737Ôćĺ
   738Ôćĺ| Field       | Type   | Example               | Description                        |
   739Ôćĺ| :---------- | :----- | :-------------------- | :--------------------------------- |

   753Ôćĺ

   808Ôćĺ

   815Ôćĺ
   816Ôćĺ#### PreToolUse decision control
   817Ôćĺ
   818Ôćĺ`PreToolUse` hooks can control whether a tool call proceeds. Unlike other hooks that use a top-level `decision` field, PreToolUse returns its decision inside a `hookSpecificOutput` object. This gives it richer control: three outcomes (allow, deny, or ask) plus the ability to modify tool input before execution.
   819Ôćĺ
   820Ôćĺ| Field                      | Description                                                                                                                                      |
   821Ôćĺ| :------------------------- | :----------------------------------------------------------------------------------------------------------------------------------------------- |
   822Ôćĺ| `permissionDecision`       | `"allow"` bypasses the permission system, `"deny"` prevents the tool call, `"ask"` prompts the user to confirm                                   |
   823Ôćĺ| `permissionDecisionReason` | For `"allow"` and `"ask"`, shown to the user but not Claude. For `"deny"`, shown to Claude                                                       |
   824Ôćĺ| `updatedInput`             | Modifies the tool's input parameters before execution. Combine with `"allow"` to auto-approve, or `"ask"` to show the modified input to the user |
   825Ôćĺ| `additionalContext`        | String added to Claude's context before the tool executes                                                                                        |
   826Ôćĺ
   827Ôćĺ```json  theme={null}
   828Ôćĺ{
   829Ôćĺ  "hookSpecificOutput": {
   830Ôćĺ    "hookEventName": "PreToolUse",
   831Ôćĺ    "permissionDecision": "allow",
   832Ôćĺ    "permissionDecisionReason": "My reason here",
   833Ôćĺ    "updatedInput": {
   834Ôćĺ      "field_to_modify": "new value"
   835Ôćĺ    },
   836Ôćĺ    "additionalContext": "Current environment: production. Proceed with caution."
   837Ôćĺ  }
   838Ôćĺ}
   839Ôćĺ```
   840Ôćĺ
   841Ôćĺ<Note>
   842Ôćĺ  PreToolUse previously used top-level `decision` and `reason` fields, but these are deprecated for this event. Use `hookSpecificOutput.permissionDecision` and `hookSpecificOutput.permissionDecisionReason` instead. The deprecated values `"approve"` and `"block"` map to `"allow"` and `"deny"` respectively. Other events like PostToolUse and Stop continue to use top-level `decision` and `reason` as their current format.
   843Ôćĺ</Note>
   844Ôćĺ
   845Ôćĺ### PermissionRequest
   846Ôćĺ
   847ÔćĺRuns when the user is shown a permission dialog.
   848ÔćĺUse [PermissionRequest decision control](#permissionrequest-decision-control) to allow or deny on behalf of the user.
   849Ôćĺ
   850ÔćĺMatches on tool name, same values as PreToolUse.
   851Ôćĺ
   852Ôćĺ#### PermissionRequest input
   853Ôćĺ
   854ÔćĺPermissionRequest hooks receive `tool_name` and `tool_input` fields like PreToolUse hooks, but without `tool_use_id`. An optional `permission_suggestions` array contains the "always allow" options the user would normally see in the permission dialog. The difference is when the hook fires: PermissionRequest hooks run when a permission dialog is about to be shown to the user, while PreToolUse hooks run before tool execution regardless of permission status.
   855Ôćĺ
   856Ôćĺ```json  theme={null}
   857Ôćĺ{
   858Ôćĺ  "session_id": "abc123",
   859Ôćĺ  "transcript_path": "/Users/.../.claude/projects/.../00893aaf-19fa-41d2-8238-13269b9b3ca0.jsonl",
   860Ôćĺ  "cwd": "/Users/...",
   861Ôćĺ  "permission_mode": "default",
   862Ôćĺ  "hook_event_name": "PermissionRequest",
   863Ôćĺ  "tool_name": "Bash",
   864Ôćĺ  "tool_input": {
   865Ôćĺ    "command": "rm -rf node_modules",
   866Ôćĺ    "description": "Remove node_modules directory"
   867Ôćĺ  },
   868Ôćĺ  "permission_suggestions": [
   869Ôćĺ    { "type": "toolAlwaysAllow", "tool": "Bash" }
   870Ôćĺ  ]
   871Ôćĺ}
   872Ôćĺ```
   873Ôćĺ
   874Ôćĺ#### PermissionRequest decision control
   875Ôćĺ
   876Ôćĺ`PermissionRequest` hooks can allow or deny permission requests. In addition to the [JSON output fields](#json-output) available to all hooks, your hook script can return a `decision` object with these event-specific fields:
   877Ôćĺ
   878Ôćĺ| Field                | Description                                                                                                    |
   879Ôćĺ| :------------------- | :------------------------------------------------------------------------------------------------------------- |
   880Ôćĺ| `behavior`           | `"allow"` grants the permission, `"deny"` denies it                                                            |
   881Ôćĺ| `updatedInput`       | For `"allow"` only: modifies the tool's input parameters before execution                                      |
   882Ôćĺ| `updatedPermissions` | For `"allow"` only: applies permission rule updates, equivalent to the user selecting an "always allow" option |
   883Ôćĺ| `message`            | For `"deny"` only: tells Claude why the permission was denied                                                  |
   884Ôćĺ| `interrupt`          | For `"deny"` only: if `true`, stops Claude                                                                     |
   885Ôćĺ
   886Ôćĺ```json  theme={null}
   887Ôćĺ{
   888Ôćĺ  "hookSpecificOutput": {
   889Ôćĺ    "hookEventName": "PermissionRequest",
   890Ôćĺ    "decision": {
   891Ôćĺ      "behavior": "allow",
   892Ôćĺ      "updatedInput": {
   893Ôćĺ        "command": "npm run lint"
   894Ôćĺ      }
   895Ôćĺ    }
   896Ôćĺ  }
   897Ôćĺ}
   898Ôćĺ```
   899Ôćĺ
   900Ôćĺ### PostToolUse
   901Ôćĺ
   902ÔćĺRuns immediately after a tool completes successfully.
   903Ôćĺ
   904ÔćĺMatches on tool name, same values as PreToolUse.
   905Ôćĺ
   906Ôćĺ#### PostToolUse input
   907Ôćĺ
   908Ôćĺ`PostToolUse` hooks fire after a tool has already executed successfully. The input includes both `tool_input`, the arguments sent to the tool, and `tool_response`, the result it returned. The exact schema for both depends on the tool.
   909Ôćĺ
   910Ôćĺ```json  theme={null}
   911Ôćĺ{
   912Ôćĺ  "session_id": "abc123",
   913Ôćĺ  "transcript_path": "/Users/.../.claude/projects/.../00893aaf-19fa-41d2-8238-13269b9b3ca0.jsonl",
   914Ôćĺ  "cwd": "/Users/...",
   915Ôćĺ  "permission_mode": "default",
   916Ôćĺ  "hook_event_name": "PostToolUse",
   917Ôćĺ  "tool_name": "Write",
   918Ôćĺ  "tool_input": {
   919Ôćĺ    "file_path": "/path/to/file.txt",
   920Ôćĺ    "content": "file content"
   921Ôćĺ  },
   922Ôćĺ  "tool_response": {
   923Ôćĺ    "filePath": "/path/to/file.txt",
   924Ôćĺ    "success": true
   925Ôćĺ  },
   926Ôćĺ  "tool_use_id": "toolu_01ABC123..."
   927Ôćĺ}
   928Ôćĺ```
   929Ôćĺ
   930Ôćĺ#### PostToolUse decision control
   931Ôćĺ
   932Ôćĺ`PostToolUse` hooks can provide feedback to Claude after tool execution. In addition to the [JSON output fields](#json-output) available to all hooks, your hook script can return these event-specific fields:
   933Ôćĺ
   934Ôćĺ| Field                  | Description                                                                                |
   935Ôćĺ| :--------------------- | :----------------------------------------------------------------------------------------- |
   936Ôćĺ| `decision`             | `"block"` prompts Claude with the `reason`. Omit to allow the action to proceed            |
   937Ôćĺ| `reason`               | Explanation shown to Claude when `decision` is `"block"`                                   |
   938Ôćĺ| `additionalContext`    | Additional context for Claude to consider                                                  |
   939Ôćĺ| `updatedMCPToolOutput` | For [MCP tools](#match-mcp-tools) only: replaces the tool's output with the provided value |
   940Ôćĺ
   941Ôćĺ```json  theme={null}
   942Ôćĺ{
   943Ôćĺ  "decision": "block",
   944Ôćĺ  "reason": "Explanation for decision",
   945Ôćĺ  "hookSpecificOutput": {
   946Ôćĺ    "hookEventName": "PostToolUse",
   947Ôćĺ    "additionalContext": "Additional information for Claude"
   948Ôćĺ  }
   949Ôćĺ}
   950Ôćĺ```
   951Ôćĺ
   952Ôćĺ### PostToolUseFailure

   963Ôćĺ{

   966Ôćĺ  "cwd": "/Users/...",
   967Ôćĺ  "permission_mode": "default",
   968Ôćĺ  "hook_event_name": "PostToolUseFailure",
   969Ôćĺ  "tool_name": "Bash",
   970Ôćĺ  "tool_input": {
   971Ôćĺ    "command": "npm test",

   979Ôćĺ
   980Ôćĺ| Field          | Description                                                                     |
   981Ôćĺ| :------------- | :------------------------------------------------------------------------------ |

   989Ôćĺ| Field               | Description                                                   |

  1002Ôćĺ### Notification
  1003Ôćĺ
  1004ÔćĺRuns when Claude Code sends notifications. Matches on notification type: `permission_prompt`, `idle_prompt`, `auth_success`, `elicitation_dialog`. Omit the matcher to run hooks for all notification types.
  1005Ôćĺ
  1006ÔćĺUse separate matchers to run different handlers depending on the notification type. This configuration triggers a permission-specific alert script when Claude needs permission approval and a different notification when Claude has been idle:
  1007Ôćĺ
  1008Ôćĺ```json  theme={null}
  1009Ôćĺ{
  1010Ôćĺ  "hooks": {
  1011Ôćĺ    "Notification": [
  1012Ôćĺ      {
  1013Ôćĺ        "matcher": "permission_prompt",
  1014Ôćĺ        "hooks": [
  1015Ôćĺ          {
  1016Ôćĺ            "type": "command",
  1017Ôćĺ            "command": "/path/to/permission-alert.sh"
  1018Ôćĺ          }
  1019Ôćĺ        ]
  1020Ôćĺ      },
  1021Ôćĺ      {
  1022Ôćĺ        "matcher": "idle_prompt",
  1023Ôćĺ        "hooks": [
  1024Ôćĺ          {
  1025Ôćĺ            "type": "command",
  1026Ôćĺ            "command": "/path/to/idle-notification.sh"
  1027Ôćĺ          }
  1028Ôćĺ        ]
  1029Ôćĺ      }
  1030Ôćĺ    ]
  1031Ôćĺ  }
  1032Ôćĺ}
  1033Ôćĺ```
  1034Ôćĺ
  1035Ôćĺ#### Notification input
  1036Ôćĺ
  1037ÔćĺIn addition to the [common input fields](#common-input-fields), Notification hooks receive `message` with the notification text, an optional `title`, and `notification_type` indicating which type fired.
  1038Ôćĺ
  1039Ôćĺ```json  theme={null}
  1040Ôćĺ{
  1041Ôćĺ  "session_id": "abc123",
  1042Ôćĺ  "transcript_path": "/Users/.../.claude/projects/.../00893aaf-19fa-41d2-8238-13269b9b3ca0.jsonl",
  1043Ôćĺ  "cwd": "/Users/...",
  1044Ôćĺ  "permission_mode": "default",
  1045Ôćĺ  "hook_event_name": "Notification",
  1046Ôćĺ  "message": "Claude needs your permission to use Bash",
  1047Ôćĺ  "title": "Permission needed",
  1048Ôćĺ  "notification_type": "permission_prompt"
  1049Ôćĺ}
  1050Ôćĺ```
  1051Ôćĺ
  1052ÔćĺNotification hooks cannot block or modify notifications. In addition to the [JSON output fields](#json-output) available to all hooks, you can return `additionalContext` to add context to the conversation:
  1053Ôćĺ
  1054Ôćĺ| Field               | Description                      |
  1055Ôćĺ| :------------------ | :------------------------------- |
  1056Ôćĺ| `additionalContext` | String added to Claude's context |
  1057Ôćĺ
  1058Ôćĺ### SubagentStart
  1059Ôćĺ
  1060ÔćĺRuns when a Claude Code subagent is spawned via the Task tool. Supports matchers to filter by agent type name (built-in agents like `Bash`, `Explore`, `Plan`, or custom agent names from `.claude/agents/`).
  1061Ôćĺ
  1062Ôćĺ#### SubagentStart input
  1063Ôćĺ
  1064ÔćĺIn addition to the [common input fields](#common-input-fields), SubagentStart hooks receive `agent_id` with the unique identifier for the subagent and `agent_type` with the agent name (built-in agents like `"Bash"`, `"Explore"`, `"Plan"`, or custom agent names).
  1065Ôćĺ
  1066Ôćĺ```json  theme={null}

  1070Ôćĺ  "cwd": "/Users/...",
  1071Ôćĺ  "permission_mode": "default",
  1072Ôćĺ  "hook_event_name": "SubagentStart",
  1073Ôćĺ  "agent_id": "agent-abc123",
  1074Ôćĺ  "agent_type": "Explore"
  1075Ôćĺ}
  1076Ôćĺ```
  1077Ôćĺ
  1078ÔćĺSubagentStart hooks cannot block subagent creation, but they can inject context into the subagent. In addition to the [JSON output fields](#json-output) available to all hooks, you can return:
  1079Ôćĺ
  1080Ôćĺ| Field               | Description                            |
  1081Ôćĺ| :------------------ | :------------------------------------- |
  1082Ôćĺ| `additionalContext` | String added to the subagent's context |
  1083Ôćĺ
  1084Ôćĺ```json  theme={null}
  1085Ôćĺ{
  1086Ôćĺ  "hookSpecificOutput": {
  1087Ôćĺ    "hookEventName": "SubagentStart",

  1093Ôćĺ### SubagentStop
  1094Ôćĺ
  1095ÔćĺRuns when a Claude Code subagent has finished responding. Matches on agent type, same values as SubagentStart.
  1096Ôćĺ
  1097Ôćĺ#### SubagentStop input
  1098Ôćĺ
  1099ÔćĺIn addition to the [common input fields](#common-input-fields), SubagentStop hooks receive `stop_hook_active`, `agent_id`, `agent_type`, `agent_transcript_path`, and `last_assistant_message`. The `agent_type` field is the value used for matcher filtering. The `transcript_path` is the main session's transcript, while `agent_transcript_path` is the subagent's own transcript stored in a nested `subagents/` folder. The `last_assistant_message` field contains the text content of the subagent's final response, so hooks can access it without parsing the transcript file.
  1100Ôćĺ
  1101Ôćĺ```json  theme={null}
  1102Ôćĺ{
  1103Ôćĺ  "session_id": "abc123",
  1104Ôćĺ  "transcript_path": "~/.claude/projects/.../abc123.jsonl",
  1105Ôćĺ  "cwd": "/Users/...",
  1106Ôćĺ  "permission_mode": "default",
  1107Ôćĺ  "hook_event_name": "SubagentStop",
  1108Ôćĺ  "stop_hook_active": false,
  1109Ôćĺ  "agent_id": "def456",
  1110Ôćĺ  "agent_type": "Explore",
  1111Ôćĺ  "agent_transcript_path": "~/.claude/projects/.../abc123/subagents/agent-def456.jsonl",

  1113Ôćĺ}
  1114Ôćĺ```
  1115Ôćĺ
  1116ÔćĺSubagentStop hooks use the same decision control format as [Stop hooks](#stop-decision-control).
  1117Ôćĺ
  1118Ôćĺ### Stop
  1119Ôćĺ
  1120ÔćĺRuns when the main Claude Code agent has finished responding. Does not run if
  1121Ôćĺthe stoppage occurred due to a user interrupt.
  1122Ôćĺ
  1123Ôćĺ#### Stop input
  1124Ôćĺ
  1125ÔćĺIn addition to the [common input fields](#common-input-fields), Stop hooks receive `stop_hook_active` and `last_assistant_message`. The `stop_hook_active` field is `true` when Claude Code is already continuing as a result of a stop hook. Check this value or process the transcript to prevent Claude Code from running indefinitely. The `last_assistant_message` field contains the text content of Claude's final response, so hooks can access it without parsing the transcript file.
  1126Ôćĺ
  1127Ôćĺ```json  theme={null}
  1128Ôćĺ{
  1129Ôćĺ  "session_id": "abc123",
  1130Ôćĺ  "transcript_path": "~/.claude/projects/.../00893aaf-19fa-41d2-8238-13269b9b3ca0.jsonl",
  1131Ôćĺ  "cwd": "/Users/...",
  1132Ôćĺ  "permission_mode": "default",
  1133Ôćĺ  "hook_event_name": "Stop",
  1134Ôćĺ  "stop_hook_active": true,
  1135Ôćĺ  "last_assistant_message": "I've completed the refactoring. Here's a summary..."
  1136Ôćĺ}
  1137Ôćĺ```
  1138Ôćĺ
  1139Ôćĺ#### Stop decision control
  1140Ôćĺ
  1141Ôćĺ`Stop` and `SubagentStop` hooks can control whether Claude continues. In addition to the [JSON output fields](#json-output) available to all hooks, your hook script can return these event-specific fields:
  1142Ôćĺ
  1143Ôćĺ| Field      | Description                                                                |
  1144Ôćĺ| :--------- | :------------------------------------------------------------------------- |
  1145Ôćĺ| `decision` | `"block"` prevents Claude from stopping. Omit to allow Claude to stop      |
  1146Ôćĺ| `reason`   | Required when `decision` is `"block"`. Tells Claude why it should continue |
  1147Ôćĺ
  1148Ôćĺ```json  theme={null}
  1149Ôćĺ{
  1150Ôćĺ  "decision": "block",
  1151Ôćĺ  "reason": "Must be provided when Claude is blocked from stopping"
  1152Ôćĺ}
  1153Ôćĺ```

  1155Ôćĺ### TeammateIdle
  1156Ôćĺ
  1157ÔćĺRuns when an [agent team](/en/agent-teams) teammate is about to go idle after finishing its turn. Use this to enforce quality gates before a teammate stops working, such as requiring passing lint checks or verifying that output files exist.
  1158Ôćĺ
  1159ÔćĺWhen a `TeammateIdle` hook exits with code 2, the teammate receives the stderr message as feedback and continues working instead of going idle. TeammateIdle hooks do not support matchers and fire on every occurrence.
  1160Ôćĺ
  1161Ôćĺ#### TeammateIdle input
  1162Ôćĺ
  1163ÔćĺIn addition to the [common input fields](#common-input-fields), TeammateIdle hooks receive `teammate_name` and `team_name`.
  1164Ôćĺ
  1165Ôćĺ```json  theme={null}
  1166Ôćĺ{
  1167Ôćĺ  "session_id": "abc123",
  1168Ôćĺ  "transcript_path": "/Users/.../.claude/projects/.../00893aaf-19fa-41d2-8238-13269b9b3ca0.jsonl",
  1169Ôćĺ  "cwd": "/Users/...",
  1170Ôćĺ  "permission_mode": "default",
  1171Ôćĺ  "hook_event_name": "TeammateIdle",
  1172Ôćĺ  "teammate_name": "researcher",
  1173Ôćĺ  "team_name": "my-project"
  1174Ôćĺ}
  1175Ôćĺ```
  1176Ôćĺ
  1177Ôćĺ| Field           | Description                                   |
  1178Ôćĺ| :-------------- | :-------------------------------------------- |
  1179Ôćĺ| `teammate_name` | Name of the teammate that is about to go idle |
  1180Ôćĺ| `team_name`     | Name of the team                              |
  1181Ôćĺ
  1182Ôćĺ#### TeammateIdle decision control
  1183Ôćĺ
  1184ÔćĺTeammateIdle hooks use exit codes only, not JSON decision control. This example checks that a build artifact exists before allowing a teammate to go idle:
  1185Ôćĺ
  1186Ôćĺ```bash  theme={null}
  1187Ôćĺ#!/bin/bash
  1188Ôćĺ
  1189Ôćĺif [ ! -f "./dist/output.js" ]; then
  1190Ôćĺ  echo "Build artifact missing. Run the build before stopping." >&2
  1191Ôćĺ  exit 2
  1192Ôćĺfi
  1193Ôćĺ
  1194Ôćĺexit 0
  1195Ôćĺ```

  1197Ôćĺ### TaskCompleted
  1198Ôćĺ
  1199ÔćĺRuns when a task is being marked as completed. This fires in two situations: when any agent explicitly marks a task as completed through the TaskUpdate tool, or when an [agent team](/en/agent-teams) teammate finishes its turn with in-progress tasks. Use this to enforce completion criteria like passing tests or lint checks before a task can close.
  1200Ôćĺ
  1201ÔćĺWhen a `TaskCompleted` hook exits with code 2, the task is not marked as completed and the stderr message is fed back to the model as feedback. TaskCompleted hooks do not support matchers and fire on every occurrence.
  1202Ôćĺ
  1203Ôćĺ#### TaskCompleted input
  1204Ôćĺ
  1205ÔćĺIn addition to the [common input fields](#common-input-fields), TaskCompleted hooks receive `task_id`, `task_subject`, and optionally `task_description`, `teammate_name`, and `team_name`.
  1206Ôćĺ
  1207Ôćĺ```json  theme={null}
  1208Ôćĺ{
  1209Ôćĺ  "session_id": "abc123",
  1210Ôćĺ  "transcript_path": "/Users/.../.claude/projects/.../00893aaf-19fa-41d2-8238-13269b9b3ca0.jsonl",
  1211Ôćĺ  "cwd": "/Users/...",
  1212Ôćĺ  "permission_mode": "default",
  1213Ôćĺ  "hook_event_name": "TaskCompleted",
  1214Ôćĺ  "task_id": "task-001",
  1215Ôćĺ  "task_subject": "Implement user authentication",
  1216Ôćĺ  "task_description": "Add login and signup endpoints",
  1217Ôćĺ  "teammate_name": "implementer",
  1218Ôćĺ  "team_name": "my-project"
  1219Ôćĺ}
  1220Ôćĺ```
  1221Ôćĺ
  1222Ôćĺ| Field              | Description                                             |
  1223Ôćĺ| :----------------- | :------------------------------------------------------ |
  1224Ôćĺ| `task_id`          | Identifier of the task being completed                  |
  1225Ôćĺ| `task_subject`     | Title of the task                                       |
  1226Ôćĺ| `task_description` | Detailed description of the task. May be absent         |
  1227Ôćĺ| `teammate_name`    | Name of the teammate completing the task. May be absent |
  1228Ôćĺ| `team_name`        | Name of the team. May be absent                         |
  1229Ôćĺ
  1230Ôćĺ#### TaskCompleted decision control
  1231Ôćĺ
  1232ÔćĺTaskCompleted hooks use exit codes only, not JSON decision control. This example runs tests and blocks task completion if they fail:
  1233Ôćĺ
  1234Ôćĺ```bash  theme={null}
  1235Ôćĺ#!/bin/bash
  1236ÔćĺINPUT=$(cat)
  1237ÔćĺTASK_SUBJECT=$(echo "$INPUT" | jq -r '.task_subject')
  1238Ôćĺ
  1239Ôćĺ# Run the test suite
  1240Ôćĺif ! npm test 2>&1; then
  1241Ôćĺ  echo "Tests not passing. Fix failing tests before completing: $TASK_SUBJECT" >&2
  1242Ôćĺ  exit 2
  1243Ôćĺfi
  1244Ôćĺ
  1245Ôćĺexit 0
  1246Ôćĺ```
  1247Ôćĺ
  1248Ôćĺ### ConfigChange
  1249Ôćĺ
  1250ÔćĺRuns when a configuration file changes during a session. Use this to audit settings changes, enforce security policies, or block unauthorized modifications to configuration files.
  1251Ôćĺ
  1252ÔćĺConfigChange hooks fire for changes to settings files, managed policy settings, and skill files. The `source` field in the input tells you which type of configuration changed, and the optional `file_path` field provides the path to the changed file.
  1253Ôćĺ
  1254ÔćĺThe matcher filters on the configuration source:
  1255Ôćĺ
  1256Ôćĺ| Matcher            | When it fires                             |
  1257Ôćĺ| :----------------- | :---------------------------------------- |
  1258Ôćĺ| `user_settings`    | `~/.claude/settings.json` changes         |
  1259Ôćĺ| `project_settings` | `.claude/settings.json` changes           |
  1260Ôćĺ| `local_settings`   | `.claude/settings.local.json` changes     |
  1261Ôćĺ| `policy_settings`  | Managed policy settings change            |
  1262Ôćĺ| `skills`           | A skill file in `.claude/skills/` changes |
  1263Ôćĺ
  1264ÔćĺThis example logs all configuration changes for security auditing:
  1265Ôćĺ
  1266Ôćĺ```json  theme={null}
  1267Ôćĺ{
  1268Ôćĺ  "hooks": {
  1269Ôćĺ    "ConfigChange": [
  1270Ôćĺ      {
  1271Ôćĺ        "hooks": [
  1272Ôćĺ          {
  1273Ôćĺ            "type": "command",
  1274Ôćĺ            "command": "\"$CLAUDE_PROJECT_DIR\"/.claude/hooks/audit-config-change.sh"
  1275Ôćĺ          }
  1276Ôćĺ        ]
  1277Ôćĺ      }
  1278Ôćĺ    ]
  1279Ôćĺ  }
  1280Ôćĺ}
  1281Ôćĺ```
  1282Ôćĺ
  1283Ôćĺ#### ConfigChange input
  1284Ôćĺ
  1285ÔćĺIn addition to the [common input fields](#common-input-fields), ConfigChange hooks receive `source` and optionally `file_path`. The `source` field indicates which configuration type changed, and `file_path` provides the path to the specific file that was modified.
  1286Ôćĺ
  1287Ôćĺ```json  theme={null}
  1288Ôćĺ{
  1289Ôćĺ  "session_id": "abc123",
  1290Ôćĺ  "transcript_path": "/Users/.../.claude/projects/.../00893aaf-19fa-41d2-8238-13269b9b3ca0.jsonl",
  1291Ôćĺ  "cwd": "/Users/...",
  1292Ôćĺ  "permission_mode": "default",
  1293Ôćĺ  "hook_event_name": "ConfigChange",
  1294Ôćĺ  "source": "project_settings",
  1295Ôćĺ  "file_path": "/Users/.../my-project/.claude/settings.json"
  1296Ôćĺ}
  1297Ôćĺ```
  1298Ôćĺ
  1299Ôćĺ#### ConfigChange decision control
  1300Ôćĺ
  1301ÔćĺConfigChange hooks can block configuration changes from taking effect. Use exit code 2 or a JSON `decision` to prevent the change. When blocked, the new settings are not applied to the running session.
  1302Ôćĺ
  1303Ôćĺ| Field      | Description                                                                              |
  1304Ôćĺ| :--------- | :--------------------------------------------------------------------------------------- |
  1305Ôćĺ| `decision` | `"block"` prevents the configuration change from being applied. Omit to allow the change |
  1306Ôćĺ| `reason`   | Explanation shown to the user when `decision` is `"block"`                               |
  1307Ôćĺ
  1308Ôćĺ```json  theme={null}
  1309Ôćĺ{
  1310Ôćĺ  "decision": "block",
  1311Ôćĺ  "reason": "Configuration changes to project settings require admin approval"
  1312Ôćĺ}
  1313Ôćĺ```
  1314Ôćĺ
  1315Ôćĺ`policy_settings` changes cannot be blocked. Hooks still fire for `policy_settings` sources, so you can use them for audit logging, but any blocking decision is ignored. This ensures enterprise-managed settings always take effect.
  1316Ôćĺ
  1317Ôćĺ### WorktreeCreate
  1318Ôćĺ
  1319ÔćĺWhen you run `claude --worktree` or a [subagent uses `isolation: "worktree"`](/en/sub-agents#choose-the-subagent-scope), Claude Code creates an isolated working copy using `git worktree`. If you configure a WorktreeCreate hook, it replaces the default git behavior, letting you use a different version control system like SVN, Perforce, or Mercurial.
  1320Ôćĺ
  1321ÔćĺThe hook must print the absolute path to the created worktree directory on stdout. Claude Code uses this path as the working directory for the isolated session.
  1322Ôćĺ
  1323ÔćĺThis example creates an SVN working copy and prints the path for Claude Code to use. Replace the repository URL with your own:
  1324Ôćĺ
  1325Ôćĺ```json  theme={null}
  1326Ôćĺ{
  1327Ôćĺ  "hooks": {
  1328Ôćĺ    "WorktreeCreate": [
  1329Ôćĺ      {
  1330Ôćĺ        "hooks": [
  1331Ôćĺ          {
  1332Ôćĺ            "type": "command",
  1333Ôćĺ            "command": "bash -c 'NAME=$(jq -r .name); DIR=\"$HOME/.claude/worktrees/$NAME\"; svn checkout https://svn.example.com/repo/trunk \"$DIR\" >&2 && echo \"$DIR\"'"
  1334Ôćĺ          }
  1335Ôćĺ        ]
  1336Ôćĺ      }
  1337Ôćĺ    ]
  1338Ôćĺ  }
  1339Ôćĺ}
  1340Ôćĺ```
  1341Ôćĺ
  1342ÔćĺThe hook reads the worktree `name` from the JSON input on stdin, checks out a fresh copy into a new directory, and prints the directory path. The `echo` on the last line is what Claude Code reads as the worktree path. Redirect any other output to stderr so it doesn't interfere with the path.
  1343Ôćĺ
  1344Ôćĺ#### WorktreeCreate input
  1345Ôćĺ
  1346ÔćĺIn addition to the [common input fields](#common-input-fields), WorktreeCreate hooks receive the `name` field. This is a slug identifier for the new worktree, either specified by the user or auto-generated (for example, `bold-oak-a3f2`).
  1347Ôćĺ
  1348Ôćĺ```json  theme={null}
  1349Ôćĺ{
  1350Ôćĺ  "session_id": "abc123",
  1351Ôćĺ  "transcript_path": "/Users/.../.claude/projects/.../00893aaf-19fa-41d2-8238-13269b9b3ca0.jsonl",
  1352Ôćĺ  "cwd": "/Users/...",
  1353Ôćĺ  "hook_event_name": "WorktreeCreate",
  1354Ôćĺ  "name": "feature-auth"
  1355Ôćĺ}
  1356Ôćĺ```
  1357Ôćĺ
  1358Ôćĺ#### WorktreeCreate output
  1359Ôćĺ
  1360ÔćĺThe hook must print the absolute path to the created worktree directory on stdout. If the hook fails or produces no output, worktree creation fails with an error.
  1361Ôćĺ
  1362ÔćĺWorktreeCreate hooks do not use the standard allow/block decision model. Instead, the hook's success or failure determines the outcome. Only `type: "command"` hooks are supported.
  1363Ôćĺ
  1364Ôćĺ### WorktreeRemove
  1365Ôćĺ
  1366ÔćĺThe cleanup counterpart to [WorktreeCreate](#worktreecreate). This hook fires when a worktree is being removed, either when you exit a `--worktree` session and choose to remove it, or when a subagent with `isolation: "worktree"` finishes. For git-based worktrees, Claude handles cleanup automatically with `git worktree remove`. If you configured a WorktreeCreate hook for a non-git version control system, pair it with a WorktreeRemove hook to handle cleanup. Without one, the worktree directory is left on disk.
  1367Ôćĺ
  1368ÔćĺClaude Code passes the path that WorktreeCreate printed on stdout as `worktree_path` in the hook input. This example reads that path and removes the directory:
  1369Ôćĺ
  1370Ôćĺ```json  theme={null}
  1371Ôćĺ{
  1372Ôćĺ  "hooks": {
  1373Ôćĺ    "WorktreeRemove": [
  1374Ôćĺ      {
  1375Ôćĺ        "hooks": [
  1376Ôćĺ          {
  1377Ôćĺ            "type": "command",
  1378Ôćĺ            "command": "bash -c 'jq -r .worktree_path | xargs rm -rf'"
  1379Ôćĺ          }
  1380Ôćĺ        ]
  1381Ôćĺ      }
  1382Ôćĺ    ]
  1383Ôćĺ  }
  1384Ôćĺ}
  1385Ôćĺ```
  1386Ôćĺ
  1387Ôćĺ#### WorktreeRemove input
  1388Ôćĺ
  1389ÔćĺIn addition to the [common input fields](#common-input-fields), WorktreeRemove hooks receive the `worktree_path` field, which is the absolute path to the worktree being removed.
  1390Ôćĺ
  1391Ôćĺ```json  theme={null}
  1392Ôćĺ{
  1393Ôćĺ  "session_id": "abc123",
  1394Ôćĺ  "transcript_path": "/Users/.../.claude/projects/.../00893aaf-19fa-41d2-8238-13269b9b3ca0.jsonl",
  1395Ôćĺ  "cwd": "/Users/...",
  1396Ôćĺ  "hook_event_name": "WorktreeRemove",
  1397Ôćĺ  "worktree_path": "/Users/.../my-project/.claude/worktrees/feature-auth"
  1398Ôćĺ}
  1399Ôćĺ```
  1400Ôćĺ
  1401ÔćĺWorktreeRemove hooks have no decision control. They cannot block worktree removal but can perform cleanup tasks like removing version control state or archiving changes. Hook failures are logged in debug mode only. Only `type: "command"` hooks are supported.

  1462Ôćĺ## Prompt-based hooks
  1463Ôćĺ
  1464ÔćĺIn addition to Bash command hooks (`type: "command"`), Claude Code supports prompt-based hooks (`type: "prompt"`) that use an LLM to evaluate whether to allow or block an action, and agent hooks (`type: "agent"`) that spawn an agentic verifier with tool access. Not all events support every hook type.
  1465Ôćĺ
  1466ÔćĺEvents that support all three hook types (`command`, `prompt`, and `agent`):
  1467Ôćĺ
  1468Ôćĺ* `PermissionRequest`
  1469Ôćĺ* `PostToolUse`
  1470Ôćĺ* `PostToolUseFailure`
  1471Ôćĺ* `PreToolUse`
  1472Ôćĺ* `Stop`
  1473Ôćĺ* `SubagentStop`
  1474Ôćĺ* `TaskCompleted`
  1475Ôćĺ* `UserPromptSubmit`
  1476Ôćĺ
  1477ÔćĺEvents that only support `type: "command"` hooks:
  1478Ôćĺ
  1479Ôćĺ* `ConfigChange`
  1480Ôćĺ* `Notification`
  1481Ôćĺ* `PreCompact`
  1482Ôćĺ* `SessionEnd`
  1483Ôćĺ* `SessionStart`
  1484Ôćĺ* `SubagentStart`
  1485Ôćĺ* `TeammateIdle`
  1486Ôćĺ* `WorktreeCreate`
  1487Ôćĺ* `WorktreeRemove`
  1488Ôćĺ
  1489Ôćĺ### How prompt-based hooks work
  1490Ôćĺ
  1491ÔćĺInstead of executing a Bash command, prompt-based hooks:
  1492Ôćĺ
  1493Ôćĺ1. Send the hook input and your prompt to a Claude model, Haiku by default
  1494Ôćĺ2. The LLM responds with structured JSON containing a decision
  1495Ôćĺ3. Claude Code processes the decision automatically
  1496Ôćĺ
  1497Ôćĺ### Prompt hook configuration
  1498Ôćĺ
  1499ÔćĺSet `type` to `"prompt"` and provide a `prompt` string instead of a `command`. Use the `$ARGUMENTS` placeholder to inject the hook's JSON input data into your prompt text. Claude Code sends the combined prompt and input to a fast Claude model, which returns a JSON decision.
  1500Ôćĺ
  1501ÔćĺThis `Stop` hook asks the LLM to evaluate whether all tasks are complete before allowing Claude to finish:
  1502Ôćĺ
  1503Ôćĺ```json  theme={null}
  1504Ôćĺ{
  1505Ôćĺ  "hooks": {
  1506Ôćĺ    "Stop": [
  1507Ôćĺ      {
  1508Ôćĺ        "hooks": [
  1509Ôćĺ          {
  1510Ôćĺ            "type": "prompt",
  1511Ôćĺ            "prompt": "Evaluate if Claude should stop: $ARGUMENTS. Check if all tasks are complete."
  1512Ôćĺ          }
  1513Ôćĺ        ]
  1514Ôćĺ      }
  1515Ôćĺ    ]
  1516Ôćĺ  }
  1517Ôćĺ}
  1518Ôćĺ```
  1519Ôćĺ
  1520Ôćĺ| Field     | Required | Description                                                                                                                                                         |
  1521Ôćĺ| :-------- | :------- | :------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
  1522Ôćĺ| `type`    | yes      | Must be `"prompt"`                                                                                                                                                  |
  1523Ôćĺ| `prompt`  | yes      | The prompt text to send to the LLM. Use `$ARGUMENTS` as a placeholder for the hook input JSON. If `$ARGUMENTS` is not present, input JSON is appended to the prompt |
  1524Ôćĺ| `model`   | no       | Model to use for evaluation. Defaults to a fast model                                                                                                               |
  1525Ôćĺ| `timeout` | no       | Timeout in seconds. Default: 30                                                                                                                                     |
  1526Ôćĺ
  1527Ôćĺ### Response schema
  1528Ôćĺ
  1529ÔćĺThe LLM must respond with JSON containing:
  1530Ôćĺ
  1531Ôćĺ```json  theme={null}
  1532Ôćĺ{
  1533Ôćĺ  "ok": true | false,
  1534Ôćĺ  "reason": "Explanation for the decision"
  1535Ôćĺ}

  1537Ôćĺ
  1538Ôćĺ| Field    | Description                                                |
  1539Ôćĺ| :------- | :--------------------------------------------------------- |
  1540Ôćĺ| `ok`     | `true` allows the action, `false` prevents it              |
  1541Ôćĺ| `reason` | Required when `ok` is `false`. Explanation shown to Claude |
  1542Ôćĺ
  1543Ôćĺ### Example: Multi-criteria Stop hook
  1544Ôćĺ
  1545ÔćĺThis `Stop` hook uses a detailed prompt to check three conditions before allowing Claude to stop. If `"ok"` is `false`, Claude continues working with the provided reason as its next instruction. `SubagentStop` hooks use the same format to evaluate whether a [subagent](/en/sub-agents) should stop:
  1546Ôćĺ
  1547Ôćĺ```json  theme={null}
  1548Ôćĺ{
  1549Ôćĺ  "hooks": {
  1550Ôćĺ    "Stop": [
  1551Ôćĺ      {
  1552Ôćĺ        "hooks": [
  1553Ôćĺ          {
  1554Ôćĺ            "type": "prompt",
  1555Ôćĺ            "prompt": "You are evaluating whether Claude should stop working. Context: $ARGUMENTS\n\nAnalyze the conversation and determine if:\n1. All user-requested tasks are complete\n2. Any errors need to be addressed\n3. Follow-up work is needed\n\nRespond with JSON: {\"ok\": true} to allow stopping, or {\"ok\": false, \"reason\": \"your explanation\"} to continue working.",
  1556Ôćĺ            "timeout": 30
  1557Ôćĺ          }
  1558Ôćĺ        ]
  1559Ôćĺ      }
  1560Ôćĺ    ]
  1561Ôćĺ  }
  1562Ôćĺ}
  1563Ôćĺ```

  1565Ôćĺ## Agent-based hooks
  1566Ôćĺ
  1567ÔćĺAgent-based hooks (`type: "agent"`) are like prompt-based hooks but with multi-turn tool access. Instead of a single LLM call, an agent hook spawns a subagent that can read files, search code, and inspect the codebase to verify conditions. Agent hooks support the same events as prompt-based hooks.
  1568Ôćĺ
  1569Ôćĺ### How agent hooks work
  1570Ôćĺ
  1571ÔćĺWhen an agent hook fires:
  1572Ôćĺ
  1573Ôćĺ1. Claude Code spawns a subagent with your prompt and the hook's JSON input
  1574Ôćĺ2. The subagent can use tools like Read, Grep, and Glob to investigate
  1575Ôćĺ3. After up to 50 turns, the subagent returns a structured `{ "ok": true/false }` decision
  1576Ôćĺ4. Claude Code processes the decision the same way as a prompt hook
  1577Ôćĺ
  1578ÔćĺAgent hooks are useful when verification requires inspecting actual files or test output, not just evaluating the hook input data alone.
  1579Ôćĺ
  1580Ôćĺ### Agent hook configuration
  1581Ôćĺ
  1582ÔćĺSet `type` to `"agent"` and provide a `prompt` string. The configuration fields are the same as [prompt hooks](#prompt-hook-configuration), with a longer default timeout:
  1583Ôćĺ
  1584Ôćĺ| Field     | Required | Description                                                                                 |
  1585Ôćĺ| :-------- | :------- | :------------------------------------------------------------------------------------------ |
  1586Ôćĺ| `type`    | yes      | Must be `"agent"`                                                                           |
  1587Ôćĺ| `prompt`  | yes      | Prompt describing what to verify. Use `$ARGUMENTS` as a placeholder for the hook input JSON |
  1588Ôćĺ| `model`   | no       | Model to use. Defaults to a fast model                                                      |
  1589Ôćĺ| `timeout` | no       | Timeout in seconds. Default: 60                                                             |
  1590Ôćĺ
  1591ÔćĺThe response schema is the same as prompt hooks: `{ "ok": true }` to allow or `{ "ok": false, "reason": "..." }` to block.
  1592Ôćĺ
  1593ÔćĺThis `Stop` hook verifies that all unit tests pass before allowing Claude to finish:
  1594Ôćĺ
  1595Ôćĺ```json  theme={null}
  1596Ôćĺ{
  1597Ôćĺ  "hooks": {
  1598Ôćĺ    "Stop": [
  1599Ôćĺ      {
  1600Ôćĺ        "hooks": [
  1601Ôćĺ          {
  1602Ôćĺ            "type": "agent",
  1603Ôćĺ            "prompt": "Verify that all unit tests pass. Run the test suite and check the results. $ARGUMENTS",
  1604Ôćĺ            "timeout": 120
  1605Ôćĺ          }
  1606Ôćĺ        ]
  1607Ôćĺ      }
  1608Ôćĺ    ]
  1609Ôćĺ  }
  1610Ôćĺ}
  1611Ôćĺ```
  1612Ôćĺ
  1613Ôćĺ## Run hooks in the background
  1614Ôćĺ
  1615ÔćĺBy default, hooks block Claude's execution until they complete. For long-running tasks like deployments, test suites, or external API calls, set `"async": true` to run the hook in the background while Claude continues working. Async hooks cannot block or control Claude's behavior: response fields like `decision`, `permissionDecision`, and `continue` have no effect, because the action they would have controlled has already completed.
  1616Ôćĺ
  1617Ôćĺ### Configure an async hook
  1618Ôćĺ
  1619ÔćĺAdd `"async": true` to a command hook's configuration to run it in the background without blocking Claude. This field is only available on `type: "command"` hooks.
  1620Ôćĺ
  1621ÔćĺThis hook runs a test script after every `Write` tool call. Claude continues working immediately while `run-tests.sh` executes for up to 120 seconds. When the script finishes, its output is delivered on the next conversation turn:
  1622Ôćĺ
  1623Ôćĺ```json  theme={null}
  1624Ôćĺ{
  1625Ôćĺ  "hooks": {
  1626Ôćĺ    "PostToolUse": [
  1627Ôćĺ      {
  1628Ôćĺ        "matcher": "Write",
  1629Ôćĺ        "hooks": [
  1630Ôćĺ          {
  1631Ôćĺ            "type": "command",
  1632Ôćĺ            "command": "/path/to/run-tests.sh",
  1633Ôćĺ            "async": true,
  1634Ôćĺ            "timeout": 120
  1635Ôćĺ          }
  1636Ôćĺ        ]
  1637Ôćĺ      }
  1638Ôćĺ    ]
  1639Ôćĺ  }
  1640Ôćĺ}
  1641Ôćĺ```
  1642Ôćĺ
  1643ÔćĺThe `timeout` field sets the maximum time in seconds for the background process. If not specified, async hooks use the same 10-minute default as sync hooks.
  1644Ôćĺ
  1645Ôćĺ### How async hooks execute
  1646Ôćĺ
  1647ÔćĺWhen an async hook fires, Claude Code starts the hook process and immediately continues without waiting for it to finish. The hook receives the same JSON input via stdin as a synchronous hook.
  1648Ôćĺ
  1649ÔćĺAfter the background process exits, if the hook produced a JSON response with a `systemMessage` or `additionalContext` field, that content is delivered to Claude as context on the next conversation turn.
  1650Ôćĺ
  1651Ôćĺ### Example: run tests after file changes
  1652Ôćĺ
  1653ÔćĺThis hook starts a test suite in the background whenever Claude writes a file, then reports the results back to Claude when the tests finish. Save this script to `.claude/hooks/run-tests-async.sh` in your project and make it executable with `chmod +x`:
  1654Ôćĺ
  1655Ôćĺ```bash  theme={null}
  1656Ôćĺ#!/bin/bash
  1657Ôćĺ# run-tests-async.sh
  1658Ôćĺ
  1659Ôćĺ# Read hook input from stdin
  1660ÔćĺINPUT=$(cat)
  1661ÔćĺFILE_PATH=$(echo "$INPUT" | jq -r '.tool_input.file_path // empty')
  1662Ôćĺ
  1663Ôćĺ# Only run tests for source files
  1664Ôćĺif [[ "$FILE_PATH" != *.ts && "$FILE_PATH" != *.js ]]; then
  1665Ôćĺ  exit 0
  1666Ôćĺfi
  1667Ôćĺ
  1668Ôćĺ# Run tests and report results via systemMessage
  1669ÔćĺRESULT=$(npm test 2>&1)
  1670ÔćĺEXIT_CODE=$?
  1671Ôćĺ
  1672Ôćĺif [ $EXIT_CODE -eq 0 ]; then
  1673Ôćĺ  echo "{\"systemMessage\": \"Tests passed after editing $FILE_PATH\"}"
  1674Ôćĺelse
  1675Ôćĺ  echo "{\"systemMessage\": \"Tests failed after editing $FILE_PATH: $RESULT\"}"
  1676Ôćĺfi
  1677Ôćĺ```
  1678Ôćĺ
  1679ÔćĺThen add this configuration to `.claude/settings.json` in your project root. The `async: true` flag lets Claude keep working while tests run:
  1680Ôćĺ
  1681Ôćĺ```json  theme={null}
  1682Ôćĺ{
  1683Ôćĺ  "hooks": {
  1684Ôćĺ    "PostToolUse": [
  1685Ôćĺ      {
  1686Ôćĺ        "matcher": "Write|Edit",
  1687Ôćĺ        "hooks": [
  1688Ôćĺ          {
  1689Ôćĺ            "type": "command",
  1690Ôćĺ            "command": "\"$CLAUDE_PROJECT_DIR\"/.claude/hooks/run-tests-async.sh",
  1691Ôćĺ            "async": true,
  1692Ôćĺ            "timeout": 300
  1693Ôćĺ          }
  1694Ôćĺ        ]
  1695Ôćĺ      }
  1696Ôćĺ    ]
  1697Ôćĺ  }
  1698Ôćĺ}
  1699Ôćĺ```
  1700Ôćĺ
  1701Ôćĺ### Limitations
  1702Ôćĺ
  1703ÔćĺAsync hooks have several constraints compared to synchronous hooks:
  1704Ôćĺ
  1705Ôćĺ* Only `type: "command"` hooks support `async`. Prompt-based hooks cannot run asynchronously.
  1706Ôćĺ* Async hooks cannot block tool calls or return decisions. By the time the hook completes, the triggering action has already proceeded.
  1707Ôćĺ* Hook output is delivered on the next conversation turn. If the session is idle, the response waits until the next user interaction.
  1708Ôćĺ* Each execution creates a separate background process. There is no deduplication across multiple firings of the same async hook.
  1709Ôćĺ
  1710Ôćĺ## Security considerations
  1711Ôćĺ
  1712Ôćĺ### Disclaimer
  1713Ôćĺ
  1714ÔćĺHooks run with your system user's full permissions.
  1715Ôćĺ
  1716Ôćĺ<Warning>
  1717Ôćĺ  Hooks execute shell commands with your full user permissions. They can modify, delete, or access any files your user account can access. Review and test all hook commands before adding them to your configuration.
  1718Ôćĺ</Warning>
  1719Ôćĺ
  1720Ôćĺ### Security best practices
  1721Ôćĺ
  1722ÔćĺKeep these practices in mind when writing hooks:
  1723Ôćĺ
  1724Ôćĺ* **Validate and sanitize inputs**: never trust input data blindly
  1725Ôćĺ* **Always quote shell variables**: use `"$VAR"` not `$VAR`
  1726Ôćĺ* **Block path traversal**: check for `..` in file paths
  1727Ôćĺ* **Use absolute paths**: specify full paths for scripts, using `"$CLAUDE_PROJECT_DIR"` for the project root
  1728Ôćĺ* **Skip sensitive files**: avoid `.env`, `.git/`, keys, etc.
