     1тЖТ   100тЖТ    { "tool_name": "Bash", "tool_input": { "command": "rm -rf /tmp/build" }, ... }
     2тЖТ   101тЖТ    ```
     3тЖТ   102тЖТ  </Step>
     4тЖТ   103тЖТ
     5тЖТ   104тЖТ  <Step title="Matcher checks">
     6тЖТ   105тЖТ    The matcher `"Bash"` matches the tool name, so `block-rm.sh` runs. If you omit the matcher or use `"*"`, the hook runs on every occurrence of the event. Hooks only skip when a matcher is defined and doesn't match.
     7тЖТ   106тЖТ  </Step>
     8тЖТ   107тЖТ
     9тЖТ   108тЖТ  <Step title="Hook handler runs">
    10тЖТ   109тЖТ    The script extracts `"rm -rf /tmp/build"` from the input and finds `rm -rf`, so it prints a decision to stdout:
    11тЖТ   110тЖТ
    12тЖТ   111тЖТ    ```json  theme={null}
    13тЖТ   112тЖТ    {
    14тЖТ   113тЖТ      "hookSpecificOutput": {
    15тЖТ   114тЖТ        "hookEventName": "PreToolUse",
    16тЖТ   115тЖТ        "permissionDecision": "deny",
    17тЖТ   116тЖТ        "permissionDecisionReason": "Destructive command blocked by hook"
    18тЖТ   117тЖТ      }
    19тЖТ   118тЖТ    }
    20тЖТ   119тЖТ    ```
    21тЖТ   120тЖТ
    22тЖТ   121тЖТ    If the command had been safe (like `npm test`), the script would hit `exit 0` instead, which tells Claude Code to allow the tool call with no further action.
    23тЖТ   122тЖТ  </Step>
    24тЖТ   123тЖТ
    25тЖТ   124тЖТ  <Step title="Claude Code acts on the result">
    26тЖТ   125тЖТ    Claude Code reads the JSON decision, blocks the tool call, and shows Claude the reason.
    27тЖТ   126тЖТ  </Step>
    28тЖТ   127тЖТ</Steps>
    29тЖТ   128тЖТ
    30тЖТ   129тЖТThe [Configuration](#configuration) section below documents the full schema, and each [hook event](#hook-events) section documents what input your command receives and what output it can return.
    31тЖТ   130тЖТ
    32тЖТ   131тЖТ## Configuration
    33тЖТ   132тЖТ
    34тЖТ   133тЖТHooks are defined in JSON settings files. The configuration has three levels of nesting:
    35тЖТ   134тЖТ
    36тЖТ   135тЖТ1. Choose a [hook event](#hook-events) to respond to, like `PreToolUse` or `Stop`
    37тЖТ   136тЖТ2. Add a [matcher group](#matcher-patterns) to filter when it fires, like "only for the Bash tool"
    38тЖТ   137тЖТ3. Define one or more [hook handlers](#hook-handler-fields) to run when matched
    39тЖТ   138тЖТ
    40тЖТ   139тЖТSee [How a hook resolves](#how-a-hook-resolves) above for a complete walkthrough with an annotated example.
    41тЖТ   140тЖТ
    42тЖТ   141тЖТ<Note>
    43тЖТ   142тЖТ  This page uses specific terms for each level: **hook event** for the lifecycle point, **matcher group** for the filter, and **hook handler** for the shell command, prompt, or agent that runs. "Hook" on its own refers to the general feature.
    44тЖТ   143тЖТ</Note>
    45тЖТ   144тЖТ
    46тЖТ   145тЖТ### Hook locations
    47тЖТ   146тЖТ
    48тЖТ   147тЖТWhere you define a hook determines its scope:
    49тЖТ   148тЖТ
    50тЖТ   149тЖТ| Location                                                   | Scope                         | Shareable                          |
    51тЖТ   150тЖТ| :--------------------------------------------------------- | :---------------------------- | :--------------------------------- |
    52тЖТ   151тЖТ| `~/.claude/settings.json`                                  | All your projects             | No, local to your machine          |
    53тЖТ   152тЖТ| `.claude/settings.json`                                    | Single project                | Yes, can be committed to the repo  |
    54тЖТ   153тЖТ| `.claude/settings.local.json`                              | Single project                | No, gitignored                     |
    55тЖТ   154тЖТ| Managed policy settings                                    | Organization-wide             | Yes, admin-controlled              |
    56тЖТ   155тЖТ| [Plugin](/en/plugins) `hooks/hooks.json`                   | When plugin is enabled        | Yes, bundled with the plugin       |
    57тЖТ   156тЖТ| [Skill](/en/skills) or [agent](/en/sub-agents) frontmatter | While the component is active | Yes, defined in the component file |
    58тЖТ   157тЖТ
    59тЖТ   158тЖТFor details on settings file resolution, see [settings](/en/settings). Enterprise administrators can use `allowManagedHooksOnly` to block user, project, and plugin hooks. See [Hook configuration](/en/settings#hook-configuration).
    60тЖТ   159тЖТ
    61тЖТ   160тЖТ### Matcher patterns
    62тЖТ   161тЖТ
    63тЖТ   162тЖТThe `matcher` field is a regex string that filters when hooks fire. Use `"*"`, `""`, or omit `matcher` entirely to match all occurrences. Each event type matches on a different field:
    64тЖТ   163тЖТ
    65тЖТ   164тЖТ| Event                                                                                           | What the matcher filters  | Example matcher values                                                             |
    66тЖТ   165тЖТ| :---------------------------------------------------------------------------------------------- | :------------------------ | :--------------------------------------------------------------------------------- |
    67тЖТ   166тЖТ| `PreToolUse`, `PostToolUse`, `PostToolUseFailure`, `PermissionRequest`                          | tool name                 | `Bash`, `Edit\|Write`, `mcp__.*`                                                   |
    68тЖТ   167тЖТ| `SessionStart`                                                                                  | how the session started   | `startup`, `resume`, `clear`, `compact`                                            |
    69тЖТ   168тЖТ| `SessionEnd`                                                                                    | why the session ended     | `clear`, `logout`, `prompt_input_exit`, `bypass_permissions_disabled`, `other`     |
    70тЖТ   169тЖТ| `Notification`                                                                                  | notification type         | `permission_prompt`, `idle_prompt`, `auth_success`, `elicitation_dialog`           |
    71тЖТ   170тЖТ| `SubagentStart`                                                                                 | agent type                | `Bash`, `Explore`, `Plan`, or custom agent names                                   |
    72тЖТ   171тЖТ| `PreCompact`                                                                                    | what triggered compaction | `manual`, `auto`                                                                   |
    73тЖТ   172тЖТ| `SubagentStop`                                                                                  | agent type                | same values as `SubagentStart`                                                     |
    74тЖТ   173тЖТ| `ConfigChange`                                                                                  | configuration source      | `user_settings`, `project_settings`, `local_settings`, `policy_settings`, `skills` |
    75тЖТ   174тЖТ| `UserPromptSubmit`, `Stop`, `TeammateIdle`, `TaskCompleted`, `WorktreeCreate`, `WorktreeRemove` | no matcher support        | always fires on every occurrence                                                   |
    76тЖТ   175тЖТ
    77тЖТ   176тЖТThe matcher is a regex, so `Edit|Write` matches either tool and `Notebook.*` matches any tool starting with Notebook. The matcher runs against a field from the [JSON input](#hook-input-and-output) that Claude Code sends to your hook on stdin. For tool events, that field is `tool_name`. Each [hook event](#hook-events) section lists the full set of matcher values and the input schema for that event.
    78тЖТ   177тЖТ
    79тЖТ   178тЖТThis example runs a linting script only when Claude writes or edits a file:
    80тЖТ   179тЖТ
    81тЖТ   180тЖТ```json  theme={null}
    82тЖТ   181тЖТ{
    83тЖТ   182тЖТ  "hooks": {
    84тЖТ   183тЖТ    "PostToolUse": [
    85тЖТ   184тЖТ      {
    86тЖТ   185тЖТ        "matcher": "Edit|Write",
    87тЖТ   186тЖТ        "hooks": [
    88тЖТ   187тЖТ          {
    89тЖТ   188тЖТ            "type": "command",
    90тЖТ   189тЖТ            "command": "/path/to/lint-check.sh"
    91тЖТ   190тЖТ          }
    92тЖТ   191тЖТ        ]
    93тЖТ   192тЖТ      }
    94тЖТ   193тЖТ    ]
    95тЖТ   194тЖТ  }
    96тЖТ   195тЖТ}
    97тЖТ   196тЖТ```
    98тЖТ   197тЖТ
    99тЖТ   198тЖТ`UserPromptSubmit`, `Stop`, `TeammateIdle`, `TaskCompleted`, `WorktreeCreate`, and `WorktreeRemove` don't support matchers and always fire on every occurrence. If you add a `matcher` field to these events, it is silently ignored.
   100тЖТ   199тЖТ
   101тЖТ   200тЖТ#### Match MCP tools
   102тЖТ   201тЖТ
   103тЖТ   202тЖТ[MCP](/en/mcp) server tools appear as regular tools in tool events (`PreToolUse`, `PostToolUse`, `PostToolUseFailure`, `PermissionRequest`), so you can match them the same way you match any other tool name.
   104тЖТ   203тЖТ
   105тЖТ   204тЖТMCP tools follow the naming pattern `mcp__<server>__<tool>`, for example:
   106тЖТ   205тЖТ
   107тЖТ   206тЖТ* `mcp__memory__create_entities`: Memory server's create entities tool
   108тЖТ   207тЖТ* `mcp__filesystem__read_file`: Filesystem server's read file tool
   109тЖТ   208тЖТ* `mcp__github__search_repositories`: GitHub server's search tool
   110тЖТ   209тЖТ
   111тЖТ   210тЖТUse regex patterns to target specific MCP tools or groups of tools:
   112тЖТ   211тЖТ
   113тЖТ   212тЖТ* `mcp__memory__.*` matches all tools from the `memory` server
   114тЖТ   213тЖТ* `mcp__.*__write.*` matches any tool containing "write" from any server
   115тЖТ   214тЖТ
   116тЖТ   215тЖТThis example logs all memory server operations and validates write operations from any MCP server:
   117тЖТ   216тЖТ
   118тЖТ   217тЖТ```json  theme={null}
   119тЖТ   218тЖТ{
   120тЖТ   219тЖТ  "hooks": {
   121тЖТ   220тЖТ    "PreToolUse": [
   122тЖТ   221тЖТ      {
   123тЖТ   222тЖТ        "matcher": "mcp__memory__.*",
   124тЖТ   223тЖТ        "hooks": [
   125тЖТ   224тЖТ          {
   126тЖТ   225тЖТ            "type": "command",
   127тЖТ   226тЖТ            "command": "echo 'Memory operation initiated' >> ~/mcp-operations.log"
   128тЖТ   227тЖТ          }
   129тЖТ   228тЖТ        ]
   130тЖТ   229тЖТ      },
   131тЖТ   230тЖТ      {
   132тЖТ   231тЖТ        "matcher": "mcp__.*__write.*",
   133тЖТ   232тЖТ        "hooks": [
   134тЖТ   233тЖТ          {
   135тЖТ   234тЖТ            "type": "command",
   136тЖТ   235тЖТ            "command": "/home/user/scripts/validate-mcp-write.py"
   137тЖТ   236тЖТ          }
   138тЖТ   237тЖТ        ]
   139тЖТ   238тЖТ      }
   140тЖТ   239тЖТ    ]
   141тЖТ   240тЖТ  }
   142тЖТ   241тЖТ}
   143тЖТ   242тЖТ```
   144тЖТ   243тЖТ
   145тЖТ   244тЖТ### Hook handler fields
   146тЖТ   245тЖТ
   147тЖТ   246тЖТEach object in the inner `hooks` array is a hook handler: the shell command, LLM prompt, or agent that runs when the matcher matches. There are three types:
   148тЖТ   247тЖТ
   149тЖТ   248тЖТ* **[Command hooks](#command-hook-fields)** (`type: "command"`): run a shell command. Your script receives the event's [JSON input](#hook-input-and-output) on stdin and communicates results back through exit codes and stdout.
   150тЖТ   249тЖТ* **[Prompt hooks](#prompt-and-agent-hook-fields)** (`type: "prompt"`): send a prompt to a Claude model for single-turn evaluation. The model returns a yes/no decision as JSON. See [Prompt-based hooks](#prompt-based-hooks).
   151тЖТ   250тЖТ* **[Agent hooks](#prompt-and-agent-hook-fields)** (`type: "agent"`): spawn a subagent that can use tools like Read, Grep, and Glob to verify conditions before returning a decision. See [Agent-based hooks](#agent-based-hooks).
   152тЖТ   251тЖТ
   153тЖТ   252тЖТ#### Common fields
   154тЖТ   253тЖТ
   155тЖТ   254тЖТThese fields apply to all hook types:
   156тЖТ   255тЖТ
   157тЖТ   256тЖТ| Field           | Required | Description                                                                                                                                   |
   158тЖТ   257тЖТ| :-------------- | :------- | :-------------------------------------------------------------------------------------------------------------------------------------------- |
   159тЖТ   258тЖТ| `type`          | yes      | `"command"`, `"prompt"`, or `"agent"`                                                                                                         |
   160тЖТ   259тЖТ| `timeout`       | no       | Seconds before canceling. Defaults: 600 for command, 30 for prompt, 60 for agent                                                              |
   161тЖТ   260тЖТ| `statusMessage` | no       | Custom spinner message displayed while the hook runs                                                                                          |
   162тЖТ   261тЖТ| `once`          | no       | If `true`, runs only once per session then is removed. Skills only, not agents. See [Hooks in skills and agents](#hooks-in-skills-and-agents) |
   163тЖТ   262тЖТ
   164тЖТ   263тЖТ#### Command hook fields
   165тЖТ   264тЖТ
   166тЖТ   265тЖТIn addition to the [common fields](#common-fields), command hooks accept these fields:
   167тЖТ   266тЖТ
   168тЖТ   267тЖТ| Field     | Required | Description                                                                                                         |
   169тЖТ   268тЖТ| :-------- | :------- | :------------------------------------------------------------------------------------------------------------------ |
   170тЖТ   269тЖТ| `command` | yes      | Shell command to execute                                                                                            |
   171тЖТ   270тЖТ| `async`   | no       | If `true`, runs in the background without blocking. See [Run hooks in the background](#run-hooks-in-the-background) |
   172тЖТ   271тЖТ
   173тЖТ   272тЖТ#### Prompt and agent hook fields
   174тЖТ   273тЖТ
   175тЖТ   274тЖТIn addition to the [common fields](#common-fields), prompt and agent hooks accept these fields:
   176тЖТ   275тЖТ
   177тЖТ   276тЖТ| Field    | Required | Description                                                                                 |
   178тЖТ   277тЖТ| :------- | :------- | :------------------------------------------------------------------------------------------ |
   179тЖТ   278тЖТ| `prompt` | yes      | Prompt text to send to the model. Use `$ARGUMENTS` as a placeholder for the hook input JSON |
   180тЖТ   279тЖТ| `model`  | no       | Model to use for evaluation. Defaults to a fast model                                       |
   181тЖТ   280тЖТ
   182тЖТ   281тЖТAll matching hooks run in parallel, and identical handlers are deduplicated automatically. Handlers run in the current directory with Claude Code's environment. The `$CLAUDE_CODE_REMOTE` environment variable is set to `"true"` in remote web environments and not set in the local CLI.
   183тЖТ   282тЖТ
   184тЖТ   283тЖТ### Reference scripts by path
   185тЖТ   284тЖТ
   186тЖТ   285тЖТUse environment variables to reference hook scripts relative to the project or plugin root, regardless of the working directory when the hook runs:
   187тЖТ   286тЖТ
   188тЖТ   287тЖТ* `$CLAUDE_PROJECT_DIR`: the project root. Wrap in quotes to handle paths with spaces.
   189тЖТ   288тЖТ* `${CLAUDE_PLUGIN_ROOT}`: the plugin's root directory, for scripts bundled with a [plugin](/en/plugins).
   190тЖТ   289тЖТ
   191тЖТ   290тЖТ<Tabs>
   192тЖТ   291тЖТ  <Tab title="Project scripts">
   193тЖТ   292тЖТ    This example uses `$CLAUDE_PROJECT_DIR` to run a style checker from the project's `.claude/hooks/` directory after any `Write` or `Edit` tool call:
   194тЖТ   293тЖТ
   195тЖТ   294тЖТ    ```json  theme={null}
   196тЖТ   295тЖТ    {
   197тЖТ   296тЖТ      "hooks": {
   198тЖТ   297тЖТ        "PostToolUse": [
   199тЖТ   298тЖТ          {
   200тЖТ   299тЖТ            "matcher": "Write|Edit",
   201тЖТ   300тЖТ            "hooks": [
   202тЖТ   301тЖТ              {
   203тЖТ   302тЖТ                "type": "command",
   204тЖТ   303тЖТ                "command": "\"$CLAUDE_PROJECT_DIR\"/.claude/hooks/check-style.sh"
   205тЖТ   304тЖТ              }
   206тЖТ   305тЖТ            ]
   207тЖТ   306тЖТ          }
   208тЖТ   307тЖТ        ]
   209тЖТ   308тЖТ      }
   210тЖТ   309тЖТ    }
   211тЖТ   310тЖТ    ```
   212тЖТ   311тЖТ  </Tab>
   213тЖТ   312тЖТ
   214тЖТ   313тЖТ  <Tab title="Plugin scripts">
   215тЖТ   314тЖТ    Define plugin hooks in `hooks/hooks.json` with an optional top-level `description` field. When a plugin is enabled, its hooks merge with your user and project hooks.
   216тЖТ   315тЖТ
   217тЖТ   316тЖТ    This example runs a formatting script bundled with the plugin:
   218тЖТ   317тЖТ
   219тЖТ   318тЖТ    ```json  theme={null}
   220тЖТ   319тЖТ    {
   221тЖТ   320тЖТ      "description": "Automatic code formatting",
   222тЖТ   321тЖТ      "hooks": {
   223тЖТ   322тЖТ        "PostToolUse": [
   224тЖТ   323тЖТ          {
   225тЖТ   324тЖТ            "matcher": "Write|Edit",
   226тЖТ   325тЖТ            "hooks": [
   227тЖТ   326тЖТ              {
   228тЖТ   327тЖТ                "type": "command",
   229тЖТ   328тЖТ                "command": "${CLAUDE_PLUGIN_ROOT}/scripts/format.sh",
   230тЖТ   329тЖТ                "timeout": 30
   231тЖТ   330тЖТ              }
   232тЖТ   331тЖТ            ]
   233тЖТ   332тЖТ          }
   234тЖТ   333тЖТ        ]
   235тЖТ   334тЖТ      }
   236тЖТ   335тЖТ    }
   237тЖТ   336тЖТ    ```
   238тЖТ   337тЖТ
   239тЖТ   338тЖТ    See the [plugin components reference](/en/plugins-reference#hooks) for details on creating plugin hooks.
   240тЖТ   339тЖТ  </Tab>
   241тЖТ   340тЖТ</Tabs>
   242тЖТ   341тЖТ
   243тЖТ   342тЖТ### Hooks in skills and agents
   244тЖТ   343тЖТ
   245тЖТ   344тЖТIn addition to settings files and plugins, hooks can be defined directly in [skills](/en/skills) and [subagents](/en/sub-agents) using frontmatter. These hooks are scoped to the component's lifecycle and only run when that component is active.
   246тЖТ   345тЖТ
   247тЖТ   346тЖТAll hook events are supported. For subagents, `Stop` hooks are automatically converted to `SubagentStop` since that is the event that fires when a subagent completes.
   248тЖТ   347тЖТ
   249тЖТ   348тЖТHooks use the same configuration format as settings-based hooks but are scoped to the component's lifetime and cleaned up when it finishes.
   250тЖТ   349тЖТ
   251тЖТ   350тЖТThis skill defines a `PreToolUse` hook that runs a security validation script before each `Bash` command:
   252тЖТ   351тЖТ
   253тЖТ   352тЖТ```yaml  theme={null}
   254тЖТ   353тЖТ---
   255тЖТ   354тЖТname: secure-operations
   256тЖТ   355тЖТdescription: Perform operations with security checks
   257тЖТ   356тЖТhooks:
   258тЖТ   357тЖТ  PreToolUse:
   259тЖТ   358тЖТ    - matcher: "Bash"
   260тЖТ   359тЖТ      hooks:
   261тЖТ   360тЖТ        - type: command
   262тЖТ   361тЖТ          command: "./scripts/security-check.sh"
   263тЖТ   362тЖТ---
   264тЖТ   363тЖТ```
   265тЖТ   364тЖТ
   266тЖТ   365тЖТAgents use the same format in their YAML frontmatter.
   267тЖТ   366тЖТ
   268тЖТ   367тЖТ### The `/hooks` menu
   269тЖТ   368тЖТ
   270тЖТ   369тЖТType `/hooks` in Claude Code to open the interactive hooks manager, where you can view, add, and delete hooks without editing settings files directly. For a step-by-step walkthrough, see [Set up your first hook](/en/hooks-guide#set-up-your-first-hook) in the guide.
   271тЖТ   370тЖТ
   272тЖТ   371тЖТEach hook in the menu is labeled with a bracket prefix indicating its source:
   273тЖТ   372тЖТ
   274тЖТ   373тЖТ* `[User]`: from `~/.claude/settings.json`
   275тЖТ   374тЖТ* `[Project]`: from `.claude/settings.json`
   276тЖТ   375тЖТ* `[Local]`: from `.claude/settings.local.json`
   277тЖТ   376тЖТ* `[Plugin]`: from a plugin's `hooks/hooks.json`, read-only
   278тЖТ   377тЖТ
   279тЖТ   378тЖТ### Disable or remove hooks
   280тЖТ   379тЖТ
   281тЖТ   380тЖТTo remove a hook, delete its entry from the settings JSON file, or use the `/hooks` menu and select the hook to delete it.
   282тЖТ   381тЖТ
   283тЖТ   382тЖТTo temporarily disable all hooks without removing them, set `"disableAllHooks": true` in your settings file or use the toggle in the `/hooks` menu. There is no way to disable an individual hook while keeping it in the configuration.
   284тЖТ   383тЖТ
   285тЖТ   384тЖТThe `disableAllHooks` setting respects the managed settings hierarchy. If an administrator has configured hooks through managed policy settings, `disableAllHooks` set in user, project, or local settings cannot disable those managed hooks. Only `disableAllHooks` set at the managed settings level can disable managed hooks.
   286тЖТ   385тЖТ
   287тЖТ   386тЖТDirect edits to hooks in settings files don't take effect immediately. Claude Code captures a snapshot of hooks at startup and uses it throughout the session. This prevents malicious or accidental hook modifications from taking effect mid-session without your review. If hooks are modified externally, Claude Code warns you and requires review in the `/hooks` menu before changes apply.
   288тЖТ   387тЖТ
   289тЖТ   388тЖТ## Hook input and output
   290тЖТ   389тЖТ
   291тЖТ   390тЖТHooks receive JSON data via stdin and communicate results through exit codes, stdout, and stderr. This section covers fields and behavior common to all events. Each event's section under [Hook events](#hook-events) includes its specific input schema and decision control options.
   292тЖТ   391тЖТ
   293тЖТ   392тЖТ### Common input fields
   294тЖТ   393тЖТ
   295тЖТ   394тЖТAll hook events receive these fields via stdin as JSON, in addition to event-specific fields documented in each [hook event](#hook-events) section:
   296тЖТ   395тЖТ
   297тЖТ   396тЖТ| Field             | Description                                                                                                                                |
   298тЖТ   397тЖТ| :---------------- | :----------------------------------------------------------------------------------------------------------------------------------------- |
   299тЖТ   398тЖТ| `session_id`      | Current session identifier                                                                                                                 |
   300тЖТ   399тЖТ| `transcript_path` | Path to conversation JSON                                                                                                                  |
   301тЖТ   400тЖТ| `cwd`             | Current working directory when the hook is invoked                                                                                         |
   302тЖТ   401тЖТ| `permission_mode` | Current [permission mode](/en/permissions#permission-modes): `"default"`, `"plan"`, `"acceptEdits"`, `"dontAsk"`, or `"bypassPermissions"` |
   303тЖТ   402тЖТ| `hook_event_name` | Name of the event that fired                                                                                                               |
   304тЖТ   403тЖТ
   305тЖТ   404тЖТFor example, a `PreToolUse` hook for a Bash command receives this on stdin:
   306тЖТ   405тЖТ
   307тЖТ   406тЖТ```json  theme={null}
   308тЖТ   407тЖТ{
   309тЖТ   408тЖТ  "session_id": "abc123",
   310тЖТ   409тЖТ  "transcript_path": "/home/user/.claude/projects/.../transcript.jsonl",
   311тЖТ   410тЖТ  "cwd": "/home/user/my-project",
   312тЖТ   411тЖТ  "permission_mode": "default",
   313тЖТ   412тЖТ  "hook_event_name": "PreToolUse",
   314тЖТ   413тЖТ  "tool_name": "Bash",
   315тЖТ   414тЖТ  "tool_input": {
   316тЖТ   415тЖТ    "command": "npm test"
   317тЖТ   416тЖТ  }
   318тЖТ   417тЖТ}
   319тЖТ   418тЖТ```
   320тЖТ   419тЖТ
   321тЖТ   420тЖТThe `tool_name` and `tool_input` fields are event-specific. Each [hook event](#hook-events) section documents the additional fields for that event.
   322тЖТ   421тЖТ
   323тЖТ   422тЖТ### Exit code output
   324тЖТ   423тЖТ
   325тЖТ   424тЖТThe exit code from your hook command tells Claude Code whether the action should proceed, be blocked, or be ignored.
   326тЖТ   425тЖТ
   327тЖТ   426тЖТ**Exit 0** means success. Claude Code parses stdout for [JSON output fields](#json-output). JSON output is only processed on exit 0. For most events, stdout is only shown in verbose mode (`Ctrl+O`). The exceptions are `UserPromptSubmit` and `SessionStart`, where stdout is added as context that Claude can see and act on.
   328тЖТ   427тЖТ
   329тЖТ   428тЖТ**Exit 2** means a blocking error. Claude Code ignores stdout and any JSON in it. Instead, stderr text is fed back to Claude as an error message. The effect depends on the event: `PreToolUse` blocks the tool call, `UserPromptSubmit` rejects the prompt, and so on. See [exit code 2 behavior](#exit-code-2-behavior-per-event) for the full list.
   330тЖТ   429тЖТ
   331тЖТ   430тЖТ**Any other exit code** is a non-blocking error. stderr is shown in verbose mode (`Ctrl+O`) and execution continues.
   332тЖТ   431тЖТ
   333тЖТ   432тЖТFor example, a hook command script that blocks dangerous Bash commands:
   334тЖТ   433тЖТ
   335тЖТ   434тЖТ```bash  theme={null}
   336тЖТ   435тЖТ#!/bin/bash
   337тЖТ   436тЖТ# Reads JSON input from stdin, checks the command
   338тЖТ   437тЖТcommand=$(jq -r '.tool_input.command' < /dev/stdin)
   339тЖТ   438тЖТ
   340тЖТ   439тЖТif [[ "$command" == rm* ]]; then
   341тЖТ   440тЖТ  echo "Blocked: rm commands are not allowed" >&2
   342тЖТ   441тЖТ  exit 2  # Blocking error: tool call is prevented
   343тЖТ   442тЖТfi
   344тЖТ   443тЖТ
   345тЖТ   444тЖТexit 0  # Success: tool call proceeds
   346тЖТ   445тЖТ```
   347тЖТ   446тЖТ
   348тЖТ   447тЖТ#### Exit code 2 behavior per event
   349тЖТ   448тЖТ
   350тЖТ   449тЖТExit code 2 is the way a hook signals "stop, don't do this." The effect depends on the event, because some events represent actions that can be blocked (like a tool call that hasn't happened yet) and others represent things that already happened or can't be prevented.
   351тЖТ   450тЖТ
   352тЖТ   451тЖТ| Hook event           | Can block? | What happens on exit 2                                                        |
   353тЖТ   452тЖТ| :------------------- | :--------- | :---------------------------------------------------------------------------- |
   354тЖТ   453тЖТ| `PreToolUse`         | Yes        | Blocks the tool call                                                          |
   355тЖТ   454тЖТ| `PermissionRequest`  | Yes        | Denies the permission                                                         |
   356тЖТ   455тЖТ| `UserPromptSubmit`   | Yes        | Blocks prompt processing and erases the prompt                                |
   357тЖТ   456тЖТ| `Stop`               | Yes        | Prevents Claude from stopping, continues the conversation                     |
   358тЖТ   457тЖТ| `SubagentStop`       | Yes        | Prevents the subagent from stopping                                           |
   359тЖТ   458тЖТ| `TeammateIdle`       | Yes        | Prevents the teammate from going idle (teammate continues working)            |
   360тЖТ   459тЖТ| `TaskCompleted`      | Yes        | Prevents the task from being marked as completed                              |
   361тЖТ   460тЖТ| `ConfigChange`       | Yes        | Blocks the configuration change from taking effect (except `policy_settings`) |
   362тЖТ   461тЖТ| `PostToolUse`        | No         | Shows stderr to Claude (tool already ran)                                     |
   363тЖТ   462тЖТ| `PostToolUseFailure` | No         | Shows stderr to Claude (tool already failed)                                  |
   364тЖТ   463тЖТ| `Notification`       | No         | Shows stderr to user only                                                     |
   365тЖТ   464тЖТ| `SubagentStart`      | No         | Shows stderr to user only                                                     |
   366тЖТ   465тЖТ| `SessionStart`       | No         | Shows stderr to user only                                                     |
   367тЖТ   466тЖТ| `SessionEnd`         | No         | Shows stderr to user only                                                     |
   368тЖТ   467тЖТ| `PreCompact`         | No         | Shows stderr to user only                                                     |
   369тЖТ   468тЖТ| `WorktreeCreate`     | Yes        | Any non-zero exit code causes worktree creation to fail                       |
   370тЖТ   469тЖТ| `WorktreeRemove`     | No         | Failures are logged in debug mode only                                        |
   371тЖТ   470тЖТ
   372тЖТ   471тЖТ### JSON output
   373тЖТ   472тЖТ
   374тЖТ   473тЖТExit codes let you allow or block, but JSON output gives you finer-grained control. Instead of exiting with code 2 to block, exit 0 and print a JSON object to stdout. Claude Code reads specific fields from that JSON to control behavior, including [decision control](#decision-control) for blocking, allowing, or escalating to the user.
   375тЖТ   474тЖТ
   376тЖТ   475тЖТ<Note>
   377тЖТ   476тЖТ  You must choose one approach per hook, not both: either use exit codes alone for signaling, or exit 0 and print JSON for structured control. Claude Code only processes JSON on exit 0. If you exit 2, any JSON is ignored.
   378тЖТ   477тЖТ</Note>
   379тЖТ   478тЖТ
   380тЖТ   479тЖТYour hook's stdout must contain only the JSON object. If your shell profile prints text on startup, it can interfere with JSON parsing. See [JSON validation failed](/en/hooks-guide#json-validation-failed) in the troubleshooting guide.
   381тЖТ   480тЖТ
   382тЖТ   481тЖТThe JSON object supports three kinds of fields:
   383тЖТ   482тЖТ
   384тЖТ   483тЖТ* **Universal fields** like `continue` work across all events. These are listed in the table below.
   385тЖТ   484тЖТ* **Top-level `decision` and `reason`** are used by some events to block or provide feedback.
   386тЖТ   485тЖТ* **`hookSpecificOutput`** is a nested object for events that need richer control. It requires a `hookEventName` field set to the event name.
   387тЖТ   486тЖТ
   388тЖТ   487тЖТ| Field            | Default | Description                                                                                                                |
   389тЖТ   488тЖТ| :--------------- | :------ | :------------------------------------------------------------------------------------------------------------------------- |
   390тЖТ   489тЖТ| `continue`       | `true`  | If `false`, Claude stops processing entirely after the hook runs. Takes precedence over any event-specific decision fields |
   391тЖТ   490тЖТ| `stopReason`     | none    | Message shown to the user when `continue` is `false`. Not shown to Claude                                                  |
   392тЖТ   491тЖТ| `suppressOutput` | `false` | If `true`, hides stdout from verbose mode output                                                                           |
   393тЖТ   492тЖТ| `systemMessage`  | none    | Warning message shown to the user                                                                                          |
   394тЖТ   493тЖТ
   395тЖТ   494тЖТTo stop Claude entirely regardless of event type:
   396тЖТ   495тЖТ
   397тЖТ   496тЖТ```json  theme={null}
   398тЖТ   497тЖТ{ "continue": false, "stopReason": "Build failed, fix errors before continuing" }
   399тЖТ   498тЖТ```
   400тЖТ   499тЖТ
   401тЖТ   500тЖТ#### Decision control
   402тЖТ   501тЖТ
   403тЖТ   502тЖТNot every event supports blocking or controlling behavior through JSON. The events that do each use a different set of fields to express that decision. Use this table as a quick reference before writing a hook:
   404тЖТ   503тЖТ
   405тЖТ   504тЖТ| Events                                                                              | Decision pattern     | Key fields                                                                  |
   406тЖТ   505тЖТ| :---------------------------------------------------------------------------------- | :------------------- | :-------------------------------------------------------------------------- |
   407тЖТ   506тЖТ| UserPromptSubmit, PostToolUse, PostToolUseFailure, Stop, SubagentStop, ConfigChange | Top-level `decision` | `decision: "block"`, `reason`                                               |
   408тЖТ   507тЖТ| TeammateIdle, TaskCompleted                                                         | Exit code only       | Exit code 2 blocks the action, stderr is fed back as feedback               |
   409тЖТ   508тЖТ| PreToolUse                                                                          | `hookSpecificOutput` | `permissionDecision` (allow/deny/ask), `permissionDecisionReason`           |
   410тЖТ   509тЖТ| PermissionRequest                                                                   | `hookSpecificOutput` | `decision.behavior` (allow/deny)                                            |
   411тЖТ   510тЖТ| WorktreeCreate                                                                      | stdout path          | Hook prints absolute path to created worktree. Non-zero exit fails creation |
   412тЖТ   511тЖТ| WorktreeRemove, Notification, SessionEnd, PreCompact                                | None                 | No decision control. Used for side effects like logging or cleanup          |
   413тЖТ   512тЖТ
   414тЖТ   513тЖТHere are examples of each pattern in action:
   415тЖТ   514тЖТ
   416тЖТ   515тЖТ<Tabs>
   417тЖТ   516тЖТ  <Tab title="Top-level decision">
   418тЖТ   517тЖТ    Used by `UserPromptSubmit`, `PostToolUse`, `PostToolUseFailure`, `Stop`, `SubagentStop`, and `ConfigChange`. The only value is `"block"`. To allow the action to proceed, omit `decision` from your JSON, or exit 0 without any JSON at all:
   419тЖТ   518тЖТ
   420тЖТ   519тЖТ    ```json  theme={null}
   421тЖТ   520тЖТ    {
   422тЖТ   521тЖТ      "decision": "block",
   423тЖТ   522тЖТ      "reason": "Test suite must pass before proceeding"
   424тЖТ   523тЖТ    }
   425тЖТ   524тЖТ    ```
   426тЖТ   525тЖТ  </Tab>
   427тЖТ   526тЖТ
   428тЖТ   527тЖТ  <Tab title="PreToolUse">
   429тЖТ   528тЖТ    Uses `hookSpecificOutput` for richer control: allow, deny, or escalate to the user. You can also modify tool input before it runs or inject additional context for Claude. See [PreToolUse decision control](#pretooluse-decision-control) for the full set of options.
   430тЖТ   529тЖТ
   431тЖТ   530тЖТ    ```json  theme={null}
   432тЖТ   531тЖТ    {
   433тЖТ   532тЖТ      "hookSpecificOutput": {
   434тЖТ   533тЖТ        "hookEventName": "PreToolUse",
   435тЖТ   534тЖТ        "permissionDecision": "deny",
   436тЖТ   535тЖТ        "permissionDecisionReason": "Database writes are not allowed"
   437тЖТ   536тЖТ      }
   438тЖТ   537тЖТ    }
   439тЖТ   538тЖТ    ```
   440тЖТ   539тЖТ  </Tab>
   441тЖТ   540тЖТ
   442тЖТ   541тЖТ  <Tab title="PermissionRequest">
   443тЖТ   542тЖТ    Uses `hookSpecificOutput` to allow or deny a permission request on behalf of the user. When allowing, you can also modify the tool's input or apply permission rules so the user isn't prompted again. See [PermissionRequest decision control](#permissionrequest-decision-control) for the full set of options.
   444тЖТ   543тЖТ
   445тЖТ   544тЖТ    ```json  theme={null}
   446тЖТ   545тЖТ    {
   447тЖТ   546тЖТ      "hookSpecificOutput": {
   448тЖТ   547тЖТ        "hookEventName": "PermissionRequest",
   449тЖТ   548тЖТ        "decision": {
   450тЖТ   549тЖТ          "behavior": "allow",
   451тЖТ   550тЖТ          "updatedInput": {
   452тЖТ   551тЖТ            "command": "npm run lint"
   453тЖТ   552тЖТ          }
   454тЖТ   553тЖТ        }
   455тЖТ   554тЖТ      }
   456тЖТ   555тЖТ    }
   457тЖТ   556тЖТ    ```
   458тЖТ   557тЖТ  </Tab>
   459тЖТ   558тЖТ</Tabs>
   460тЖТ   559тЖТ
   461тЖТ   560тЖТFor extended examples including Bash command validation, prompt filtering, and auto-approval scripts, see [What you can automate](/en/hooks-guide#what-you-can-automate) in the guide and the [Bash command validator reference implementation](https://github.com/anthropics/claude-code/blob/main/examples/hooks/bash_command_validator_example.py).
   462тЖТ   561тЖТ
   463тЖТ   562тЖТ## Hook events
   464тЖТ   563тЖТ
   465тЖТ   564тЖТEach event corresponds to a point in Claude Code's lifecycle where hooks can run. The sections below are ordered to match the lifecycle: from session setup through the agentic loop to session end. Each section describes when the event fires, what matchers it supports, the JSON input it receives, and how to control behavior through output.
   466тЖТ   565тЖТ
   467тЖТ   566тЖТ### SessionStart
   468тЖТ   567тЖТ
   469тЖТ   568тЖТRuns when Claude Code starts a new session or resumes an existing session. Useful for loading development context like existing issues or recent changes to your codebase, or setting up environment variables. For static context that does not require a script, use [CLAUDE.md](/en/memory) instead.
   470тЖТ   569тЖТ
   471тЖТ   570тЖТSessionStart runs on every session, so keep these hooks fast.
   472тЖТ   571тЖТ
   473тЖТ   572тЖТThe matcher value corresponds to how the session was initiated:
   474тЖТ   573тЖТ
   475тЖТ   574тЖТ| Matcher   | When it fires                          |
   476тЖТ   575тЖТ| :-------- | :------------------------------------- |
   477тЖТ   576тЖТ| `startup` | New session                            |
   478тЖТ   577тЖТ| `resume`  | `--resume`, `--continue`, or `/resume` |
   479тЖТ   578тЖТ| `clear`   | `/clear`                               |
   480тЖТ   579тЖТ| `compact` | Auto or manual compaction              |
   481тЖТ   580тЖТ
   482тЖТ   581тЖТ#### SessionStart input
   483тЖТ   582тЖТ
   484тЖТ   583тЖТIn addition to the [common input fields](#common-input-fields), SessionStart hooks receive `source`, `model`, and optionally `agent_type`. The `source` field indicates how the session started: `"startup"` for new sessions, `"resume"` for resumed sessions, `"clear"` after `/clear`, or `"compact"` after compaction. The `model` field contains the model identifier. If you start Claude Code with `claude --agent <name>`, an `agent_type` field contains the agent name.
   485тЖТ   584тЖТ
   486тЖТ   585тЖТ```json  theme={null}
   487тЖТ   586тЖТ{
   488тЖТ   587тЖТ  "session_id": "abc123",
   489тЖТ   588тЖТ  "transcript_path": "/Users/.../.claude/projects/.../00893aaf-19fa-41d2-8238-13269b9b3ca0.jsonl",
   490тЖТ   589тЖТ  "cwd": "/Users/...",
   491тЖТ   590тЖТ  "permission_mode": "default",
   492тЖТ   591тЖТ  "hook_event_name": "SessionStart",
   493тЖТ   592тЖТ  "source": "startup",
   494тЖТ   593тЖТ  "model": "claude-sonnet-4-6"
   495тЖТ   594тЖТ}
   496тЖТ   595тЖТ```
   497тЖТ   596тЖТ
   498тЖТ   597тЖТ#### SessionStart decision control
   499тЖТ   598тЖТ
   500тЖТ   599тЖТAny text your hook script prints to stdout is added as context for Claude. In addition to the [JSON output fields](#json-output) available to all hooks, you can return these event-specific fields:
   501тЖТ   600тЖТ
   502тЖТ   601тЖТ| Field               | Description                                                               |
   503тЖТ   602тЖТ| :------------------ | :------------------------------------------------------------------------ |
   504тЖТ   603тЖТ| `additionalContext` | String added to Claude's context. Multiple hooks' values are concatenated |
   505тЖТ   604тЖТ
   506тЖТ   605тЖТ```json  theme={null}
   507тЖТ   606тЖТ{
   508тЖТ   607тЖТ  "hookSpecificOutput": {
   509тЖТ   608тЖТ    "hookEventName": "SessionStart",
   510тЖТ   609тЖТ    "additionalContext": "My additional context here"
   511тЖТ   610тЖТ  }
   512тЖТ   611тЖТ}
   513тЖТ   612тЖТ```
   514тЖТ   613тЖТ
   515тЖТ   614тЖТ#### Persist environment variables
   516тЖТ   615тЖТ
   517тЖТ   616тЖТSessionStart hooks have access to the `CLAUDE_ENV_FILE` environment variable, which provides a file path where you can persist environment variables for subsequent Bash commands.
   518тЖТ   617тЖТ
   519тЖТ   618тЖТTo set individual environment variables, write `export` statements to `CLAUDE_ENV_FILE`. Use append (`>>`) to preserve variables set by other hooks:
   520тЖТ   619тЖТ
   521тЖТ   620тЖТ```bash  theme={null}
   522тЖТ   621тЖТ#!/bin/bash
   523тЖТ   622тЖТ
   524тЖТ   623тЖТif [ -n "$CLAUDE_ENV_FILE" ]; then
   525тЖТ   624тЖТ  echo 'export NODE_ENV=production' >> "$CLAUDE_ENV_FILE"
   526тЖТ   625тЖТ  echo 'export DEBUG_LOG=true' >> "$CLAUDE_ENV_FILE"
   527тЖТ   626тЖТ  echo 'export PATH="$PATH:./node_modules/.bin"' >> "$CLAUDE_ENV_FILE"
   528тЖТ   627тЖТfi
   529тЖТ   628тЖТ
   530тЖТ   629тЖТexit 0
   531тЖТ   630тЖТ```
   532тЖТ   631тЖТ
   533тЖТ   632тЖТTo capture all environment changes from setup commands, compare the exported variables before and after:
   534тЖТ   633тЖТ
   535тЖТ   634тЖТ```bash  theme={null}
   536тЖТ   635тЖТ#!/bin/bash
   537тЖТ   636тЖТ
   538тЖТ   637тЖТENV_BEFORE=$(export -p | sort)
   539тЖТ   638тЖТ
   540тЖТ   639тЖТ# Run your setup commands that modify the environment
   541тЖТ   640тЖТsource ~/.nvm/nvm.sh
   542тЖТ   641тЖТnvm use 20
   543тЖТ   642тЖТ
   544тЖТ   643тЖТif [ -n "$CLAUDE_ENV_FILE" ]; then
   545тЖТ   644тЖТ  ENV_AFTER=$(export -p | sort)
   546тЖТ   645тЖТ  comm -13 <(echo "$ENV_BEFORE") <(echo "$ENV_AFTER") >> "$CLAUDE_ENV_FILE"
   547тЖТ   646тЖТfi
   548тЖТ   647тЖТ
   549тЖТ   648тЖТexit 0
   550тЖТ   649тЖТ```
   551тЖТ   650тЖТ
   552тЖТ   651тЖТAny variables written to this file will be available in all subsequent Bash commands that Claude Code executes during the session.
   553тЖТ   652тЖТ
   554тЖТ   653тЖТ<Note>
   555тЖТ   654тЖТ  `CLAUDE_ENV_FILE` is available for SessionStart hooks. Other hook types do not have access to this variable.
   556тЖТ   655тЖТ</Note>
   557тЖТ   656тЖТ
   558тЖТ   657тЖТ### UserPromptSubmit
   559тЖТ   658тЖТ
   560тЖТ   659тЖТRuns when the user submits a prompt, before Claude processes it. This allows you
   561тЖТ   660тЖТto add additional context based on the prompt/conversation, validate prompts, or
   562тЖТ   661тЖТblock certain types of prompts.
   563тЖТ   662тЖТ
   564тЖТ   663тЖТ#### UserPromptSubmit input
   565тЖТ   664тЖТ
   566тЖТ   665тЖТIn addition to the [common input fields](#common-input-fields), UserPromptSubmit hooks receive the `prompt` field containing the text the user submitted.
   567тЖТ   666тЖТ
   568тЖТ   667тЖТ```json  theme={null}
   569тЖТ   668тЖТ{
   570тЖТ   669тЖТ  "session_id": "abc123",
   571тЖТ   670тЖТ  "transcript_path": "/Users/.../.claude/projects/.../00893aaf-19fa-41d2-8238-13269b9b3ca0.jsonl",
   572тЖТ   671тЖТ  "cwd": "/Users/...",
   573тЖТ   672тЖТ  "permission_mode": "default",
   574тЖТ   673тЖТ  "hook_event_name": "UserPromptSubmit",
   575тЖТ   674тЖТ  "prompt": "Write a function to calculate the factorial of a number"
   576тЖТ   675тЖТ}
   577тЖТ   676тЖТ```
   578тЖТ   677тЖТ
   579тЖТ   678тЖТ#### UserPromptSubmit decision control
   580тЖТ   679тЖТ
   581тЖТ   680тЖТ`UserPromptSubmit` hooks can control whether a user prompt is processed and add context. All [JSON output fields](#json-output) are available.
   582тЖТ   681тЖТ
   583тЖТ   682тЖТThere are two ways to add context to the conversation on exit code 0:
   584тЖТ   683тЖТ
   585тЖТ   684тЖТ* **Plain text stdout**: any non-JSON text written to stdout is added as context
   586тЖТ   685тЖТ* **JSON with `additionalContext`**: use the JSON format below for more control. The `additionalContext` field is added as context
   587тЖТ   686тЖТ
   588тЖТ   687тЖТPlain stdout is shown as hook output in the transcript. The `additionalContext` field is added more discretely.
   589тЖТ   688тЖТ
   590тЖТ   689тЖТTo block a prompt, return a JSON object with `decision` set to `"block"`:
   591тЖТ   690тЖТ
   592тЖТ   691тЖТ| Field               | Description                                                                                                        |
   593тЖТ   692тЖТ| :------------------ | :----------------------------------------------------------------------------------------------------------------- |
   594тЖТ   693тЖТ| `decision`          | `"block"` prevents the prompt from being processed and erases it from context. Omit to allow the prompt to proceed |
   595тЖТ   694тЖТ| `reason`            | Shown to the user when `decision` is `"block"`. Not added to context                                               |
   596тЖТ   695тЖТ| `additionalContext` | String added to Claude's context                                                                                   |
   597тЖТ   696тЖТ
   598тЖТ   697тЖТ```json  theme={null}
   599тЖТ   698тЖТ{
   600тЖТ   699тЖТ  "decision": "block",
   601тЖТ   700тЖТ  "reason": "Explanation for decision",
   602тЖТ   701тЖТ  "hookSpecificOutput": {
   603тЖТ   702тЖТ    "hookEventName": "UserPromptSubmit",
   604тЖТ   703тЖТ    "additionalContext": "My additional context here"
   605тЖТ   704тЖТ  }
   606тЖТ   705тЖТ}
   607тЖТ   706тЖТ```
   608тЖТ   707тЖТ
   609тЖТ   708тЖТ<Note>
   610тЖТ   709тЖТ  The JSON format isn't required for simple use cases. To add context, you can print plain text to stdout with exit code 0. Use JSON when you need to
   611тЖТ   710тЖТ  block prompts or want more structured control.
   612тЖТ   711тЖТ</Note>
   613тЖТ   712тЖТ
   614тЖТ   713тЖТ### PreToolUse
   615тЖТ   714тЖТ
   616тЖТ   715тЖТRuns after Claude creates tool parameters and before processing the tool call. Matches on tool name: `Bash`, `Edit`, `Write`, `Read`, `Glob`, `Grep`, `Task`, `WebFetch`, `WebSearch`, and any [MCP tool names](#match-mcp-tools).
   617тЖТ   716тЖТ
   618тЖТ   717тЖТUse [PreToolUse decision control](#pretooluse-decision-control) to allow, deny, or ask for permission to use the tool.

   717тЖТ   816тЖТ#### PreToolUse decision control
   718тЖТ   817тЖТ
   719тЖТ   818тЖТ`PreToolUse` hooks can control whether a tool call proceeds. Unlike other hooks that use a top-level `decision` field, PreToolUse returns its decision inside a `hookSpecificOutput` object. This gives it richer control: three outcomes (allow, deny, or ask) plus the ability to modify tool input before execution.
   720тЖТ   819тЖТ
   721тЖТ   820тЖТ| Field                      | Description                                                                                                                                      |
   722тЖТ   821тЖТ| :------------------------- | :----------------------------------------------------------------------------------------------------------------------------------------------- |
   723тЖТ   822тЖТ| `permissionDecision`       | `"allow"` bypasses the permission system, `"deny"` prevents the tool call, `"ask"` prompts the user to confirm                                   |
   724тЖТ   823тЖТ| `permissionDecisionReason` | For `"allow"` and `"ask"`, shown to the user but not Claude. For `"deny"`, shown to Claude                                                       |
   725тЖТ   824тЖТ| `updatedInput`             | Modifies the tool's input parameters before execution. Combine with `"allow"` to auto-approve, or `"ask"` to show the modified input to the user |
   726тЖТ   825тЖТ| `additionalContext`        | String added to Claude's context before the tool executes                                                                                        |
   727тЖТ   826тЖТ
   728тЖТ   827тЖТ```json  theme={null}
   729тЖТ   828тЖТ{
   730тЖТ   829тЖТ  "hookSpecificOutput": {
   731тЖТ   830тЖТ    "hookEventName": "PreToolUse",
   732тЖТ   831тЖТ    "permissionDecision": "allow",
   733тЖТ   832тЖТ    "permissionDecisionReason": "My reason here",
   734тЖТ   833тЖТ    "updatedInput": {
   735тЖТ   834тЖТ      "field_to_modify": "new value"
   736тЖТ   835тЖТ    },
   737тЖТ   836тЖТ    "additionalContext": "Current environment: production. Proceed with caution."
   738тЖТ   837тЖТ  }
   739тЖТ   838тЖТ}
   740тЖТ   839тЖТ```
   741тЖТ   840тЖТ
   742тЖТ   841тЖТ<Note>
   743тЖТ   842тЖТ  PreToolUse previously used top-level `decision` and `reason` fields, but these are deprecated for this event. Use `hookSpecificOutput.permissionDecision` and `hookSpecificOutput.permissionDecisionReason` instead. The deprecated values `"approve"` and `"block"` map to `"allow"` and `"deny"` respectively. Other events like PostToolUse and Stop continue to use top-level `decision` and `reason` as their current format.

   746тЖТ   845тЖТ### PermissionRequest
   747тЖТ   846тЖТ
   748тЖТ   847тЖТRuns when the user is shown a permission dialog.
   749тЖТ   848тЖТUse [PermissionRequest decision control](#permissionrequest-decision-control) to allow or deny on behalf of the user.
   750тЖТ   849тЖТ
   751тЖТ   850тЖТMatches on tool name, same values as PreToolUse.
   752тЖТ   851тЖТ
   753тЖТ   852тЖТ#### PermissionRequest input
   754тЖТ   853тЖТ
   755тЖТ   854тЖТPermissionRequest hooks receive `tool_name` and `tool_input` fields like PreToolUse hooks, but without `tool_use_id`. An optional `permission_suggestions` array contains the "always allow" options the user would normally see in the permission dialog. The difference is when the hook fires: PermissionRequest hooks run when a permission dialog is about to be shown to the user, while PreToolUse hooks run before tool execution regardless of permission status.
   756тЖТ   855тЖТ
   757тЖТ   856тЖТ```json  theme={null}
   758тЖТ   857тЖТ{
   759тЖТ   858тЖТ  "session_id": "abc123",
   760тЖТ   859тЖТ  "transcript_path": "/Users/.../.claude/projects/.../00893aaf-19fa-41d2-8238-13269b9b3ca0.jsonl",
   761тЖТ   860тЖТ  "cwd": "/Users/...",
   762тЖТ   861тЖТ  "permission_mode": "default",
   763тЖТ   862тЖТ  "hook_event_name": "PermissionRequest",
   764тЖТ   863тЖТ  "tool_name": "Bash",
   765тЖТ   864тЖТ  "tool_input": {
   766тЖТ   865тЖТ    "command": "rm -rf node_modules",
   767тЖТ   866тЖТ    "description": "Remove node_modules directory"
   768тЖТ   867тЖТ  },
   769тЖТ   868тЖТ  "permission_suggestions": [
   770тЖТ   869тЖТ    { "type": "toolAlwaysAllow", "tool": "Bash" }
   771тЖТ   870тЖТ  ]
   772тЖТ   871тЖТ}
   773тЖТ   872тЖТ```

   775тЖТ   874тЖТ#### PermissionRequest decision control
   776тЖТ   875тЖТ
   777тЖТ   876тЖТ`PermissionRequest` hooks can allow or deny permission requests. In addition to the [JSON output fields](#json-output) available to all hooks, your hook script can return a `decision` object with these event-specific fields:
   778тЖТ   877тЖТ
   779тЖТ   878тЖТ| Field                | Description                                                                                                    |
   780тЖТ   879тЖТ| :------------------- | :------------------------------------------------------------------------------------------------------------- |
   781тЖТ   880тЖТ| `behavior`           | `"allow"` grants the permission, `"deny"` denies it                                                            |
   782тЖТ   881тЖТ| `updatedInput`       | For `"allow"` only: modifies the tool's input parameters before execution                                      |
   783тЖТ   882тЖТ| `updatedPermissions` | For `"allow"` only: applies permission rule updates, equivalent to the user selecting an "always allow" option |
   784тЖТ   883тЖТ| `message`            | For `"deny"` only: tells Claude why the permission was denied                                                  |
   785тЖТ   884тЖТ| `interrupt`          | For `"deny"` only: if `true`, stops Claude                                                                     |
   786тЖТ   885тЖТ
   787тЖТ   886тЖТ```json  theme={null}
   788тЖТ   887тЖТ{
   789тЖТ   888тЖТ  "hookSpecificOutput": {
   790тЖТ   889тЖТ    "hookEventName": "PermissionRequest",
   791тЖТ   890тЖТ    "decision": {
   792тЖТ   891тЖТ      "behavior": "allow",
   793тЖТ   892тЖТ      "updatedInput": {
   794тЖТ   893тЖТ        "command": "npm run lint"
   795тЖТ   894тЖТ      }
   796тЖТ   895тЖТ    }
   797тЖТ   896тЖТ  }
   798тЖТ   897тЖТ}
   799тЖТ   898тЖТ```
