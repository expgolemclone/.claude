     5‚Üí# Hooks reference
     6‚Üí
     7‚Üí> Reference for Claude Code hook events, configuration schema, JSON input/output formats, exit codes, async hooks, prompt hooks, and MCP tool hooks.
     8‚Üí
     9‚Üí<Tip>
    10‚Üí  For a quickstart guide with examples, see [Automate workflows with hooks](/en/hooks-guide).
    11‚Üí</Tip>
    12‚Üí
    13‚ÜíHooks are user-defined shell commands or LLM prompts that execute automatically at specific points in Claude Code's lifecycle. Use this reference to look up event schemas, configuration options, JSON input/output formats, and advanced features like async hooks and MCP tool hooks. If you're setting up hooks for the first time, start with the [guide](/en/hooks-guide) instead.
    14‚Üí
    15‚Üí## Hook lifecycle
    16‚Üí
    17‚ÜíHooks fire at specific points during a Claude Code session. When an event fires and a matcher matches, Claude Code passes JSON context about the event to your hook handler. For command hooks, this arrives on stdin. Your handler can then inspect the input, take action, and optionally return a decision. Some events fire once per session, while others fire repeatedly inside the agentic loop:

    25‚ÜíThe table below summarizes when each event fires. The [Hook events](#hook-events) section documents the full input schema and decision control options for each one.
    26‚Üí
    27‚Üí| Event                | When it fires                                                                                               |
    28‚Üí| :------------------- | :---------------------------------------------------------------------------------------------------------- |
    29‚Üí| `SessionStart`       | When a session begins or resumes                                                                            |
    30‚Üí| `UserPromptSubmit`   | When you submit a prompt, before Claude processes it                                                        |
    31‚Üí| `PreToolUse`         | Before a tool call executes. Can block it                                                                   |
    32‚Üí| `PermissionRequest`  | When a permission dialog appears                                                                            |
    33‚Üí| `PostToolUse`        | After a tool call succeeds                                                                                  |
    34‚Üí| `PostToolUseFailure` | After a tool call fails                                                                                     |
    35‚Üí| `Notification`       | When Claude Code sends a notification                                                                       |
    36‚Üí| `SubagentStart`      | When a subagent is spawned                                                                                  |
    37‚Üí| `SubagentStop`       | When a subagent finishes                                                                                    |
    38‚Üí| `Stop`               | When Claude finishes responding                                                                             |
    39‚Üí| `TeammateIdle`       | When an [agent team](/en/agent-teams) teammate is about to go idle                                          |
    40‚Üí| `TaskCompleted`      | When a task is being marked as completed                                                                    |
    41‚Üí| `ConfigChange`       | When a configuration file changes during a session                                                          |
    42‚Üí| `WorktreeCreate`     | When a worktree is being created via `--worktree` or `isolation: "worktree"`. Replaces default git behavior |
    43‚Üí| `WorktreeRemove`     | When a worktree is being removed, either at session exit or when a subagent finishes                        |
    44‚Üí| `PreCompact`         | Before context compaction                                                                                   |
    45‚Üí| `SessionEnd`         | When a session terminates                                                                                   |

    47‚Üí### How a hook resolves
    48‚Üí
    49‚ÜíTo see how these pieces fit together, consider this `PreToolUse` hook that blocks destructive shell commands. The hook runs `block-rm.sh` before every Bash tool call:
    50‚Üí
    51‚Üí```json  theme={null}
    52‚Üí{
    53‚Üí  "hooks": {
    54‚Üí    "PreToolUse": [
    55‚Üí      {
    56‚Üí        "matcher": "Bash",
    57‚Üí        "hooks": [
    58‚Üí          {
    59‚Üí            "type": "command",
    60‚Üí            "command": ".claude/hooks/block-rm.sh"
    61‚Üí          }
    62‚Üí        ]
    63‚Üí      }
    64‚Üí    ]
    65‚Üí  }
    66‚Üí}
    67‚Üí```
    68‚Üí
    69‚ÜíThe script reads the JSON input from stdin, extracts the command, and returns a `permissionDecision` of `"deny"` if it contains `rm -rf`:
    70‚Üí
    71‚Üí```bash  theme={null}
    72‚Üí#!/bin/bash
    73‚Üí# .claude/hooks/block-rm.sh
    74‚ÜíCOMMAND=$(jq -r '.tool_input.command')
    75‚Üí
    76‚Üíif echo "$COMMAND" | grep -q 'rm -rf'; then
    77‚Üí  jq -n '{
    78‚Üí    hookSpecificOutput: {
    79‚Üí      hookEventName: "PreToolUse",
    80‚Üí      permissionDecision: "deny",
    81‚Üí      permissionDecisionReason: "Destructive command blocked by hook"
    82‚Üí    }
    83‚Üí  }'
    84‚Üíelse
    85‚Üí  exit 0  # allow the command
    86‚Üífi
    87‚Üí```
    88‚Üí

    95‚Üí<Steps>
    96‚Üí  <Step title="Event fires">
    97‚Üí    The `PreToolUse` event fires. Claude Code sends the tool input as JSON on stdin to the hook:
    98‚Üí
    99‚Üí    ```json  theme={null}
   100‚Üí    { "tool_name": "Bash", "tool_input": { "command": "rm -rf /tmp/build" }, ... }
   101‚Üí    ```
   102‚Üí  </Step>
   103‚Üí
   104‚Üí  <Step title="Matcher checks">
   105‚Üí    The matcher `"Bash"` matches the tool name, so `block-rm.sh` runs. If you omit the matcher or use `"*"`, the hook runs on every occurrence of the event. Hooks only skip when a matcher is defined and doesn't match.
   106‚Üí  </Step>
   107‚Üí
   108‚Üí  <Step title="Hook handler runs">
   109‚Üí    The script extracts `"rm -rf /tmp/build"` from the input and finds `rm -rf`, so it prints a decision to stdout:
   110‚Üí
   111‚Üí    ```json  theme={null}
   112‚Üí    {
   113‚Üí      "hookSpecificOutput": {
   114‚Üí        "hookEventName": "PreToolUse",
   115‚Üí        "permissionDecision": "deny",
   116‚Üí        "permissionDecisionReason": "Destructive command blocked by hook"
   117‚Üí      }
   118‚Üí    }
   119‚Üí    ```
   120‚Üí
   121‚Üí    If the command had been safe (like `npm test`), the script would hit `exit 0` instead, which tells Claude Code to allow the tool call with no further action.
   122‚Üí  </Step>
   123‚Üí
   124‚Üí  <Step title="Claude Code acts on the result">
   125‚Üí    Claude Code reads the JSON decision, blocks the tool call, and shows Claude the reason.
   126‚Üí  </Step>
   127‚Üí</Steps>
   128‚Üí
   129‚ÜíThe [Configuration](#configuration) section below documents the full schema, and each [hook event](#hook-events) section documents what input your command receives and what output it can return.
   130‚Üí
   131‚Üí## Configuration
   132‚Üí
   133‚ÜíHooks are defined in JSON settings files. The configuration has three levels of nesting:
   134‚Üí
   135‚Üí1. Choose a [hook event](#hook-events) to respond to, like `PreToolUse` or `Stop`
   136‚Üí2. Add a [matcher group](#matcher-patterns) to filter when it fires, like "only for the Bash tool"
   137‚Üí3. Define one or more [hook handlers](#hook-handler-fields) to run when matched
   138‚Üí
   139‚ÜíSee [How a hook resolves](#how-a-hook-resolves) above for a complete walkthrough with an annotated example.
   140‚Üí
   141‚Üí<Note>
   142‚Üí  This page uses specific terms for each level: **hook event** for the lifecycle point, **matcher group** for the filter, and **hook handler** for the shell command, prompt, or agent that runs. "Hook" on its own refers to the general feature.
   143‚Üí</Note>
   144‚Üí
   145‚Üí### Hook locations
   146‚Üí
   147‚ÜíWhere you define a hook determines its scope:
   148‚Üí
   149‚Üí| Location                                                   | Scope                         | Shareable                          |
   150‚Üí| :--------------------------------------------------------- | :---------------------------- | :--------------------------------- |
   151‚Üí| `~/.claude/settings.json`                                  | All your projects             | No, local to your machine          |
   152‚Üí| `.claude/settings.json`                                    | Single project                | Yes, can be committed to the repo  |
   153‚Üí| `.claude/settings.local.json`                              | Single project                | No, gitignored                     |
   154‚Üí| Managed policy settings                                    | Organization-wide             | Yes, admin-controlled              |
   155‚Üí| [Plugin](/en/plugins) `hooks/hooks.json`                   | When plugin is enabled        | Yes, bundled with the plugin       |
   156‚Üí| [Skill](/en/skills) or [agent](/en/sub-agents) frontmatter | While the component is active | Yes, defined in the component file |
   157‚Üí
   158‚ÜíFor details on settings file resolution, see [settings](/en/settings). Enterprise administrators can use `allowManagedHooksOnly` to block user, project, and plugin hooks. See [Hook configuration](/en/settings#hook-configuration).
   159‚Üí
   160‚Üí### Matcher patterns
   161‚Üí
   162‚ÜíThe `matcher` field is a regex string that filters when hooks fire. Use `"*"`, `""`, or omit `matcher` entirely to match all occurrences. Each event type matches on a different field:
   163‚Üí
   164‚Üí| Event                                                                                           | What the matcher filters  | Example matcher values                                                             |
   165‚Üí| :---------------------------------------------------------------------------------------------- | :------------------------ | :--------------------------------------------------------------------------------- |
   166‚Üí| `PreToolUse`, `PostToolUse`, `PostToolUseFailure`, `PermissionRequest`                          | tool name                 | `Bash`, `Edit\|Write`, `mcp__.*`                                                   |
   167‚Üí| `SessionStart`                                                                                  | how the session started   | `startup`, `resume`, `clear`, `compact`                                            |
   168‚Üí| `SessionEnd`                                                                                    | why the session ended     | `clear`, `logout`, `prompt_input_exit`, `bypass_permissions_disabled`, `other`     |
   169‚Üí| `Notification`                                                                                  | notification type         | `permission_prompt`, `idle_prompt`, `auth_success`, `elicitation_dialog`           |
   170‚Üí| `SubagentStart`                                                                                 | agent type                | `Bash`, `Explore`, `Plan`, or custom agent names                                   |
   171‚Üí| `PreCompact`                                                                                    | what triggered compaction | `manual`, `auto`                                                                   |
   172‚Üí| `SubagentStop`                                                                                  | agent type                | same values as `SubagentStart`                                                     |
   173‚Üí| `ConfigChange`                                                                                  | configuration source      | `user_settings`, `project_settings`, `local_settings`, `policy_settings`, `skills` |
   174‚Üí| `UserPromptSubmit`, `Stop`, `TeammateIdle`, `TaskCompleted`, `WorktreeCreate`, `WorktreeRemove` | no matcher support        | always fires on every occurrence                                                   |
   175‚Üí
   176‚ÜíThe matcher is a regex, so `Edit|Write` matches either tool and `Notebook.*` matches any tool starting with Notebook. The matcher runs against a field from the [JSON input](#hook-input-and-output) that Claude Code sends to your hook on stdin. For tool events, that field is `tool_name`. Each [hook event](#hook-events) section lists the full set of matcher values and the input schema for that event.
   177‚Üí
   178‚ÜíThis example runs a linting script only when Claude writes or edits a file:
   179‚Üí
   180‚Üí```json  theme={null}
   181‚Üí{
   182‚Üí  "hooks": {
   183‚Üí    "PostToolUse": [
   184‚Üí      {
   185‚Üí        "matcher": "Edit|Write",
   186‚Üí        "hooks": [
   187‚Üí          {
   188‚Üí            "type": "command",
   189‚Üí            "command": "/path/to/lint-check.sh"
   190‚Üí          }
   191‚Üí        ]
   192‚Üí      }
   193‚Üí    ]
   194‚Üí  }
   195‚Üí}
   196‚Üí```
   197‚Üí
   198‚Üí`UserPromptSubmit`, `Stop`, `TeammateIdle`, `TaskCompleted`, `WorktreeCreate`, and `WorktreeRemove` don't support matchers and always fire on every occurrence. If you add a `matcher` field to these events, it is silently ignored.
   199‚Üí
   200‚Üí#### Match MCP tools
   201‚Üí
   202‚Üí[MCP](/en/mcp) server tools appear as regular tools in tool events (`PreToolUse`, `PostToolUse`, `PostToolUseFailure`, `PermissionRequest`), so you can match them the same way you match any other tool name.
   203‚Üí
   204‚ÜíMCP tools follow the naming pattern `mcp__<server>__<tool>`, for example:
   205‚Üí
   206‚Üí* `mcp__memory__create_entities`: Memory server's create entities tool
   207‚Üí* `mcp__filesystem__read_file`: Filesystem server's read file tool
   208‚Üí* `mcp__github__search_repositories`: GitHub server's search tool
   209‚Üí
   210‚ÜíUse regex patterns to target specific MCP tools or groups of tools:
   211‚Üí
   212‚Üí* `mcp__memory__.*` matches all tools from the `memory` server
   213‚Üí* `mcp__.*__write.*` matches any tool containing "write" from any server
   214‚Üí
   215‚ÜíThis example logs all memory server operations and validates write operations from any MCP server:
   216‚Üí
   217‚Üí```json  theme={null}
   218‚Üí{
   219‚Üí  "hooks": {
   220‚Üí    "PreToolUse": [
   221‚Üí      {
   222‚Üí        "matcher": "mcp__memory__.*",
   223‚Üí        "hooks": [
   224‚Üí          {
   225‚Üí            "type": "command",
   226‚Üí            "command": "echo 'Memory operation initiated' >> ~/mcp-operations.log"
   227‚Üí          }
   228‚Üí        ]
   229‚Üí      },
   230‚Üí      {
   231‚Üí        "matcher": "mcp__.*__write.*",
   232‚Üí        "hooks": [
   233‚Üí          {
   234‚Üí            "type": "command",
   235‚Üí            "command": "/home/user/scripts/validate-mcp-write.py"
   236‚Üí          }
   237‚Üí        ]
   238‚Üí      }
   239‚Üí    ]
   240‚Üí  }
   241‚Üí}
   242‚Üí```
   243‚Üí
   244‚Üí### Hook handler fields
   245‚Üí
   246‚ÜíEach object in the inner `hooks` array is a hook handler: the shell command, LLM prompt, or agent that runs when the matcher matches. There are three types:
   247‚Üí
   248‚Üí* **[Command hooks](#command-hook-fields)** (`type: "command"`): run a shell command. Your script receives the event's [JSON input](#hook-input-and-output) on stdin and communicates results back through exit codes and stdout.
   249‚Üí* **[Prompt hooks](#prompt-and-agent-hook-fields)** (`type: "prompt"`): send a prompt to a Claude model for single-turn evaluation. The model returns a yes/no decision as JSON. See [Prompt-based hooks](#prompt-based-hooks).
   250‚Üí* **[Agent hooks](#prompt-and-agent-hook-fields)** (`type: "agent"`): spawn a subagent that can use tools like Read, Grep, and Glob to verify conditions before returning a decision. See [Agent-based hooks](#agent-based-hooks).
   251‚Üí
   252‚Üí#### Common fields
   253‚Üí
   254‚ÜíThese fields apply to all hook types:
   255‚Üí
   256‚Üí| Field           | Required | Description                                                                                                                                   |
   257‚Üí| :-------------- | :------- | :-------------------------------------------------------------------------------------------------------------------------------------------- |
   258‚Üí| `type`          | yes      | `"command"`, `"prompt"`, or `"agent"`                                                                                                         |
   259‚Üí| `timeout`       | no       | Seconds before canceling. Defaults: 600 for command, 30 for prompt, 60 for agent                                                              |
   260‚Üí| `statusMessage` | no       | Custom spinner message displayed while the hook runs                                                                                          |
   261‚Üí| `once`          | no       | If `true`, runs only once per session then is removed. Skills only, not agents. See [Hooks in skills and agents](#hooks-in-skills-and-agents) |
   262‚Üí
   263‚Üí#### Command hook fields
   264‚Üí
   265‚ÜíIn addition to the [common fields](#common-fields), command hooks accept these fields:
   266‚Üí
   267‚Üí| Field     | Required | Description                                                                                                         |
   268‚Üí| :-------- | :------- | :------------------------------------------------------------------------------------------------------------------ |
   269‚Üí| `command` | yes      | Shell command to execute                                                                                            |
   270‚Üí| `async`   | no       | If `true`, runs in the background without blocking. See [Run hooks in the background](#run-hooks-in-the-background) |
   271‚Üí
   272‚Üí#### Prompt and agent hook fields
   273‚Üí
   274‚ÜíIn addition to the [common fields](#common-fields), prompt and agent hooks accept these fields:
   275‚Üí
   276‚Üí| Field    | Required | Description                                                                                 |
   277‚Üí| :------- | :------- | :------------------------------------------------------------------------------------------ |
   278‚Üí| `prompt` | yes      | Prompt text to send to the model. Use `$ARGUMENTS` as a placeholder for the hook input JSON |
   279‚Üí| `model`  | no       | Model to use for evaluation. Defaults to a fast model                                       |
   280‚Üí
   281‚ÜíAll matching hooks run in parallel, and identical handlers are deduplicated automatically. Handlers run in the current directory with Claude Code's environment. The `$CLAUDE_CODE_REMOTE` environment variable is set to `"true"` in remote web environments and not set in the local CLI.
   282‚Üí
   283‚Üí### Reference scripts by path
   284‚Üí
   285‚ÜíUse environment variables to reference hook scripts relative to the project or plugin root, regardless of the working directory when the hook runs:
   286‚Üí
   287‚Üí* `$CLAUDE_PROJECT_DIR`: the project root. Wrap in quotes to handle paths with spaces.
   288‚Üí* `${CLAUDE_PLUGIN_ROOT}`: the plugin's root directory, for scripts bundled with a [plugin](/en/plugins).
   289‚Üí
   290‚Üí<Tabs>
   291‚Üí  <Tab title="Project scripts">
   292‚Üí    This example uses `$CLAUDE_PROJECT_DIR` to run a style checker from the project's `.claude/hooks/` directory after any `Write` or `Edit` tool call:
   293‚Üí
   294‚Üí    ```json  theme={null}
   295‚Üí    {
   296‚Üí      "hooks": {
   297‚Üí        "PostToolUse": [
   298‚Üí          {
   299‚Üí            "matcher": "Write|Edit",
   300‚Üí            "hooks": [
   301‚Üí              {
   302‚Üí                "type": "command",
   303‚Üí                "command": "\"$CLAUDE_PROJECT_DIR\"/.claude/hooks/check-style.sh"
   304‚Üí              }
   305‚Üí            ]
   306‚Üí          }
   307‚Üí        ]
   308‚Üí      }
   309‚Üí    }
   310‚Üí    ```
   311‚Üí  </Tab>
   312‚Üí
   313‚Üí  <Tab title="Plugin scripts">
   314‚Üí    Define plugin hooks in `hooks/hooks.json` with an optional top-level `description` field. When a plugin is enabled, its hooks merge with your user and project hooks.
   315‚Üí
   316‚Üí    This example runs a formatting script bundled with the plugin:
   317‚Üí
   318‚Üí    ```json  theme={null}
   319‚Üí    {
   320‚Üí      "description": "Automatic code formatting",
   321‚Üí      "hooks": {
   322‚Üí        "PostToolUse": [
   323‚Üí          {
   324‚Üí            "matcher": "Write|Edit",
   325‚Üí            "hooks": [
   326‚Üí              {
   327‚Üí                "type": "command",
   328‚Üí                "command": "${CLAUDE_PLUGIN_ROOT}/scripts/format.sh",
   329‚Üí                "timeout": 30
   330‚Üí              }
   331‚Üí            ]
   332‚Üí          }
   333‚Üí        ]
   334‚Üí      }
   335‚Üí    }
   336‚Üí    ```
   337‚Üí
   338‚Üí    See the [plugin components reference](/en/plugins-reference#hooks) for details on creating plugin hooks.
   339‚Üí  </Tab>
   340‚Üí</Tabs>
   341‚Üí
   342‚Üí### Hooks in skills and agents
   343‚Üí
   344‚ÜíIn addition to settings files and plugins, hooks can be defined directly in [skills](/en/skills) and [subagents](/en/sub-agents) using frontmatter. These hooks are scoped to the component's lifecycle and only run when that component is active.
   345‚Üí
   346‚ÜíAll hook events are supported. For subagents, `Stop` hooks are automatically converted to `SubagentStop` since that is the event that fires when a subagent completes.
   347‚Üí
   348‚ÜíHooks use the same configuration format as settings-based hooks but are scoped to the component's lifetime and cleaned up when it finishes.
   349‚Üí
   350‚ÜíThis skill defines a `PreToolUse` hook that runs a security validation script before each `Bash` command:
   351‚Üí
   352‚Üí```yaml  theme={null}
   353‚Üí---
   354‚Üíname: secure-operations
   355‚Üídescription: Perform operations with security checks
   356‚Üíhooks:
   357‚Üí  PreToolUse:
   358‚Üí    - matcher: "Bash"
   359‚Üí      hooks:
   360‚Üí        - type: command
   361‚Üí          command: "./scripts/security-check.sh"
   362‚Üí---
   363‚Üí```
   364‚Üí
   365‚ÜíAgents use the same format in their YAML frontmatter.
   366‚Üí
   367‚Üí### The `/hooks` menu
   368‚Üí
   369‚ÜíType `/hooks` in Claude Code to open the interactive hooks manager, where you can view, add, and delete hooks without editing settings files directly. For a step-by-step walkthrough, see [Set up your first hook](/en/hooks-guide#set-up-your-first-hook) in the guide.
   370‚Üí
   371‚ÜíEach hook in the menu is labeled with a bracket prefix indicating its source:
   372‚Üí
   373‚Üí* `[User]`: from `~/.claude/settings.json`
   374‚Üí* `[Project]`: from `.claude/settings.json`
   375‚Üí* `[Local]`: from `.claude/settings.local.json`
   376‚Üí* `[Plugin]`: from a plugin's `hooks/hooks.json`, read-only
   377‚Üí
   378‚Üí### Disable or remove hooks
   379‚Üí
   380‚ÜíTo remove a hook, delete its entry from the settings JSON file, or use the `/hooks` menu and select the hook to delete it.
   381‚Üí
   382‚ÜíTo temporarily disable all hooks without removing them, set `"disableAllHooks": true` in your settings file or use the toggle in the `/hooks` menu. There is no way to disable an individual hook while keeping it in the configuration.
   383‚Üí
   384‚ÜíThe `disableAllHooks` setting respects the managed settings hierarchy. If an administrator has configured hooks through managed policy settings, `disableAllHooks` set in user, project, or local settings cannot disable those managed hooks. Only `disableAllHooks` set at the managed settings level can disable managed hooks.
   385‚Üí
   386‚ÜíDirect edits to hooks in settings files don't take effect immediately. Claude Code captures a snapshot of hooks at startup and uses it throughout the session. This prevents malicious or accidental hook modifications from taking effect mid-session without your review. If hooks are modified externally, Claude Code warns you and requires review in the `/hooks` menu before changes apply.
   387‚Üí
   388‚Üí## Hook input and output
   389‚Üí
   390‚ÜíHooks receive JSON data via stdin and communicate results through exit codes, stdout, and stderr. This section covers fields and behavior common to all events. Each event's section under [Hook events](#hook-events) includes its specific input schema and decision control options.
   391‚Üí
   392‚Üí### Common input fields
   393‚Üí
   394‚ÜíAll hook events receive these fields via stdin as JSON, in addition to event-specific fields documented in each [hook event](#hook-events) section:
   395‚Üí
   396‚Üí| Field             | Description                                                                                                                                |
   397‚Üí| :---------------- | :----------------------------------------------------------------------------------------------------------------------------------------- |
   398‚Üí| `session_id`      | Current session identifier                                                                                                                 |
   399‚Üí| `transcript_path` | Path to conversation JSON                                                                                                                  |
   400‚Üí| `cwd`             | Current working directory when the hook is invoked                                                                                         |
   401‚Üí| `permission_mode` | Current [permission mode](/en/permissions#permission-modes): `"default"`, `"plan"`, `"acceptEdits"`, `"dontAsk"`, or `"bypassPermissions"` |
   402‚Üí| `hook_event_name` | Name of the event that fired                                                                                                               |
   403‚Üí
   404‚ÜíFor example, a `PreToolUse` hook for a Bash command receives this on stdin:
   405‚Üí
   406‚Üí```json  theme={null}
   407‚Üí{
   408‚Üí  "session_id": "abc123",
   409‚Üí  "transcript_path": "/home/user/.claude/projects/.../transcript.jsonl",
   410‚Üí  "cwd": "/home/user/my-project",
   411‚Üí  "permission_mode": "default",
   412‚Üí  "hook_event_name": "PreToolUse",
   413‚Üí  "tool_name": "Bash",
   414‚Üí  "tool_input": {
   415‚Üí    "command": "npm test"
   416‚Üí  }
   417‚Üí}
   418‚Üí```
   419‚Üí
   420‚ÜíThe `tool_name` and `tool_input` fields are event-specific. Each [hook event](#hook-events) section documents the additional fields for that event.
   421‚Üí
   422‚Üí### Exit code output
   423‚Üí
   424‚ÜíThe exit code from your hook command tells Claude Code whether the action should proceed, be blocked, or be ignored.
   425‚Üí
   426‚Üí**Exit 0** means success. Claude Code parses stdout for [JSON output fields](#json-output). JSON output is only processed on exit 0. For most events, stdout is only shown in verbose mode (`Ctrl+O`). The exceptions are `UserPromptSubmit` and `SessionStart`, where stdout is added as context that Claude can see and act on.
   427‚Üí
   428‚Üí**Exit 2** means a blocking error. Claude Code ignores stdout and any JSON in it. Instead, stderr text is fed back to Claude as an error message. The effect depends on the event: `PreToolUse` blocks the tool call, `UserPromptSubmit` rejects the prompt, and so on. See [exit code 2 behavior](#exit-code-2-behavior-per-event) for the full list.
   429‚Üí
   430‚Üí**Any other exit code** is a non-blocking error. stderr is shown in verbose mode (`Ctrl+O`) and execution continues.
   431‚Üí
   432‚ÜíFor example, a hook command script that blocks dangerous Bash commands:
   433‚Üí
   434‚Üí```bash  theme={null}
   435‚Üí#!/bin/bash
   436‚Üí# Reads JSON input from stdin, checks the command
   437‚Üícommand=$(jq -r '.tool_input.command' < /dev/stdin)
   438‚Üí
   439‚Üíif [[ "$command" == rm* ]]; then
   440‚Üí  echo "Blocked: rm commands are not allowed" >&2
   441‚Üí  exit 2  # Blocking error: tool call is prevented
   442‚Üífi
   443‚Üí
   444‚Üíexit 0  # Success: tool call proceeds
   445‚Üí```
   446‚Üí
   447‚Üí#### Exit code 2 behavior per event
   448‚Üí
   449‚ÜíExit code 2 is the way a hook signals "stop, don't do this." The effect depends on the event, because some events represent actions that can be blocked (like a tool call that hasn't happened yet) and others represent things that already happened or can't be prevented.
   450‚Üí
   451‚Üí| Hook event           | Can block? | What happens on exit 2                                                        |
   452‚Üí| :------------------- | :--------- | :---------------------------------------------------------------------------- |
   453‚Üí| `PreToolUse`         | Yes        | Blocks the tool call                                                          |
   454‚Üí| `PermissionRequest`  | Yes        | Denies the permission                                                         |
   455‚Üí| `UserPromptSubmit`   | Yes        | Blocks prompt processing and erases the prompt                                |
   456‚Üí| `Stop`               | Yes        | Prevents Claude from stopping, continues the conversation                     |
   457‚Üí| `SubagentStop`       | Yes        | Prevents the subagent from stopping                                           |
   458‚Üí| `TeammateIdle`       | Yes        | Prevents the teammate from going idle (teammate continues working)            |
   459‚Üí| `TaskCompleted`      | Yes        | Prevents the task from being marked as completed                              |
   460‚Üí| `ConfigChange`       | Yes        | Blocks the configuration change from taking effect (except `policy_settings`) |
   461‚Üí| `PostToolUse`        | No         | Shows stderr to Claude (tool already ran)                                     |
   462‚Üí| `PostToolUseFailure` | No         | Shows stderr to Claude (tool already failed)                                  |
   463‚Üí| `Notification`       | No         | Shows stderr to user only                                                     |
   464‚Üí| `SubagentStart`      | No         | Shows stderr to user only                                                     |
   465‚Üí| `SessionStart`       | No         | Shows stderr to user only                                                     |
   466‚Üí| `SessionEnd`         | No         | Shows stderr to user only                                                     |
   467‚Üí| `PreCompact`         | No         | Shows stderr to user only                                                     |
   468‚Üí| `WorktreeCreate`     | Yes        | Any non-zero exit code causes worktree creation to fail                       |
   469‚Üí| `WorktreeRemove`     | No         | Failures are logged in debug mode only                                        |
   470‚Üí
   471‚Üí### JSON output
   472‚Üí
   473‚ÜíExit codes let you allow or block, but JSON output gives you finer-grained control. Instead of exiting with code 2 to block, exit 0 and print a JSON object to stdout. Claude Code reads specific fields from that JSON to control behavior, including [decision control](#decision-control) for blocking, allowing, or escalating to the user.
   474‚Üí
   475‚Üí<Note>
   476‚Üí  You must choose one approach per hook, not both: either use exit codes alone for signaling, or exit 0 and print JSON for structured control. Claude Code only processes JSON on exit 0. If you exit 2, any JSON is ignored.
   477‚Üí</Note>
   478‚Üí
   479‚ÜíYour hook's stdout must contain only the JSON object. If your shell profile prints text on startup, it can interfere with JSON parsing. See [JSON validation failed](/en/hooks-guide#json-validation-failed) in the troubleshooting guide.
   480‚Üí
   481‚ÜíThe JSON object supports three kinds of fields:
   482‚Üí
   483‚Üí* **Universal fields** like `continue` work across all events. These are listed in the table below.
   484‚Üí* **Top-level `decision` and `reason`** are used by some events to block or provide feedback.
   485‚Üí* **`hookSpecificOutput`** is a nested object for events that need richer control. It requires a `hookEventName` field set to the event name.
   486‚Üí
   487‚Üí| Field            | Default | Description                                                                                                                |
   488‚Üí| :--------------- | :------ | :------------------------------------------------------------------------------------------------------------------------- |
   489‚Üí| `continue`       | `true`  | If `false`, Claude stops processing entirely after the hook runs. Takes precedence over any event-specific decision fields |
   490‚Üí| `stopReason`     | none    | Message shown to the user when `continue` is `false`. Not shown to Claude                                                  |
   491‚Üí| `suppressOutput` | `false` | If `true`, hides stdout from verbose mode output                                                                           |
   492‚Üí| `systemMessage`  | none    | Warning message shown to the user                                                                                          |
   493‚Üí
   494‚ÜíTo stop Claude entirely regardless of event type:
   495‚Üí
   496‚Üí```json  theme={null}
   497‚Üí{ "continue": false, "stopReason": "Build failed, fix errors before continuing" }
   498‚Üí```
   499‚Üí
   500‚Üí#### Decision control
   501‚Üí
   502‚ÜíNot every event supports blocking or controlling behavior through JSON. The events that do each use a different set of fields to express that decision. Use this table as a quick reference before writing a hook:
   503‚Üí
   504‚Üí| Events                                                                              | Decision pattern     | Key fields                                                                  |
   505‚Üí| :---------------------------------------------------------------------------------- | :------------------- | :-------------------------------------------------------------------------- |
   506‚Üí| UserPromptSubmit, PostToolUse, PostToolUseFailure, Stop, SubagentStop, ConfigChange | Top-level `decision` | `decision: "block"`, `reason`                                               |
   507‚Üí| TeammateIdle, TaskCompleted                                                         | Exit code only       | Exit code 2 blocks the action, stderr is fed back as feedback               |
   508‚Üí| PreToolUse                                                                          | `hookSpecificOutput` | `permissionDecision` (allow/deny/ask), `permissionDecisionReason`           |
   509‚Üí| PermissionRequest                                                                   | `hookSpecificOutput` | `decision.behavior` (allow/deny)                                            |
   510‚Üí| WorktreeCreate                                                                      | stdout path          | Hook prints absolute path to created worktree. Non-zero exit fails creation |
   511‚Üí| WorktreeRemove, Notification, SessionEnd, PreCompact                                | None                 | No decision control. Used for side effects like logging or cleanup          |
   512‚Üí
   513‚ÜíHere are examples of each pattern in action:
   514‚Üí
   515‚Üí<Tabs>
   516‚Üí  <Tab title="Top-level decision">
   517‚Üí    Used by `UserPromptSubmit`, `PostToolUse`, `PostToolUseFailure`, `Stop`, `SubagentStop`, and `ConfigChange`. The only value is `"block"`. To allow the action to proceed, omit `decision` from your JSON, or exit 0 without any JSON at all:
   518‚Üí
   519‚Üí    ```json  theme={null}
   520‚Üí    {
   521‚Üí      "decision": "block",
   522‚Üí      "reason": "Test suite must pass before proceeding"
   523‚Üí    }
   524‚Üí    ```
   525‚Üí  </Tab>
   526‚Üí
   527‚Üí  <Tab title="PreToolUse">
   528‚Üí    Uses `hookSpecificOutput` for richer control: allow, deny, or escalate to the user. You can also modify tool input before it runs or inject additional context for Claude. See [PreToolUse decision control](#pretooluse-decision-control) for the full set of options.
   529‚Üí
   530‚Üí    ```json  theme={null}
   531‚Üí    {
   532‚Üí      "hookSpecificOutput": {
   533‚Üí        "hookEventName": "PreToolUse",
   534‚Üí        "permissionDecision": "deny",
   535‚Üí        "permissionDecisionReason": "Database writes are not allowed"
   536‚Üí      }
   537‚Üí    }
   538‚Üí    ```
   539‚Üí  </Tab>
   540‚Üí
   541‚Üí  <Tab title="PermissionRequest">
   542‚Üí    Uses `hookSpecificOutput` to allow or deny a permission request on behalf of the user. When allowing, you can also modify the tool's input or apply permission rules so the user isn't prompted again. See [PermissionRequest decision control](#permissionrequest-decision-control) for the full set of options.
   543‚Üí
   544‚Üí    ```json  theme={null}
   545‚Üí    {
   546‚Üí      "hookSpecificOutput": {
   547‚Üí        "hookEventName": "PermissionRequest",
   548‚Üí        "decision": {
   549‚Üí          "behavior": "allow",
   550‚Üí          "updatedInput": {
   551‚Üí            "command": "npm run lint"
   552‚Üí          }
   553‚Üí        }
   554‚Üí      }
   555‚Üí    }
   556‚Üí    ```
   557‚Üí  </Tab>
   558‚Üí</Tabs>
   559‚Üí
   560‚ÜíFor extended examples including Bash command validation, prompt filtering, and auto-approval scripts, see [What you can automate](/en/hooks-guide#what-you-can-automate) in the guide and the [Bash command validator reference implementation](https://github.com/anthropics/claude-code/blob/main/examples/hooks/bash_command_validator_example.py).
   561‚Üí
   562‚Üí## Hook events
   563‚Üí
   564‚ÜíEach event corresponds to a point in Claude Code's lifecycle where hooks can run. The sections below are ordered to match the lifecycle: from session setup through the agentic loop to session end. Each section describes when the event fires, what matchers it supports, the JSON input it receives, and how to control behavior through output.
   565‚Üí
   566‚Üí### SessionStart
   567‚Üí
   568‚ÜíRuns when Claude Code starts a new session or resumes an existing session. Useful for loading development context like existing issues or recent changes to your codebase, or setting up environment variables. For static context that does not require a script, use [CLAUDE.md](/en/memory) instead.
   569‚Üí
   570‚ÜíSessionStart runs on every session, so keep these hooks fast.
   571‚Üí
   572‚ÜíThe matcher value corresponds to how the session was initiated:
   573‚Üí
   574‚Üí| Matcher   | When it fires                          |
   575‚Üí| :-------- | :------------------------------------- |
   576‚Üí| `startup` | New session                            |
   577‚Üí| `resume`  | `--resume`, `--continue`, or `/resume` |
   578‚Üí| `clear`   | `/clear`                               |
   579‚Üí| `compact` | Auto or manual compaction              |
   580‚Üí
   581‚Üí#### SessionStart input
   582‚Üí
   583‚ÜíIn addition to the [common input fields](#common-input-fields), SessionStart hooks receive `source`, `model`, and optionally `agent_type`. The `source` field indicates how the session started: `"startup"` for new sessions, `"resume"` for resumed sessions, `"clear"` after `/clear`, or `"compact"` after compaction. The `model` field contains the model identifier. If you start Claude Code with `claude --agent <name>`, an `agent_type` field contains the agent name.
   584‚Üí
   585‚Üí```json  theme={null}
   586‚Üí{
   587‚Üí  "session_id": "abc123",
   588‚Üí  "transcript_path": "/Users/.../.claude/projects/.../00893aaf-19fa-41d2-8238-13269b9b3ca0.jsonl",
   589‚Üí  "cwd": "/Users/...",
   590‚Üí  "permission_mode": "default",
   591‚Üí  "hook_event_name": "SessionStart",
   592‚Üí  "source": "startup",
   593‚Üí  "model": "claude-sonnet-4-6"
   594‚Üí}
   595‚Üí```
   596‚Üí
   597‚Üí#### SessionStart decision control
   598‚Üí
   599‚ÜíAny text your hook script prints to stdout is added as context for Claude. In addition to the [JSON output fields](#json-output) available to all hooks, you can return these event-specific fields:
   600‚Üí
   601‚Üí| Field               | Description                                                               |
   602‚Üí| :------------------ | :------------------------------------------------------------------------ |
   603‚Üí| `additionalContext` | String added to Claude's context. Multiple hooks' values are concatenated |
   604‚Üí
   605‚Üí```json  theme={null}
   606‚Üí{
   607‚Üí  "hookSpecificOutput": {
   608‚Üí    "hookEventName": "SessionStart",
   609‚Üí    "additionalContext": "My additional context here"
   610‚Üí  }
   611‚Üí}
   612‚Üí```
   613‚Üí
   614‚Üí#### Persist environment variables
   615‚Üí
   616‚ÜíSessionStart hooks have access to the `CLAUDE_ENV_FILE` environment variable, which provides a file path where you can persist environment variables for subsequent Bash commands.
   617‚Üí
   618‚ÜíTo set individual environment variables, write `export` statements to `CLAUDE_ENV_FILE`. Use append (`>>`) to preserve variables set by other hooks:
   619‚Üí
   620‚Üí```bash  theme={null}
   621‚Üí#!/bin/bash
   622‚Üí
   623‚Üíif [ -n "$CLAUDE_ENV_FILE" ]; then
   624‚Üí  echo 'export NODE_ENV=production' >> "$CLAUDE_ENV_FILE"
   625‚Üí  echo 'export DEBUG_LOG=true' >> "$CLAUDE_ENV_FILE"
   626‚Üí  echo 'export PATH="$PATH:./node_modules/.bin"' >> "$CLAUDE_ENV_FILE"
   627‚Üífi
   628‚Üí
   629‚Üíexit 0
   630‚Üí```
   631‚Üí
   632‚ÜíTo capture all environment changes from setup commands, compare the exported variables before and after:
   633‚Üí
   634‚Üí```bash  theme={null}
   635‚Üí#!/bin/bash
   636‚Üí
   637‚ÜíENV_BEFORE=$(export -p | sort)
   638‚Üí
   639‚Üí# Run your setup commands that modify the environment
   640‚Üísource ~/.nvm/nvm.sh
   641‚Üínvm use 20
   642‚Üí
   643‚Üíif [ -n "$CLAUDE_ENV_FILE" ]; then
   644‚Üí  ENV_AFTER=$(export -p | sort)
   645‚Üí  comm -13 <(echo "$ENV_BEFORE") <(echo "$ENV_AFTER") >> "$CLAUDE_ENV_FILE"
   646‚Üífi
   647‚Üí
   648‚Üíexit 0
   649‚Üí```
   650‚Üí
   651‚ÜíAny variables written to this file will be available in all subsequent Bash commands that Claude Code executes during the session.
   652‚Üí
   653‚Üí<Note>
   654‚Üí  `CLAUDE_ENV_FILE` is available for SessionStart hooks. Other hook types do not have access to this variable.
   655‚Üí</Note>
   656‚Üí
   657‚Üí### UserPromptSubmit
   658‚Üí
   659‚ÜíRuns when the user submits a prompt, before Claude processes it. This allows you
   660‚Üíto add additional context based on the prompt/conversation, validate prompts, or
   661‚Üíblock certain types of prompts.
   662‚Üí
   663‚Üí#### UserPromptSubmit input
   664‚Üí
   665‚ÜíIn addition to the [common input fields](#common-input-fields), UserPromptSubmit hooks receive the `prompt` field containing the text the user submitted.
   666‚Üí
   667‚Üí```json  theme={null}
   668‚Üí{
   669‚Üí  "session_id": "abc123",
   670‚Üí  "transcript_path": "/Users/.../.claude/projects/.../00893aaf-19fa-41d2-8238-13269b9b3ca0.jsonl",
   671‚Üí  "cwd": "/Users/...",
   672‚Üí  "permission_mode": "default",
   673‚Üí  "hook_event_name": "UserPromptSubmit",
   674‚Üí  "prompt": "Write a function to calculate the factorial of a number"
   675‚Üí}
   676‚Üí```
   677‚Üí
   678‚Üí#### UserPromptSubmit decision control
   679‚Üí
   680‚Üí`UserPromptSubmit` hooks can control whether a user prompt is processed and add context. All [JSON output fields](#json-output) are available.
   681‚Üí
   682‚ÜíThere are two ways to add context to the conversation on exit code 0:
   683‚Üí
   684‚Üí* **Plain text stdout**: any non-JSON text written to stdout is added as context
   685‚Üí* **JSON with `additionalContext`**: use the JSON format below for more control. The `additionalContext` field is added as context
   686‚Üí
   687‚ÜíPlain stdout is shown as hook output in the transcript. The `additionalContext` field is added more discretely.
   688‚Üí
   689‚ÜíTo block a prompt, return a JSON object with `decision` set to `"block"`:
   690‚Üí
   691‚Üí| Field               | Description                                                                                                        |
   692‚Üí| :------------------ | :----------------------------------------------------------------------------------------------------------------- |
   693‚Üí| `decision`          | `"block"` prevents the prompt from being processed and erases it from context. Omit to allow the prompt to proceed |
   694‚Üí| `reason`            | Shown to the user when `decision` is `"block"`. Not added to context                                               |
   695‚Üí| `additionalContext` | String added to Claude's context                                                                                   |
   696‚Üí
   697‚Üí```json  theme={null}
   698‚Üí{
   699‚Üí  "decision": "block",
   700‚Üí  "reason": "Explanation for decision",
   701‚Üí  "hookSpecificOutput": {
   702‚Üí    "hookEventName": "UserPromptSubmit",
   703‚Üí    "additionalContext": "My additional context here"
   704‚Üí  }
   705‚Üí}
   706‚Üí```
   707‚Üí
   708‚Üí<Note>
   709‚Üí  The JSON format isn't required for simple use cases. To add context, you can print plain text to stdout with exit code 0. Use JSON when you need to
   710‚Üí  block prompts or want more structured control.
   711‚Üí</Note>
   712‚Üí
   713‚Üí### PreToolUse
   714‚Üí
   715‚ÜíRuns after Claude creates tool parameters and before processing the tool call. Matches on tool name: `Bash`, `Edit`, `Write`, `Read`, `Glob`, `Grep`, `Task`, `WebFetch`, `WebSearch`, and any [MCP tool names](#match-mcp-tools).
   716‚Üí
   717‚ÜíUse [PreToolUse decision control](#pretooluse-decision-control) to allow, deny, or ask for permission to use the tool.
   718‚Üí
   719‚Üí#### PreToolUse input
   720‚Üí
   721‚ÜíIn addition to the [common input fields](#common-input-fields), PreToolUse hooks receive `tool_name`, `tool_input`, and `tool_use_id`. The `tool_input` fields depend on the tool:
   722‚Üí
   723‚Üí##### Bash
   724‚Üí
   725‚ÜíExecutes shell commands.
   726‚Üí
   727‚Üí| Field               | Type    | Example            | Description                                   |
   728‚Üí| :------------------ | :------ | :----------------- | :-------------------------------------------- |
   729‚Üí| `command`           | string  | `"npm test"`       | The shell command to execute                  |
   730‚Üí| `description`       | string  | `"Run test suite"` | Optional description of what the command does |
   731‚Üí| `timeout`           | number  | `120000`           | Optional timeout in milliseconds              |
   732‚Üí| `run_in_background` | boolean | `false`            | Whether to run the command in background      |
   733‚Üí
   734‚Üí##### Write
   735‚Üí
   736‚ÜíCreates or overwrites a file.
   737‚Üí
   738‚Üí| Field       | Type   | Example               | Description                        |
   739‚Üí| :---------- | :----- | :-------------------- | :--------------------------------- |
   740‚Üí| `file_path` | string | `"/path/to/file.txt"` | Absolute path to the file to write |
   741‚Üí| `content`   | string | `"file content"`      | Content to write to the file       |
   742‚Üí
   743‚Üí##### Edit
   744‚Üí
   745‚ÜíReplaces a string in an existing file.
   746‚Üí
   747‚Üí| Field         | Type    | Example               | Description                        |
   748‚Üí| :------------ | :------ | :-------------------- | :--------------------------------- |
   749‚Üí| `file_path`   | string  | `"/path/to/file.txt"` | Absolute path to the file to edit  |
   750‚Üí| `old_string`  | string  | `"original text"`     | Text to find and replace           |
   751‚Üí| `new_string`  | string  | `"replacement text"`  | Replacement text                   |
   752‚Üí| `replace_all` | boolean | `false`               | Whether to replace all occurrences |
   753‚Üí
   754‚Üí##### Read
   755‚Üí
   756‚ÜíReads file contents.
   757‚Üí
   758‚Üí| Field       | Type   | Example               | Description                                |
   759‚Üí| :---------- | :----- | :-------------------- | :----------------------------------------- |
   760‚Üí| `file_path` | string | `"/path/to/file.txt"` | Absolute path to the file to read          |
   761‚Üí| `offset`    | number | `10`                  | Optional line number to start reading from |
   762‚Üí| `limit`     | number | `50`                  | Optional number of lines to read           |
   763‚Üí
   764‚Üí##### Glob
   765‚Üí
   766‚ÜíFinds files matching a glob pattern.
   767‚Üí
   768‚Üí| Field     | Type   | Example          | Description                                                            |
   769‚Üí| :-------- | :----- | :--------------- | :--------------------------------------------------------------------- |
   770‚Üí| `pattern` | string | `"**/*.ts"`      | Glob pattern to match files against                                    |
   771‚Üí| `path`    | string | `"/path/to/dir"` | Optional directory to search in. Defaults to current working directory |
   772‚Üí
   773‚Üí##### Grep
   774‚Üí
   775‚ÜíSearches file contents with regular expressions.
   776‚Üí
   777‚Üí| Field         | Type    | Example          | Description                                                                           |
   778‚Üí| :------------ | :------ | :--------------- | :------------------------------------------------------------------------------------ |
   779‚Üí| `pattern`     | string  | `"TODO.*fix"`    | Regular expression pattern to search for                                              |
   780‚Üí| `path`        | string  | `"/path/to/dir"` | Optional file or directory to search in                                               |
   781‚Üí| `glob`        | string  | `"*.ts"`         | Optional glob pattern to filter files                                                 |
   782‚Üí| `output_mode` | string  | `"content"`      | `"content"`, `"files_with_matches"`, or `"count"`. Defaults to `"files_with_matches"` |
   783‚Üí| `-i`          | boolean | `true`           | Case insensitive search                                                               |
   784‚Üí| `multiline`   | boolean | `false`          | Enable multiline matching                                                             |
   785‚Üí
   786‚Üí##### WebFetch
   787‚Üí
   788‚ÜíFetches and processes web content.
   789‚Üí
   790‚Üí| Field    | Type   | Example                       | Description                          |
   791‚Üí| :------- | :----- | :---------------------------- | :----------------------------------- |
   792‚Üí| `url`    | string | `"https://example.com/api"`   | URL to fetch content from            |
   793‚Üí| `prompt` | string | `"Extract the API endpoints"` | Prompt to run on the fetched content |
   794‚Üí
   795‚Üí##### WebSearch
   796‚Üí
   797‚ÜíSearches the web.
   798‚Üí
   799‚Üí| Field             | Type   | Example                        | Description                                       |
   800‚Üí| :---------------- | :----- | :----------------------------- | :------------------------------------------------ |
   801‚Üí| `query`           | string | `"react hooks best practices"` | Search query                                      |
   802‚Üí| `allowed_domains` | array  | `["docs.example.com"]`         | Optional: only include results from these domains |
   803‚Üí| `blocked_domains` | array  | `["spam.example.com"]`         | Optional: exclude results from these domains      |
   804‚Üí
   805‚Üí##### Task
   806‚Üí
   807‚ÜíSpawns a [subagent](/en/sub-agents).
   808‚Üí
   809‚Üí| Field           | Type   | Example                    | Description                                  |
   810‚Üí| :-------------- | :----- | :------------------------- | :------------------------------------------- |
   811‚Üí| `prompt`        | string | `"Find all API endpoints"` | The task for the agent to perform            |
   812‚Üí| `description`   | string | `"Find API endpoints"`     | Short description of the task                |
   813‚Üí| `subagent_type` | string | `"Explore"`                | Type of specialized agent to use             |
   814‚Üí| `model`         | string | `"sonnet"`                 | Optional model alias to override the default |
   815‚Üí
   816‚Üí#### PreToolUse decision control
   817‚Üí
   818‚Üí`PreToolUse` hooks can control whether a tool call proceeds. Unlike other hooks that use a top-level `decision` field, PreToolUse returns its decision inside a `hookSpecificOutput` object. This gives it richer control: three outcomes (allow, deny, or ask) plus the ability to modify tool input before execution.
   819‚Üí
   820‚Üí| Field                      | Description                                                                                                                                      |
   821‚Üí| :------------------------- | :----------------------------------------------------------------------------------------------------------------------------------------------- |
   822‚Üí| `permissionDecision`       | `"allow"` bypasses the permission system, `"deny"` prevents the tool call, `"ask"` prompts the user to confirm                                   |
   823‚Üí| `permissionDecisionReason` | For `"allow"` and `"ask"`, shown to the user but not Claude. For `"deny"`, shown to Claude                                                       |
   824‚Üí| `updatedInput`             | Modifies the tool's input parameters before execution. Combine with `"allow"` to auto-approve, or `"ask"` to show the modified input to the user |
   825‚Üí| `additionalContext`        | String added to Claude's context before the tool executes                                                                                        |
   826‚Üí
   827‚Üí```json  theme={null}
   828‚Üí{
   829‚Üí  "hookSpecificOutput": {
   830‚Üí    "hookEventName": "PreToolUse",
   831‚Üí    "permissionDecision": "allow",
   832‚Üí    "permissionDecisionReason": "My reason here",
   833‚Üí    "updatedInput": {
   834‚Üí      "field_to_modify": "new value"
   835‚Üí    },
   836‚Üí    "additionalContext": "Current environment: production. Proceed with caution."
   837‚Üí  }
   838‚Üí}
   839‚Üí```
   840‚Üí
   841‚Üí<Note>
   842‚Üí  PreToolUse previously used top-level `decision` and `reason` fields, but these are deprecated for this event. Use `hookSpecificOutput.permissionDecision` and `hookSpecificOutput.permissionDecisionReason` instead. The deprecated values `"approve"` and `"block"` map to `"allow"` and `"deny"` respectively. Other events like PostToolUse and Stop continue to use top-level `decision` and `reason` as their current format.
   843‚Üí</Note>
   844‚Üí
   845‚Üí### PermissionRequest
   846‚Üí
   847‚ÜíRuns when the user is shown a permission dialog.
   848‚ÜíUse [PermissionRequest decision control](#permissionrequest-decision-control) to allow or deny on behalf of the user.
   849‚Üí
   850‚ÜíMatches on tool name, same values as PreToolUse.
   851‚Üí
   852‚Üí#### PermissionRequest input
   853‚Üí
   854‚ÜíPermissionRequest hooks receive `tool_name` and `tool_input` fields like PreToolUse hooks, but without `tool_use_id`. An optional `permission_suggestions` array contains the "always allow" options the user would normally see in the permission dialog. The difference is when the hook fires: PermissionRequest hooks run when a permission dialog is about to be shown to the user, while PreToolUse hooks run before tool execution regardless of permission status.
   855‚Üí
   856‚Üí```json  theme={null}
   857‚Üí{
   858‚Üí  "session_id": "abc123",
   859‚Üí  "transcript_path": "/Users/.../.claude/projects/.../00893aaf-19fa-41d2-8238-13269b9b3ca0.jsonl",
   860‚Üí  "cwd": "/Users/...",
   861‚Üí  "permission_mode": "default",
   862‚Üí  "hook_event_name": "PermissionRequest",
   863‚Üí  "tool_name": "Bash",
   864‚Üí  "tool_input": {
   865‚Üí    "command": "rm -rf node_modules",
   866‚Üí    "description": "Remove node_modules directory"
   867‚Üí  },
   868‚Üí  "permission_suggestions": [
   869‚Üí    { "type": "toolAlwaysAllow", "tool": "Bash" }
   870‚Üí  ]
   871‚Üí}
   872‚Üí```
   873‚Üí
   874‚Üí#### PermissionRequest decision control
   875‚Üí
   876‚Üí`PermissionRequest` hooks can allow or deny permission requests. In addition to the [JSON output fields](#json-output) available to all hooks, your hook script can return a `decision` object with these event-specific fields:
   877‚Üí
   878‚Üí| Field                | Description                                                                                                    |
   879‚Üí| :------------------- | :------------------------------------------------------------------------------------------------------------- |
   880‚Üí| `behavior`           | `"allow"` grants the permission, `"deny"` denies it                                                            |
   881‚Üí| `updatedInput`       | For `"allow"` only: modifies the tool's input parameters before execution                                      |
   882‚Üí| `updatedPermissions` | For `"allow"` only: applies permission rule updates, equivalent to the user selecting an "always allow" option |
   883‚Üí| `message`            | For `"deny"` only: tells Claude why the permission was denied                                                  |
   884‚Üí| `interrupt`          | For `"deny"` only: if `true`, stops Claude                                                                     |
   885‚Üí
   886‚Üí```json  theme={null}
   887‚Üí{
   888‚Üí  "hookSpecificOutput": {
   889‚Üí    "hookEventName": "PermissionRequest",
   890‚Üí    "decision": {
   891‚Üí      "behavior": "allow",
   892‚Üí      "updatedInput": {
   893‚Üí        "command": "npm run lint"
   894‚Üí      }
   895‚Üí    }
   896‚Üí  }
   897‚Üí}
   898‚Üí```
   899‚Üí
   900‚Üí### PostToolUse
   901‚Üí
   902‚ÜíRuns immediately after a tool completes successfully.
   903‚Üí
   904‚ÜíMatches on tool name, same values as PreToolUse.
   905‚Üí
   906‚Üí#### PostToolUse input
   907‚Üí
   908‚Üí`PostToolUse` hooks fire after a tool has already executed successfully. The input includes both `tool_input`, the arguments sent to the tool, and `tool_response`, the result it returned. The exact schema for both depends on the tool.
   909‚Üí
   910‚Üí```json  theme={null}
   911‚Üí{
   912‚Üí  "session_id": "abc123",
   913‚Üí  "transcript_path": "/Users/.../.claude/projects/.../00893aaf-19fa-41d2-8238-13269b9b3ca0.jsonl",
   914‚Üí  "cwd": "/Users/...",
   915‚Üí  "permission_mode": "default",
   916‚Üí  "hook_event_name": "PostToolUse",
   917‚Üí  "tool_name": "Write",
   918‚Üí  "tool_input": {
   919‚Üí    "file_path": "/path/to/file.txt",
   920‚Üí    "content": "file content"
   921‚Üí  },
   922‚Üí  "tool_response": {
   923‚Üí    "filePath": "/path/to/file.txt",
   924‚Üí    "success": true
   925‚Üí  },
   926‚Üí  "tool_use_id": "toolu_01ABC123..."
   927‚Üí}
   928‚Üí```
   929‚Üí
   930‚Üí#### PostToolUse decision control
   931‚Üí
   932‚Üí`PostToolUse` hooks can provide feedback to Claude after tool execution. In addition to the [JSON output fields](#json-output) available to all hooks, your hook script can return these event-specific fields:
   933‚Üí
   934‚Üí| Field                  | Description                                                                                |
   935‚Üí| :--------------------- | :----------------------------------------------------------------------------------------- |
   936‚Üí| `decision`             | `"block"` prompts Claude with the `reason`. Omit to allow the action to proceed            |
   937‚Üí| `reason`               | Explanation shown to Claude when `decision` is `"block"`                                   |
   938‚Üí| `additionalContext`    | Additional context for Claude to consider                                                  |
   939‚Üí| `updatedMCPToolOutput` | For [MCP tools](#match-mcp-tools) only: replaces the tool's output with the provided value |
   940‚Üí
   941‚Üí```json  theme={null}
   942‚Üí{
   943‚Üí  "decision": "block",
   944‚Üí  "reason": "Explanation for decision",
   945‚Üí  "hookSpecificOutput": {
   946‚Üí    "hookEventName": "PostToolUse",
   947‚Üí    "additionalContext": "Additional information for Claude"
   948‚Üí  }
   949‚Üí}
   950‚Üí```
   951‚Üí
   952‚Üí### PostToolUseFailure
   953‚Üí
   954‚ÜíRuns when a tool execution fails. This event fires for tool calls that throw errors or return failure results. Use this to log failures, send alerts, or provide corrective feedback to Claude.
   955‚Üí
   956‚ÜíMatches on tool name, same values as PreToolUse.
   957‚Üí
   958‚Üí#### PostToolUseFailure input
   959‚Üí
   960‚ÜíPostToolUseFailure hooks receive the same `tool_name` and `tool_input` fields as PostToolUse, along with error information as top-level fields:
   961‚Üí
   962‚Üí```json  theme={null}
   963‚Üí{
   964‚Üí  "session_id": "abc123",
   965‚Üí  "transcript_path": "/Users/.../.claude/projects/.../00893aaf-19fa-41d2-8238-13269b9b3ca0.jsonl",
   966‚Üí  "cwd": "/Users/...",
   967‚Üí  "permission_mode": "default",
   968‚Üí  "hook_event_name": "PostToolUseFailure",
   969‚Üí  "tool_name": "Bash",
   970‚Üí  "tool_input": {
   971‚Üí    "command": "npm test",
   972‚Üí    "description": "Run test suite"
   973‚Üí  },
   974‚Üí  "tool_use_id": "toolu_01ABC123...",
   975‚Üí  "error": "Command exited with non-zero status code 1",
   976‚Üí  "is_interrupt": false
   977‚Üí}
   978‚Üí```
   979‚Üí
   980‚Üí| Field          | Description                                                                     |
   981‚Üí| :------------- | :------------------------------------------------------------------------------ |
   982‚Üí| `error`        | String describing what went wrong                                               |
   983‚Üí| `is_interrupt` | Optional boolean indicating whether the failure was caused by user interruption |
   984‚Üí
   985‚Üí#### PostToolUseFailure decision control
   986‚Üí
   987‚Üí`PostToolUseFailure` hooks can provide context to Claude after a tool failure. In addition to the [JSON output fields](#json-output) available to all hooks, your hook script can return these event-specific fields:
   988‚Üí
   989‚Üí| Field               | Description                                                   |
   990‚Üí| :------------------ | :------------------------------------------------------------ |
   991‚Üí| `additionalContext` | Additional context for Claude to consider alongside the error |
   992‚Üí
   993‚Üí```json  theme={null}
   994‚Üí{
   995‚Üí  "hookSpecificOutput": {
   996‚Üí    "hookEventName": "PostToolUseFailure",
   997‚Üí    "additionalContext": "Additional information about the failure for Claude"
   998‚Üí  }
   999‚Üí}
  1000‚Üí```
  1001‚Üí
  1002‚Üí### Notification
  1003‚Üí
  1004‚ÜíRuns when Claude Code sends notifications. Matches on notification type: `permission_prompt`, `idle_prompt`, `auth_success`, `elicitation_dialog`. Omit the matcher to run hooks for all notification types.
  1005‚Üí
  1006‚ÜíUse separate matchers to run different handlers depending on the notification type. This configuration triggers a permission-specific alert script when Claude needs permission approval and a different notification when Claude has been idle:
  1007‚Üí
  1008‚Üí```json  theme={null}
  1009‚Üí{
  1010‚Üí  "hooks": {
  1011‚Üí    "Notification": [
  1012‚Üí      {
  1013‚Üí        "matcher": "permission_prompt",
  1014‚Üí        "hooks": [
  1015‚Üí          {
  1016‚Üí            "type": "command",
  1017‚Üí            "command": "/path/to/permission-alert.sh"
  1018‚Üí          }
  1019‚Üí        ]
  1020‚Üí      },
  1021‚Üí      {
  1022‚Üí        "matcher": "idle_prompt",
  1023‚Üí        "hooks": [
  1024‚Üí          {
  1025‚Üí            "type": "command",
  1026‚Üí            "command": "/path/to/idle-notification.sh"
  1027‚Üí          }
  1028‚Üí        ]
  1029‚Üí      }
  1030‚Üí    ]
  1031‚Üí  }
  1032‚Üí}
  1033‚Üí```
  1034‚Üí
  1035‚Üí#### Notification input
  1036‚Üí
  1037‚ÜíIn addition to the [common input fields](#common-input-fields), Notification hooks receive `message` with the notification text, an optional `title`, and `notification_type` indicating which type fired.
  1038‚Üí
  1039‚Üí```json  theme={null}
  1040‚Üí{
  1041‚Üí  "session_id": "abc123",
  1042‚Üí  "transcript_path": "/Users/.../.claude/projects/.../00893aaf-19fa-41d2-8238-13269b9b3ca0.jsonl",
  1043‚Üí  "cwd": "/Users/...",
  1044‚Üí  "permission_mode": "default",
  1045‚Üí  "hook_event_name": "Notification",
  1046‚Üí  "message": "Claude needs your permission to use Bash",
  1047‚Üí  "title": "Permission needed",
  1048‚Üí  "notification_type": "permission_prompt"
  1049‚Üí}
  1050‚Üí```
  1051‚Üí
  1052‚ÜíNotification hooks cannot block or modify notifications. In addition to the [JSON output fields](#json-output) available to all hooks, you can return `additionalContext` to add context to the conversation:
  1053‚Üí
  1054‚Üí| Field               | Description                      |
  1055‚Üí| :------------------ | :------------------------------- |
  1056‚Üí| `additionalContext` | String added to Claude's context |
  1057‚Üí
  1058‚Üí### SubagentStart
  1059‚Üí
  1060‚ÜíRuns when a Claude Code subagent is spawned via the Task tool. Supports matchers to filter by agent type name (built-in agents like `Bash`, `Explore`, `Plan`, or custom agent names from `.claude/agents/`).
  1061‚Üí
  1062‚Üí#### SubagentStart input
  1063‚Üí
  1064‚ÜíIn addition to the [common input fields](#common-input-fields), SubagentStart hooks receive `agent_id` with the unique identifier for the subagent and `agent_type` with the agent name (built-in agents like `"Bash"`, `"Explore"`, `"Plan"`, or custom agent names).
  1065‚Üí
  1066‚Üí```json  theme={null}
  1067‚Üí{
  1068‚Üí  "session_id": "abc123",
  1069‚Üí  "transcript_path": "/Users/.../.claude/projects/.../00893aaf-19fa-41d2-8238-13269b9b3ca0.jsonl",
  1070‚Üí  "cwd": "/Users/...",
  1071‚Üí  "permission_mode": "default",
  1072‚Üí  "hook_event_name": "SubagentStart",
  1073‚Üí  "agent_id": "agent-abc123",
  1074‚Üí  "agent_type": "Explore"
  1075‚Üí}
  1076‚Üí```
  1077‚Üí
  1078‚ÜíSubagentStart hooks cannot block subagent creation, but they can inject context into the subagent. In addition to the [JSON output fields](#json-output) available to all hooks, you can return:
  1079‚Üí
  1080‚Üí| Field               | Description                            |
  1081‚Üí| :------------------ | :------------------------------------- |
  1082‚Üí| `additionalContext` | String added to the subagent's context |
  1083‚Üí
  1084‚Üí```json  theme={null}
  1085‚Üí{
  1086‚Üí  "hookSpecificOutput": {
  1087‚Üí    "hookEventName": "SubagentStart",
  1088‚Üí    "additionalContext": "Follow security guidelines for this task"
  1089‚Üí  }
  1090‚Üí}
  1091‚Üí```
  1092‚Üí
  1093‚Üí### SubagentStop
  1094‚Üí
  1095‚ÜíRuns when a Claude Code subagent has finished responding. Matches on agent type, same values as SubagentStart.
  1096‚Üí
  1097‚Üí#### SubagentStop input
  1098‚Üí
  1099‚ÜíIn addition to the [common input fields](#common-input-fields), SubagentStop hooks receive `stop_hook_active`, `agent_id`, `agent_type`, `agent_transcript_path`, and `last_assistant_message`. The `agent_type` field is the value used for matcher filtering. The `transcript_path` is the main session's transcript, while `agent_transcript_path` is the subagent's own transcript stored in a nested `subagents/` folder. The `last_assistant_message` field contains the text content of the subagent's final response, so hooks can access it without parsing the transcript file.
  1100‚Üí
  1101‚Üí```json  theme={null}
  1102‚Üí{
  1103‚Üí  "session_id": "abc123",
  1104‚Üí  "transcript_path": "~/.claude/projects/.../abc123.jsonl",
  1105‚Üí  "cwd": "/Users/...",
  1106‚Üí  "permission_mode": "default",
  1107‚Üí  "hook_event_name": "SubagentStop",
  1108‚Üí  "stop_hook_active": false,
  1109‚Üí  "agent_id": "def456",
  1110‚Üí  "agent_type": "Explore",
  1111‚Üí  "agent_transcript_path": "~/.claude/projects/.../abc123/subagents/agent-def456.jsonl",
  1112‚Üí  "last_assistant_message": "Analysis complete. Found 3 potential issues..."
  1113‚Üí}
  1114‚Üí```
  1115‚Üí
  1116‚ÜíSubagentStop hooks use the same decision control format as [Stop hooks](#stop-decision-control).
  1117‚Üí
  1118‚Üí### Stop
  1119‚Üí
  1120‚ÜíRuns when the main Claude Code agent has finished responding. Does not run if
  1121‚Üíthe stoppage occurred due to a user interrupt.
  1122‚Üí
  1123‚Üí#### Stop input
  1124‚Üí
  1125‚ÜíIn addition to the [common input fields](#common-input-fields), Stop hooks receive `stop_hook_active` and `last_assistant_message`. The `stop_hook_active` field is `true` when Claude Code is already continuing as a result of a stop hook. Check this value or process the transcript to prevent Claude Code from running indefinitely. The `last_assistant_message` field contains the text content of Claude's final response, so hooks can access it without parsing the transcript file.
  1126‚Üí
  1127‚Üí```json  theme={null}
  1128‚Üí{
  1129‚Üí  "session_id": "abc123",
  1130‚Üí  "transcript_path": "~/.claude/projects/.../00893aaf-19fa-41d2-8238-13269b9b3ca0.jsonl",
  1131‚Üí  "cwd": "/Users/...",
  1132‚Üí  "permission_mode": "default",
  1133‚Üí  "hook_event_name": "Stop",
  1134‚Üí  "stop_hook_active": true,
  1135‚Üí  "last_assistant_message": "I've completed the refactoring. Here's a summary..."
  1136‚Üí}
  1137‚Üí```
  1138‚Üí
  1139‚Üí#### Stop decision control
  1140‚Üí
  1141‚Üí`Stop` and `SubagentStop` hooks can control whether Claude continues. In addition to the [JSON output fields](#json-output) available to all hooks, your hook script can return these event-specific fields:
  1142‚Üí
  1143‚Üí| Field      | Description                                                                |
  1144‚Üí| :--------- | :------------------------------------------------------------------------- |
  1145‚Üí| `decision` | `"block"` prevents Claude from stopping. Omit to allow Claude to stop      |
  1146‚Üí| `reason`   | Required when `decision` is `"block"`. Tells Claude why it should continue |
  1147‚Üí
  1148‚Üí```json  theme={null}
  1149‚Üí{
  1150‚Üí  "decision": "block",
  1151‚Üí  "reason": "Must be provided when Claude is blocked from stopping"
  1152‚Üí}
  1153‚Üí```
  1154‚Üí
  1155‚Üí### TeammateIdle
  1156‚Üí
  1157‚ÜíRuns when an [agent team](/en/agent-teams) teammate is about to go idle after finishing its turn. Use this to enforce quality gates before a teammate stops working, such as requiring passing lint checks or verifying that output files exist.
  1158‚Üí
  1159‚ÜíWhen a `TeammateIdle` hook exits with code 2, the teammate receives the stderr message as feedback and continues working instead of going idle. TeammateIdle hooks do not support matchers and fire on every occurrence.
  1160‚Üí
  1161‚Üí#### TeammateIdle input
  1162‚Üí
  1163‚ÜíIn addition to the [common input fields](#common-input-fields), TeammateIdle hooks receive `teammate_name` and `team_name`.
  1164‚Üí
  1165‚Üí```json  theme={null}
  1166‚Üí{
  1167‚Üí  "session_id": "abc123",
  1168‚Üí  "transcript_path": "/Users/.../.claude/projects/.../00893aaf-19fa-41d2-8238-13269b9b3ca0.jsonl",
  1169‚Üí  "cwd": "/Users/...",
  1170‚Üí  "permission_mode": "default",
  1171‚Üí  "hook_event_name": "TeammateIdle",
  1172‚Üí  "teammate_name": "researcher",
  1173‚Üí  "team_name": "my-project"
  1174‚Üí}
  1175‚Üí```
  1176‚Üí
  1177‚Üí| Field           | Description                                   |
  1178‚Üí| :-------------- | :-------------------------------------------- |
  1179‚Üí| `teammate_name` | Name of the teammate that is about to go idle |
  1180‚Üí| `team_name`     | Name of the team                              |
  1181‚Üí
  1182‚Üí#### TeammateIdle decision control
  1183‚Üí
  1184‚ÜíTeammateIdle hooks use exit codes only, not JSON decision control. This example checks that a build artifact exists before allowing a teammate to go idle:
  1185‚Üí
  1186‚Üí```bash  theme={null}
  1187‚Üí#!/bin/bash
  1188‚Üí
  1189‚Üíif [ ! -f "./dist/output.js" ]; then
  1190‚Üí  echo "Build artifact missing. Run the build before stopping." >&2
  1191‚Üí  exit 2
  1192‚Üífi
  1193‚Üí
  1194‚Üíexit 0
  1195‚Üí```
  1196‚Üí
  1197‚Üí### TaskCompleted
  1198‚Üí
  1199‚ÜíRuns when a task is being marked as completed. This fires in two situations: when any agent explicitly marks a task as completed through the TaskUpdate tool, or when an [agent team](/en/agent-teams) teammate finishes its turn with in-progress tasks. Use this to enforce completion criteria like passing tests or lint checks before a task can close.
  1200‚Üí
  1201‚ÜíWhen a `TaskCompleted` hook exits with code 2, the task is not marked as completed and the stderr message is fed back to the model as feedback. TaskCompleted hooks do not support matchers and fire on every occurrence.
  1202‚Üí
  1203‚Üí#### TaskCompleted input
  1204‚Üí
  1205‚ÜíIn addition to the [common input fields](#common-input-fields), TaskCompleted hooks receive `task_id`, `task_subject`, and optionally `task_description`, `teammate_name`, and `team_name`.
  1206‚Üí
  1207‚Üí```json  theme={null}
  1208‚Üí{
  1209‚Üí  "session_id": "abc123",
  1210‚Üí  "transcript_path": "/Users/.../.claude/projects/.../00893aaf-19fa-41d2-8238-13269b9b3ca0.jsonl",
  1211‚Üí  "cwd": "/Users/...",
  1212‚Üí  "permission_mode": "default",
  1213‚Üí  "hook_event_name": "TaskCompleted",
  1214‚Üí  "task_id": "task-001",
  1215‚Üí  "task_subject": "Implement user authentication",
  1216‚Üí  "task_description": "Add login and signup endpoints",
  1217‚Üí  "teammate_name": "implementer",
  1218‚Üí  "team_name": "my-project"
  1219‚Üí}
  1220‚Üí```
  1221‚Üí
  1222‚Üí| Field              | Description                                             |
  1223‚Üí| :----------------- | :------------------------------------------------------ |
  1224‚Üí| `task_id`          | Identifier of the task being completed                  |
  1225‚Üí| `task_subject`     | Title of the task                                       |
  1226‚Üí| `task_description` | Detailed description of the task. May be absent         |
  1227‚Üí| `teammate_name`    | Name of the teammate completing the task. May be absent |
  1228‚Üí| `team_name`        | Name of the team. May be absent                         |
  1229‚Üí
  1230‚Üí#### TaskCompleted decision control
  1231‚Üí
  1232‚ÜíTaskCompleted hooks use exit codes only, not JSON decision control. This example runs tests and blocks task completion if they fail:
  1233‚Üí
  1234‚Üí```bash  theme={null}
  1235‚Üí#!/bin/bash
  1236‚ÜíINPUT=$(cat)
  1237‚ÜíTASK_SUBJECT=$(echo "$INPUT" | jq -r '.task_subject')
  1238‚Üí
  1239‚Üí# Run the test suite
  1240‚Üíif ! npm test 2>&1; then
  1241‚Üí  echo "Tests not passing. Fix failing tests before completing: $TASK_SUBJECT" >&2
  1242‚Üí  exit 2
  1243‚Üífi
  1244‚Üí
  1245‚Üíexit 0
  1246‚Üí```
  1247‚Üí
  1248‚Üí### ConfigChange
  1249‚Üí
  1250‚ÜíRuns when a configuration file changes during a session. Use this to audit settings changes, enforce security policies, or block unauthorized modifications to configuration files.
  1251‚Üí
  1252‚ÜíConfigChange hooks fire for changes to settings files, managed policy settings, and skill files. The `source` field in the input tells you which type of configuration changed, and the optional `file_path` field provides the path to the changed file.
  1253‚Üí
  1254‚ÜíThe matcher filters on the configuration source:
  1255‚Üí
  1256‚Üí| Matcher            | When it fires                             |
  1257‚Üí| :----------------- | :---------------------------------------- |
  1258‚Üí| `user_settings`    | `~/.claude/settings.json` changes         |
  1259‚Üí| `project_settings` | `.claude/settings.json` changes           |
  1260‚Üí| `local_settings`   | `.claude/settings.local.json` changes     |
  1261‚Üí| `policy_settings`  | Managed policy settings change            |
  1262‚Üí| `skills`           | A skill file in `.claude/skills/` changes |
  1263‚Üí
  1264‚ÜíThis example logs all configuration changes for security auditing:
  1265‚Üí
  1266‚Üí```json  theme={null}
  1267‚Üí{
  1268‚Üí  "hooks": {
  1269‚Üí    "ConfigChange": [
  1270‚Üí      {
  1271‚Üí        "hooks": [
  1272‚Üí          {
  1273‚Üí            "type": "command",
  1274‚Üí            "command": "\"$CLAUDE_PROJECT_DIR\"/.claude/hooks/audit-config-change.sh"
  1275‚Üí          }
  1276‚Üí        ]
  1277‚Üí      }
  1278‚Üí    ]
  1279‚Üí  }
  1280‚Üí}
  1281‚Üí```
  1282‚Üí
  1283‚Üí#### ConfigChange input
  1284‚Üí
  1285‚ÜíIn addition to the [common input fields](#common-input-fields), ConfigChange hooks receive `source` and optionally `file_path`. The `source` field indicates which configuration type changed, and `file_path` provides the path to the specific file that was modified.
  1286‚Üí
  1287‚Üí```json  theme={null}
  1288‚Üí{
  1289‚Üí  "session_id": "abc123",
  1290‚Üí  "transcript_path": "/Users/.../.claude/projects/.../00893aaf-19fa-41d2-8238-13269b9b3ca0.jsonl",
  1291‚Üí  "cwd": "/Users/...",
  1292‚Üí  "permission_mode": "default",
  1293‚Üí  "hook_event_name": "ConfigChange",
  1294‚Üí  "source": "project_settings",
  1295‚Üí  "file_path": "/Users/.../my-project/.claude/settings.json"
  1296‚Üí}
  1297‚Üí```
  1298‚Üí
  1299‚Üí#### ConfigChange decision control
  1300‚Üí
  1301‚ÜíConfigChange hooks can block configuration changes from taking effect. Use exit code 2 or a JSON `decision` to prevent the change. When blocked, the new settings are not applied to the running session.
  1302‚Üí
  1303‚Üí| Field      | Description                                                                              |
  1304‚Üí| :--------- | :--------------------------------------------------------------------------------------- |
  1305‚Üí| `decision` | `"block"` prevents the configuration change from being applied. Omit to allow the change |
  1306‚Üí| `reason`   | Explanation shown to the user when `decision` is `"block"`                               |
  1307‚Üí
  1308‚Üí```json  theme={null}
  1309‚Üí{
  1310‚Üí  "decision": "block",
  1311‚Üí  "reason": "Configuration changes to project settings require admin approval"
  1312‚Üí}
  1313‚Üí```
  1314‚Üí
  1315‚Üí`policy_settings` changes cannot be blocked. Hooks still fire for `policy_settings` sources, so you can use them for audit logging, but any blocking decision is ignored. This ensures enterprise-managed settings always take effect.
  1316‚Üí
  1317‚Üí### WorktreeCreate
  1318‚Üí
  1319‚ÜíWhen you run `claude --worktree` or a [subagent uses `isolation: "worktree"`](/en/sub-agents#choose-the-subagent-scope), Claude Code creates an isolated working copy using `git worktree`. If you configure a WorktreeCreate hook, it replaces the default git behavior, letting you use a different version control system like SVN, Perforce, or Mercurial.
  1320‚Üí
  1321‚ÜíThe hook must print the absolute path to the created worktree directory on stdout. Claude Code uses this path as the working directory for the isolated session.
  1322‚Üí
  1323‚ÜíThis example creates an SVN working copy and prints the path for Claude Code to use. Replace the repository URL with your own:
  1324‚Üí
  1325‚Üí```json  theme={null}
  1326‚Üí{
  1327‚Üí  "hooks": {
  1328‚Üí    "WorktreeCreate": [
  1329‚Üí      {
  1330‚Üí        "hooks": [
  1331‚Üí          {
  1332‚Üí            "type": "command",
  1333‚Üí            "command": "bash -c 'NAME=$(jq -r .name); DIR=\"$HOME/.claude/worktrees/$NAME\"; svn checkout https://svn.example.com/repo/trunk \"$DIR\" >&2 && echo \"$DIR\"'"
  1334‚Üí          }
  1335‚Üí        ]
  1336‚Üí      }
  1337‚Üí    ]
  1338‚Üí  }
  1339‚Üí}
  1340‚Üí```
  1341‚Üí
  1342‚ÜíThe hook reads the worktree `name` from the JSON input on stdin, checks out a fresh copy into a new directory, and prints the directory path. The `echo` on the last line is what Claude Code reads as the worktree path. Redirect any other output to stderr so it doesn't interfere with the path.
  1343‚Üí
  1344‚Üí#### WorktreeCreate input
  1345‚Üí
  1346‚ÜíIn addition to the [common input fields](#common-input-fields), WorktreeCreate hooks receive the `name` field. This is a slug identifier for the new worktree, either specified by the user or auto-generated (for example, `bold-oak-a3f2`).
  1347‚Üí
  1348‚Üí```json  theme={null}
  1349‚Üí{
  1350‚Üí  "session_id": "abc123",
  1351‚Üí  "transcript_path": "/Users/.../.claude/projects/.../00893aaf-19fa-41d2-8238-13269b9b3ca0.jsonl",
  1352‚Üí  "cwd": "/Users/...",
  1353‚Üí  "hook_event_name": "WorktreeCreate",
  1354‚Üí  "name": "feature-auth"
  1355‚Üí}
  1356‚Üí```
  1357‚Üí
  1358‚Üí#### WorktreeCreate output
  1359‚Üí
  1360‚ÜíThe hook must print the absolute path to the created worktree directory on stdout. If the hook fails or produces no output, worktree creation fails with an error.
  1361‚Üí
  1362‚ÜíWorktreeCreate hooks do not use the standard allow/block decision model. Instead, the hook's success or failure determines the outcome. Only `type: "command"` hooks are supported.
  1363‚Üí
  1364‚Üí### WorktreeRemove
  1365‚Üí
  1366‚ÜíThe cleanup counterpart to [WorktreeCreate](#worktreecreate). This hook fires when a worktree is being removed, either when you exit a `--worktree` session and choose to remove it, or when a subagent with `isolation: "worktree"` finishes. For git-based worktrees, Claude handles cleanup automatically with `git worktree remove`. If you configured a WorktreeCreate hook for a non-git version control system, pair it with a WorktreeRemove hook to handle cleanup. Without one, the worktree directory is left on disk.
  1367‚Üí
  1368‚ÜíClaude Code passes the path that WorktreeCreate printed on stdout as `worktree_path` in the hook input. This example reads that path and removes the directory:
  1369‚Üí
  1370‚Üí```json  theme={null}
  1371‚Üí{
  1372‚Üí  "hooks": {
  1373‚Üí    "WorktreeRemove": [
  1374‚Üí      {
  1375‚Üí        "hooks": [
  1376‚Üí          {
  1377‚Üí            "type": "command",
  1378‚Üí            "command": "bash -c 'jq -r .worktree_path | xargs rm -rf'"
  1379‚Üí          }
  1380‚Üí        ]
  1381‚Üí      }
  1382‚Üí    ]
  1383‚Üí  }
  1384‚Üí}
  1385‚Üí```
  1386‚Üí
  1387‚Üí#### WorktreeRemove input
  1388‚Üí
  1389‚ÜíIn addition to the [common input fields](#common-input-fields), WorktreeRemove hooks receive the `worktree_path` field, which is the absolute path to the worktree being removed.
  1390‚Üí
  1391‚Üí```json  theme={null}
  1392‚Üí{
  1393‚Üí  "session_id": "abc123",
  1394‚Üí  "transcript_path": "/Users/.../.claude/projects/.../00893aaf-19fa-41d2-8238-13269b9b3ca0.jsonl",
  1395‚Üí  "cwd": "/Users/...",
  1396‚Üí  "hook_event_name": "WorktreeRemove",
  1397‚Üí  "worktree_path": "/Users/.../my-project/.claude/worktrees/feature-auth"
  1398‚Üí}
  1399‚Üí```
  1400‚Üí
  1401‚ÜíWorktreeRemove hooks have no decision control. They cannot block worktree removal but can perform cleanup tasks like removing version control state or archiving changes. Hook failures are logged in debug mode only. Only `type: "command"` hooks are supported.
  1402‚Üí

  1462‚Üí## Prompt-based hooks
  1463‚Üí
  1464‚ÜíIn addition to Bash command hooks (`type: "command"`), Claude Code supports prompt-based hooks (`type: "prompt"`) that use an LLM to evaluate whether to allow or block an action, and agent hooks (`type: "agent"`) that spawn an agentic verifier with tool access. Not all events support every hook type.
  1465‚Üí
  1466‚ÜíEvents that support all three hook types (`command`, `prompt`, and `agent`):
  1467‚Üí
  1468‚Üí* `PermissionRequest`
  1469‚Üí* `PostToolUse`
  1470‚Üí* `PostToolUseFailure`
  1471‚Üí* `PreToolUse`
  1472‚Üí* `Stop`
  1473‚Üí* `SubagentStop`
  1474‚Üí* `TaskCompleted`
  1475‚Üí* `UserPromptSubmit`
  1476‚Üí
  1477‚ÜíEvents that only support `type: "command"` hooks:
  1478‚Üí
  1479‚Üí* `ConfigChange`
  1480‚Üí* `Notification`
  1481‚Üí* `PreCompact`
  1482‚Üí* `SessionEnd`
  1483‚Üí* `SessionStart`
  1484‚Üí* `SubagentStart`
  1485‚Üí* `TeammateIdle`
  1486‚Üí* `WorktreeCreate`
  1487‚Üí* `WorktreeRemove`
  1488‚Üí
  1489‚Üí### How prompt-based hooks work
  1490‚Üí
  1491‚ÜíInstead of executing a Bash command, prompt-based hooks:
  1492‚Üí
  1493‚Üí1. Send the hook input and your prompt to a Claude model, Haiku by default
  1494‚Üí2. The LLM responds with structured JSON containing a decision
  1495‚Üí3. Claude Code processes the decision automatically
  1496‚Üí
  1497‚Üí### Prompt hook configuration
  1498‚Üí
  1499‚ÜíSet `type` to `"prompt"` and provide a `prompt` string instead of a `command`. Use the `$ARGUMENTS` placeholder to inject the hook's JSON input data into your prompt text. Claude Code sends the combined prompt and input to a fast Claude model, which returns a JSON decision.
  1500‚Üí
  1501‚ÜíThis `Stop` hook asks the LLM to evaluate whether all tasks are complete before allowing Claude to finish:
  1502‚Üí
  1503‚Üí```json  theme={null}
  1504‚Üí{
  1505‚Üí  "hooks": {
  1506‚Üí    "Stop": [
  1507‚Üí      {
  1508‚Üí        "hooks": [
  1509‚Üí          {
  1510‚Üí            "type": "prompt",
  1511‚Üí            "prompt": "Evaluate if Claude should stop: $ARGUMENTS. Check if all tasks are complete."
  1512‚Üí          }
  1513‚Üí        ]
  1514‚Üí      }
  1515‚Üí    ]
  1516‚Üí  }
  1517‚Üí}
  1518‚Üí```
  1519‚Üí
  1520‚Üí| Field     | Required | Description                                                                                                                                                         |
  1521‚Üí| :-------- | :------- | :------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
  1522‚Üí| `type`    | yes      | Must be `"prompt"`                                                                                                                                                  |
  1523‚Üí| `prompt`  | yes      | The prompt text to send to the LLM. Use `$ARGUMENTS` as a placeholder for the hook input JSON. If `$ARGUMENTS` is not present, input JSON is appended to the prompt |
  1524‚Üí| `model`   | no       | Model to use for evaluation. Defaults to a fast model                                                                                                               |
  1525‚Üí| `timeout` | no       | Timeout in seconds. Default: 30                                                                                                                                     |
  1526‚Üí
  1527‚Üí### Response schema
  1528‚Üí
  1529‚ÜíThe LLM must respond with JSON containing:
  1530‚Üí
  1531‚Üí```json  theme={null}
  1532‚Üí{
  1533‚Üí  "ok": true | false,
  1534‚Üí  "reason": "Explanation for the decision"
  1535‚Üí}
  1536‚Üí```
  1537‚Üí
  1538‚Üí| Field    | Description                                                |
  1539‚Üí| :------- | :--------------------------------------------------------- |
  1540‚Üí| `ok`     | `true` allows the action, `false` prevents it              |
  1541‚Üí| `reason` | Required when `ok` is `false`. Explanation shown to Claude |
  1542‚Üí
  1543‚Üí### Example: Multi-criteria Stop hook
  1544‚Üí
  1545‚ÜíThis `Stop` hook uses a detailed prompt to check three conditions before allowing Claude to stop. If `"ok"` is `false`, Claude continues working with the provided reason as its next instruction. `SubagentStop` hooks use the same format to evaluate whether a [subagent](/en/sub-agents) should stop:
  1546‚Üí
  1547‚Üí```json  theme={null}
  1548‚Üí{
  1549‚Üí  "hooks": {
  1550‚Üí    "Stop": [
  1551‚Üí      {
  1552‚Üí        "hooks": [
  1553‚Üí          {
  1554‚Üí            "type": "prompt",
  1555‚Üí            "prompt": "You are evaluating whether Claude should stop working. Context: $ARGUMENTS\n\nAnalyze the conversation and determine if:\n1. All user-requested tasks are complete\n2. Any errors need to be addressed\n3. Follow-up work is needed\n\nRespond with JSON: {\"ok\": true} to allow stopping, or {\"ok\": false, \"reason\": \"your explanation\"} to continue working.",
  1556‚Üí            "timeout": 30
  1557‚Üí          }
  1558‚Üí        ]
  1559‚Üí      }
  1560‚Üí    ]
  1561‚Üí  }
  1562‚Üí}
  1563‚Üí```
  1564‚Üí
  1565‚Üí## Agent-based hooks
  1566‚Üí
  1567‚ÜíAgent-based hooks (`type: "agent"`) are like prompt-based hooks but with multi-turn tool access. Instead of a single LLM call, an agent hook spawns a subagent that can read files, search code, and inspect the codebase to verify conditions. Agent hooks support the same events as prompt-based hooks.
  1568‚Üí
  1569‚Üí### How agent hooks work
  1570‚Üí
  1571‚ÜíWhen an agent hook fires:
  1572‚Üí
  1573‚Üí1. Claude Code spawns a subagent with your prompt and the hook's JSON input
  1574‚Üí2. The subagent can use tools like Read, Grep, and Glob to investigate
  1575‚Üí3. After up to 50 turns, the subagent returns a structured `{ "ok": true/false }` decision
  1576‚Üí4. Claude Code processes the decision the same way as a prompt hook
  1577‚Üí
  1578‚ÜíAgent hooks are useful when verification requires inspecting actual files or test output, not just evaluating the hook input data alone.
  1579‚Üí
  1580‚Üí### Agent hook configuration
  1581‚Üí
  1582‚ÜíSet `type` to `"agent"` and provide a `prompt` string. The configuration fields are the same as [prompt hooks](#prompt-hook-configuration), with a longer default timeout:
  1583‚Üí
  1584‚Üí| Field     | Required | Description                                                                                 |
  1585‚Üí| :-------- | :------- | :------------------------------------------------------------------------------------------ |
  1586‚Üí| `type`    | yes      | Must be `"agent"`                                                                           |
  1587‚Üí| `prompt`  | yes      | Prompt describing what to verify. Use `$ARGUMENTS` as a placeholder for the hook input JSON |
  1588‚Üí| `model`   | no       | Model to use. Defaults to a fast model                                                      |
  1589‚Üí| `timeout` | no       | Timeout in seconds. Default: 60                                                             |
  1590‚Üí
  1591‚ÜíThe response schema is the same as prompt hooks: `{ "ok": true }` to allow or `{ "ok": false, "reason": "..." }` to block.
  1592‚Üí
  1593‚ÜíThis `Stop` hook verifies that all unit tests pass before allowing Claude to finish:
  1594‚Üí
  1595‚Üí```json  theme={null}
  1596‚Üí{
  1597‚Üí  "hooks": {
  1598‚Üí    "Stop": [
  1599‚Üí      {
  1600‚Üí        "hooks": [
  1601‚Üí          {
  1602‚Üí            "type": "agent",
  1603‚Üí            "prompt": "Verify that all unit tests pass. Run the test suite and check the results. $ARGUMENTS",
  1604‚Üí            "timeout": 120
  1605‚Üí          }
  1606‚Üí        ]
  1607‚Üí      }
  1608‚Üí    ]
  1609‚Üí  }
  1610‚Üí}
  1611‚Üí```
  1612‚Üí
  1613‚Üí## Run hooks in the background
  1614‚Üí
  1615‚ÜíBy default, hooks block Claude's execution until they complete. For long-running tasks like deployments, test suites, or external API calls, set `"async": true` to run the hook in the background while Claude continues working. Async hooks cannot block or control Claude's behavior: response fields like `decision`, `permissionDecision`, and `continue` have no effect, because the action they would have controlled has already completed.
  1616‚Üí
  1617‚Üí### Configure an async hook
  1618‚Üí
  1619‚ÜíAdd `"async": true` to a command hook's configuration to run it in the background without blocking Claude. This field is only available on `type: "command"` hooks.
  1620‚Üí
  1621‚ÜíThis hook runs a test script after every `Write` tool call. Claude continues working immediately while `run-tests.sh` executes for up to 120 seconds. When the script finishes, its output is delivered on the next conversation turn:
  1622‚Üí
  1623‚Üí```json  theme={null}
  1624‚Üí{
  1625‚Üí  "hooks": {
  1626‚Üí    "PostToolUse": [
  1627‚Üí      {
  1628‚Üí        "matcher": "Write",
  1629‚Üí        "hooks": [
  1630‚Üí          {
  1631‚Üí            "type": "command",
  1632‚Üí            "command": "/path/to/run-tests.sh",
  1633‚Üí            "async": true,
  1634‚Üí            "timeout": 120
  1635‚Üí          }
  1636‚Üí        ]
  1637‚Üí      }
  1638‚Üí    ]
  1639‚Üí  }
  1640‚Üí}
  1641‚Üí```
  1642‚Üí
  1643‚ÜíThe `timeout` field sets the maximum time in seconds for the background process. If not specified, async hooks use the same 10-minute default as sync hooks.
  1644‚Üí
  1645‚Üí### How async hooks execute
  1646‚Üí
  1647‚ÜíWhen an async hook fires, Claude Code starts the hook process and immediately continues without waiting for it to finish. The hook receives the same JSON input via stdin as a synchronous hook.
  1648‚Üí
  1649‚ÜíAfter the background process exits, if the hook produced a JSON response with a `systemMessage` or `additionalContext` field, that content is delivered to Claude as context on the next conversation turn.
  1650‚Üí
  1651‚Üí### Example: run tests after file changes
  1652‚Üí
  1653‚ÜíThis hook starts a test suite in the background whenever Claude writes a file, then reports the results back to Claude when the tests finish. Save this script to `.claude/hooks/run-tests-async.sh` in your project and make it executable with `chmod +x`:
  1654‚Üí
  1655‚Üí```bash  theme={null}
  1656‚Üí#!/bin/bash
  1657‚Üí# run-tests-async.sh
  1658‚Üí
  1659‚Üí# Read hook input from stdin
  1660‚ÜíINPUT=$(cat)
  1661‚ÜíFILE_PATH=$(echo "$INPUT" | jq -r '.tool_input.file_path // empty')
  1662‚Üí
  1663‚Üí# Only run tests for source files
  1664‚Üíif [[ "$FILE_PATH" != *.ts && "$FILE_PATH" != *.js ]]; then
  1665‚Üí  exit 0
  1666‚Üífi
  1667‚Üí
  1668‚Üí# Run tests and report results via systemMessage
  1669‚ÜíRESULT=$(npm test 2>&1)
  1670‚ÜíEXIT_CODE=$?
  1671‚Üí
  1672‚Üíif [ $EXIT_CODE -eq 0 ]; then
  1673‚Üí  echo "{\"systemMessage\": \"Tests passed after editing $FILE_PATH\"}"
  1674‚Üíelse
  1675‚Üí  echo "{\"systemMessage\": \"Tests failed after editing $FILE_PATH: $RESULT\"}"
  1676‚Üífi
  1677‚Üí```
  1678‚Üí
  1679‚ÜíThen add this configuration to `.claude/settings.json` in your project root. The `async: true` flag lets Claude keep working while tests run:
  1680‚Üí
  1681‚Üí```json  theme={null}
  1682‚Üí{
  1683‚Üí  "hooks": {
  1684‚Üí    "PostToolUse": [
  1685‚Üí      {
  1686‚Üí        "matcher": "Write|Edit",
  1687‚Üí        "hooks": [
  1688‚Üí          {
  1689‚Üí            "type": "command",
  1690‚Üí            "command": "\"$CLAUDE_PROJECT_DIR\"/.claude/hooks/run-tests-async.sh",
  1691‚Üí            "async": true,
  1692‚Üí            "timeout": 300
  1693‚Üí          }
  1694‚Üí        ]
  1695‚Üí      }
  1696‚Üí    ]
  1697‚Üí  }
  1698‚Üí}
  1699‚Üí```
  1700‚Üí
  1701‚Üí### Limitations
  1702‚Üí
  1703‚ÜíAsync hooks have several constraints compared to synchronous hooks:
  1704‚Üí
  1705‚Üí* Only `type: "command"` hooks support `async`. Prompt-based hooks cannot run asynchronously.
  1706‚Üí* Async hooks cannot block tool calls or return decisions. By the time the hook completes, the triggering action has already proceeded.
  1707‚Üí* Hook output is delivered on the next conversation turn. If the session is idle, the response waits until the next user interaction.
  1708‚Üí* Each execution creates a separate background process. There is no deduplication across multiple firings of the same async hook.
  1709‚Üí
  1710‚Üí## Security considerations
  1711‚Üí
  1712‚Üí### Disclaimer
  1713‚Üí
  1714‚ÜíHooks run with your system user's full permissions.
  1715‚Üí
  1716‚Üí<Warning>
  1717‚Üí  Hooks execute shell commands with your full user permissions. They can modify, delete, or access any files your user account can access. Review and test all hook commands before adding them to your configuration.
  1718‚Üí</Warning>
  1719‚Üí
  1720‚Üí### Security best practices
  1721‚Üí
  1722‚ÜíKeep these practices in mind when writing hooks:
  1723‚Üí
  1724‚Üí* **Validate and sanitize inputs**: never trust input data blindly
  1725‚Üí* **Always quote shell variables**: use `"$VAR"` not `$VAR`
  1726‚Üí* **Block path traversal**: check for `..` in file paths
  1727‚Üí* **Use absolute paths**: specify full paths for scripts, using `"$CLAUDE_PROJECT_DIR"` for the project root
  1728‚Üí* **Skip sensitive files**: avoid `.env`, `.git/`, keys, etc.
  1729‚Üí
